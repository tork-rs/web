# OpenAPI and Docs

Tork generates an OpenAPI document from your routes and `#[api_model]` types and serves an
interactive documentation page. Nothing is hand-written: the document is assembled from
the same route metadata and schemas the framework already has.

OpenAPI support lives behind a default-on `openapi` feature. If you do not need it, disable
default features to drop the dependency.

## Enabling it

Configure OpenAPI on the `App` with the `OpenApi` builder.

```rust
use tork::{App, OpenApi};

App::new()
    .include_router(orders::router())
    .openapi(
        OpenApi::new()
            .title("My API")
            .version("1.0.0")
            .json("/openapi.json")
            .docs("/docs"),
    )
    .serve("0.0.0.0:8000")
    .await
```

The builder methods:

| Method | Effect | Default |
| --- | --- | --- |
| `title(...)` | Document title in `info`. | `"API"` |
| `version(...)` | API version in `info`. | `"0.1.0"` |
| `description(...)` | Description in `info`. | none |
| `json(path)` | Path the JSON spec is served at. | `/openapi.json` |
| `docs(path)` | Path the interactive docs page is served at. | not served unless set |
| `protect(predicate)` | Gate the spec and docs routes behind a check. | open |

Calling `.json` is optional: the spec is served at `/openapi.json` by default. The docs
page is only served when you call `.docs(...)`.

## A documented endpoint

Route metadata drives the document. A route's `summary` and `description` come from the
handler's doc comment, `tags` from the router or route, the request body from a `Valid<T>`
or `Json<T>` parameter, and the response from `response_model` (or the return type). Models
are defined with `#[api_model]`, which provides the JSON Schema.

::code-group

```rust [src/schemas/order.rs]
use tork::api_model;

#[api_model]
pub struct CreateOrder {
    #[field(min_length = 1)]
    pub item: String,
    #[field(gt = 0)]
    pub quantity: u32,
}

#[api_model]
pub struct OrderOut {
    pub id: i64,
    pub item: String,
}
```

```rust [src/routers/orders.rs]
use tork::{api_router, post, Valid};

use crate::schemas::order::{CreateOrder, OrderOut};

#[api_router(prefix = "/users/{user_id}/orders", tags = ["orders"])]
pub mod orders_router {
    use super::*;

    /// Create an order for a user.
    #[post("/", response_model = OrderOut, status_code = 201)]
    pub async fn create(user_id: i64, body: Valid<CreateOrder>) -> tork::Result<OrderOut> {
        let input = body.into_inner();
        Ok(OrderOut { id: 1, item: input.item })
    }
}

pub use orders_router::router;
```

```rust [src/main.rs]
use tork::{App, OpenApi};

use crate::routers::orders;

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .include_router(orders::router())
        .openapi(OpenApi::new().title("My API").version("1.0.0").docs("/docs"))
        .serve("0.0.0.0:8000")
        .await
}
```

::

A request to `/openapi.json` then returns:

```json
{
  "openapi": "3.1.0",
  "info": { "title": "My API", "version": "1.0.0" },
  "paths": {
    "/users/{user_id}/orders/": {
      "post": {
        "summary": "Create an order for a user.",
        "tags": ["orders"],
        "operationId": "post_users_user_id_orders_",
        "parameters": [
          { "name": "user_id", "in": "path", "required": true, "schema": { "type": "string" } }
        ],
        "requestBody": {
          "required": true,
          "content": { "application/json": { "schema": { "$ref": "#/components/schemas/CreateOrder" } } }
        },
        "responses": {
          "201": {
            "description": "Created",
            "content": { "application/json": { "schema": { "$ref": "#/components/schemas/OrderOut" } } }
          }
        }
      }
    }
  },
  "components": { "schemas": { "CreateOrder": {}, "OrderOut": {} } }
}
```

## What the document contains

For each route the document records:

- **summary**, **description**, and **tags** from the route metadata.
- an **operationId** derived from the method and path (here `post_users_user_id_orders_`).
- **parameters** for each path placeholder, marked `in: path`, `required: true`, with a
  string schema. Query parameters are not enumerated.
- a **requestBody** when the handler declares one, with the media type matching the body
  encoding: `application/json` for a `Valid<T>` or `Json<T>`, `application/x-www-form-urlencoded`
  for a `Form<T>`, or `multipart/form-data` for a `Multipart<T>` (whose file fields carry
  `format: binary`). The body is marked required.
- a **responses** entry keyed by the route's status code (the `status_code` attribute, or
  `200`), with the description set to the status's reason phrase and an `application/json`
  schema when a response model is declared. A Server-Sent Events route instead documents a
  `text/event-stream` response.

Schemas come from `#[api_model]` types. Each model is registered once under
`components.schemas` and referenced by `$ref`, so nested models are shared rather than
duplicated. All text pulled into the document (titles, summaries, tags) is HTML-escaped, so
a doc comment cannot inject markup into the rendered page.

## The documentation page

`docs("/docs")` serves an HTML page that renders the document with the Scalar API
reference. Open it in a browser to read and try the API. The Scalar bundle is loaded from a
CDN pinned to an exact version and guarded with a Subresource Integrity hash, so the
browser rejects the script if its bytes ever change. A compromised or bumped CDN cannot
inject code into the page.

## Protecting the spec and docs

The spec and docs describe your whole API surface, which you may not want publicly
discoverable. `protect` gates both routes behind a predicate; a request the predicate
rejects gets a `404`, hiding that the routes exist at all.

```rust
use tork::security::constant_time_eq;

OpenApi::new()
    .json("/openapi.json")
    .docs("/docs")
    .protect(|ctx| {
        ctx.headers()
            .get("authorization")
            .and_then(|v| v.to_str().ok())
            .map(|header| constant_time_eq(header, "Bearer secret-docs-token"))
            .unwrap_or(false)
    });
```

Gate it on a bearer token, an internal-only network, or an environment flag, or simply do
not call `.openapi(...)` in production. Compare secrets with `constant_time_eq` rather than
`==`: a plain comparison returns as soon as it hits the first wrong byte, and that timing
leaks, over many tries, how much of the secret an attacker has guessed.

## AsyncAPI for realtime

OpenAPI does not describe WebSocket or SSE routes, so Tork emits a separate AsyncAPI 3
document for them with the `AsyncApi` builder. It is registered on the app the same way and
serves at `/asyncapi.json` by default.

```rust
use tork::{App, AsyncApi};

App::new()
    .include_router(chat::router())
    .include_router(ticks::router())
    .asyncapi(
        AsyncApi::new()
            .title("Realtime API")
            .version("1.0.0")
            .json("/asyncapi.json"),
    );
```

Each realtime route becomes a channel addressed by its path:

- an **SSE route** becomes a channel with a `data` message carrying the event payload
  schema, and a `send` operation.
- a **WebSocket route** becomes a channel with an `incoming` message (a `receive`
  operation) and an `outgoing` message (a `send` operation), built from the handler's
  declared receive and send types.

Message payloads are collected under `components.schemas` and referenced by `$ref`, just
like the OpenAPI document. `AsyncApi` shares `title`, `version`, and `description` with the
OpenAPI builder, but has no docs UI or `protect`.
