# Testing

Tork ships an in-process test client. It builds your app once, runs the lifespan
startup, then sends requests straight through the request pipeline without a network
socket. No port to bind, no race conditions, fast tests. The same client opens
WebSockets, reads Server-Sent Events, overrides resources and dependencies, captures
logs, and runs the lifespan shutdown when finished.

## A first test

Build a `TestApp` with `App::build_test`, wrap it in a `TestClient`, and send requests.
Tests are async, so use `#[tokio::test]`.

```rust
use serde_json::json;
use tork::testing::TestClient;
use tork::{get, App};

#[get("/")]
async fn read_main() -> tork::Result<serde_json::Value> {
    Ok(json!({ "msg": "Hello World" }))
}

#[tokio::test]
async fn reads_main() -> tork::Result<()> {
    let app = App::new().include(read_main).build_test().await?;
    let client = TestClient::new(app).await?;

    let response = client.get("/").send().await?;
    assert_eq!(response.status(), 200);
    assert_eq!(
        response.json::<serde_json::Value>().await?,
        json!({ "msg": "Hello World" })
    );
    Ok(())
}
```

`App::include(handler)` registers a single handler by its generated route factory.
`include_router` still works for mounting a whole router.

## Requests and responses

The verb methods (`get`, `post`, `put`, `patch`, `delete`) return a request builder.
Set a body, add headers and query parameters, then `send`.

```rust
// JSON body.
let response = client
    .post("/items")
    .header("x-tenant", "acme")
    .query("dry_run", "true")
    .json(&json!({ "id": "foo", "name": "Foo" }))
    .send()
    .await?;

// Urlencoded form.
let response = client
    .post("/login")
    .form(&json!({ "username": "ada", "password": "secret" }))
    .send()
    .await?;

// Raw bytes, with the content type you set yourself.
let response = client
    .post("/raw")
    .header("content-type", "application/octet-stream")
    .bytes(vec![0u8, 1, 2, 3])
    .send()
    .await?;
```

Read the response. The body is read once when the request is sent, so these accessors
are cheap and can be called more than once.

```rust
response.status();                       // u16
response.status_code();                  // http::StatusCode
response.headers();                      // &HeaderMap
response.text()?;                        // String
response.bytes();                        // Bytes
response.json::<ItemOut>().await?;       // deserialize JSON
```

### File uploads

`multipart()` builds a `multipart/form-data` body with text fields and files, matching
the multipart parser your handlers use.

```rust
let response = client
    .post("/files")
    .multipart()
    .text("token", "secret-token")
    .file_bytes("file", "hello.txt", "text/plain", b"hello world".to_vec())
    .send()
    .await?;
```

## Cookies and default headers

A `TestClient` keeps a cookie jar. Every `Set-Cookie` in a response is stored and
replayed on later requests, so a login flow carries its session automatically. Use the
builder to set headers and cookies sent with every request.

```rust
let client = TestClient::builder(app)
    .default_header("X-Token", "cone-of-silence")
    .cookie("session", "seed-value")
    .build()
    .await?;
```

In-process tests reject security-sensitive routing and proxy headers (`Host`,
`Forwarded`, `X-Forwarded-*`) by default, so a unit-style test cannot silently spoof
proxy state. When you need one, opt in explicitly per request with `unsafe_header` (or
`unsafe_default_header` on the builder).

```rust
let response = client
    .get("/")
    .unsafe_header("host", "example.com")
    .unsafe_header("x-forwarded-proto", "https")
    .send()
    .await?;
```

## Testing a router

A real test usually exercises a router defined in your application code rather than a
handler declared inline. Keep the schema and router in `src/` and the test in `tests/`,
mounting the router with `include_router`.

::code-group

```rust [src/schemas/item.rs]
use tork::api_model;

#[api_model]
pub struct CreateItem {
    #[field(min_length = 1)]
    pub name: String,
}

#[api_model]
pub struct ItemOut {
    pub id: i64,
    pub name: String,
}
```

```rust [src/routers/items.rs]
use tork::{api_router, post, Valid};

use crate::schemas::item::{CreateItem, ItemOut};

#[api_router(prefix = "/items", tags = ["items"])]
pub mod items_router {
    use super::*;

    #[post("/", response_model = ItemOut, status_code = 201)]
    pub async fn create(body: Valid<CreateItem>) -> tork::Result<ItemOut> {
        let input = body.into_inner();
        Ok(ItemOut { id: 1, name: input.name })
    }
}

pub use items_router::router;
```

```rust [tests/items.rs]
use serde_json::json;
use tork::testing::TestClient;
use tork::App;

use my_app::routers::items;
use my_app::schemas::item::ItemOut;

#[tokio::test]
async fn creates_an_item() -> tork::Result<()> {
    let app = App::new().include_router(items::router()).build_test().await?;
    let client = TestClient::new(app).await?;

    let response = client
        .post("/items/")
        .json(&json!({ "name": "Foo" }))
        .send()
        .await?;

    assert_eq!(response.status(), 201);
    assert_eq!(response.json::<ItemOut>().await?.name, "Foo");
    Ok(())
}
```

::

A body that fails validation is rejected with `422` before the handler runs, so a test
can assert on that status directly.

## Overriding resources and dependencies

Tests should not touch a production database or read `.env`. The builder overrides both
resources and injected dependencies. A resource override is applied after the lifespan
runs, so it wins over the value the lifespan registered.

```rust
let client = TestClient::builder(app)
    .resource(Arc::new(Config { /* test values */ }))
    .resource(TestDb::new())
    .override_dependency::<EmailService>(EmailService::mock())
    .build()
    .await?;
```

Because Tork injects concrete types, an override supplies an instance of the same type
the handler asks for, not an unrelated mock type. `override_dependency` takes a value
cloned per request; `override_dependency_with` takes a factory for types that are not
`Clone`. The override bypasses the service's normal field resolution, so its
dependencies need not be registered in the test.

## Asserting on logs

Attach a `LogRecorder` to capture what the application logged during a test. The client
routes its log records to the recorder for the test's lifetime, including the automatic
HTTP request log. This works with the default current-thread test runtime.

```rust
use tork::testing::{LogRecorder, TestClient};
use tork::{assert_logs, App};

#[tokio::test]
async fn logs_the_greeting() -> tork::Result<()> {
    let recorder = LogRecorder::new();
    let client = TestClient::builder(App::new().include_router(greeter::router()))
        .logger(recorder.clone())
        .build()
        .await?;

    client.get("/greet").send().await?;

    // Convenience predicates.
    assert!(recorder.contains_context("Greeter"));
    assert!(recorder.contains_message("Greeting the world"));

    // Or assert on context and a message substring together.
    assert_logs!(recorder, context = "Greeter", message = "Greeting");

    // The automatic HTTP request log is captured too.
    assert!(recorder.contains_context("HTTP"));
    Ok(())
}
```

`recorder.records()` returns a snapshot of every captured `LogRecord` (its `level`,
`context`, `message`, and structured `fields`) for finer assertions, and `clear()`
empties it between phases of a test.

## WebSockets

`websocket(path).connect()` opens a connection over an in-memory duplex, no socket. It
returns an error if the handshake or a dependency is rejected before the upgrade.

```rust
let mut ws = client.websocket("/ws").connect().await?;
ws.send_text("hello").await?;
assert_eq!(ws.receive_text().await?, "hello");

ws.send_json(&json!({ "message": "hi" })).await?;
let reply = ws.receive_json::<ChatOut>().await?;

ws.send_binary(vec![1u8, 2, 3]).await?;

ws.close().await?;
```

`receive()` returns the next raw `WsMessage` (or `None` once the connection closes),
while `receive_text` and `receive_json` skip control frames for you. To assert on a
clean shutdown, `receive_close()` waits for the close frame and returns its code and
reason.

The builder accepts headers, query parameters, and subprotocols:

```rust
let mut ws = client
    .websocket("/rooms/general")
    .query("token", "abc")
    .subprotocol("chat.v1")
    .connect()
    .await?;
```

## Server-Sent Events

`sse()` reads a streaming response event by event. Heartbeat comments are skipped, so
every returned event carries a name, data, or id.

```rust
let mut stream = client.get("/events").sse().await?;

let event = stream.next_event().await?.expect("an event");
assert_eq!(event.event(), Some("tick"));
assert_eq!(event.json::<EventOut>()?.n, 1);

// Or bound the wait so a stalled stream fails the test instead of hanging.
let event = stream.next_event_timeout(Duration::from_secs(1)).await?;
```

## Lifespan and shutdown

`build_test` runs the lifespan startup, so resources a lifespan registers are available
in tests. Rust has no async `Drop`, so call `shutdown` to run the lifespan teardown when
the test is done.

```rust
let app = App::new().lifespan::<AppState>().include_router(api::router());
let client = TestClient::new(app.build_test().await?).await?;

// ... requests ...

client.shutdown().await?;
```

## End-to-end over a real port

The in-process client is the default and covers most needs. For a true end-to-end test
with real sockets and the full HTTP stack, bind the app to a loopback port.

```rust
let client = TestClient::serve(app).bind_random_port().await?;
let response = client.get("/health").send().await?;
let mut ws = client.websocket("/ws").connect().await?;
client.shutdown().await?;
```

The request, response, and WebSocket API are identical in both modes, so a test reads
the same whether it runs in process or over a socket. Use `local_addr()` if you need the
bound address.
