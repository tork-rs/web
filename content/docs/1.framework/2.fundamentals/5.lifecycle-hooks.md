# Lifecycle Hooks

Hooks let you observe the request lifecycle without changing it. Exception handlers let you
map a typed error into a response. The panic boundary turns a handler panic into a clean
`500`. Together they cover logging, metrics, tracing, custom error pages, and a safety net
for bugs. Hooks sit beside the request path and only watch; [middleware](/docs/framework/getting-started/middleware)
sits in the path and can change the request and response.

## Where each piece runs

```
on_request                         (every request, before middleware)
  -> middleware chain
       -> handler / extractors / validation   (panic boundary, on by default)
            on_error or on_validation_error    (when an error is produced)
            exception_handler                  (when a typed error matches)
            response
       <- middleware unwind
  on_response                        (every response, after middleware)
```

A handler error is rendered inside the chain, so response middleware (compression, CORS
headers) still decorates error responses.

## Observability hooks

The five observability hooks are observe-only: they receive an event, return nothing, and
cannot change the response. They run in registration order, inline with the request, so
keep them quick.

```rust
use tork::{App, RequestEvent, ResponseEvent, ErrorEvent, ValidationErrorEvent};

App::new()
    .on_request(|event: RequestEvent| async move {
        println!("--> {} {}", event.method(), event.path());
    })
    .on_response(|event: ResponseEvent| async move {
        println!(
            "<-- {} {} {} ({:.1?})",
            event.method(), event.path(), event.status(), event.elapsed()
        );
    })
    .on_error(|event: ErrorEvent| async move {
        // Fires for every non-validation error.
        eprintln!("error {} on {}: {}", event.code(), event.path(), event.message());
    })
    .on_validation_error(|event: ValidationErrorEvent| async move {
        // Fires only for a 422 validation failure.
        eprintln!("{} field(s) failed on {}", event.details().len(), event.path());
    })
    .include_router(users::router())
    .serve("0.0.0.0:8000")
    .await
```

Each event carries `method()`, `path()`, `route()`, and `request_id()`, plus a few extras:

| Event | Hook | Fires when | Extra accessors |
| --- | --- | --- | --- |
| `RequestEvent` | `on_request` | a request arrives | — |
| `ResponseEvent` | `on_response` | a response is ready | `status()`, `elapsed()` |
| `ErrorEvent` | `on_error` | a non-validation error is produced | `status()`, `code()`, `message()` |
| `ValidationErrorEvent` | `on_validation_error` | a body fails validation (`422`) | `details()` |
| `PanicEvent` | `on_panic` | a caught panic occurs | `message()` |

A few details:

- `route()` is the matched route pattern (for example `/users/{id}`). It is populated for
  `on_error`, `on_validation_error`, `on_panic`, and scoped hooks, which fire after routing.
  It is `None` for the app-global `on_request` and `on_response` (they fire outside routing)
  and when no route matched.
- `request_id()` is the `x-request-id` value when present. Register the `RequestId`
  middleware so every request carries one.
- A `422` validation failure fires only `on_validation_error`; every other error fires
  `on_error`. The two never both fire for the same error.

Registering several hooks of one kind is fine; they run in order.

```rust
App::new()
    .on_request(record_metric)
    .on_request(start_trace_span);   // both run, in this order
```

## Scoping hooks to routers and routes

The four request/response/error hooks can also attach to a single router or route, so a
concern targets part of the API instead of every request. Scoped hooks run in addition to
the app-global ones. (`on_panic`, `exception_handler`, and the panic boundary stay
app-level.) Because an attribute cannot hold a closure, a route-level hook names a function
path; each kind is repeatable.

::code-group

```rust [src/observability/audit.rs]
use tork::{ErrorEvent, RequestEvent, ResponseEvent};

pub async fn audit_request(event: RequestEvent) {
    println!("audit: {} {}", event.method(), event.path());
}

pub async fn record_latency(event: ResponseEvent) {
    println!("{} took {:.1?}", event.path(), event.elapsed());
}

pub async fn alert_on_error(event: ErrorEvent) {
    eprintln!("admin error {} on {}", event.code(), event.path());
}
```

```rust [src/routers/admin.rs]
use tork::{api_router, get};

use crate::observability::audit::{audit_request, record_latency};
use crate::schemas::user::UserOut;

#[api_router(prefix = "/admin", tags = ["admin"])]
pub mod admin_router {
    use super::*;

    // Route-level hooks: only this route.
    #[get("/users", on_request = audit_request, on_response = record_latency)]
    pub async fn list_users() -> tork::Result<Vec<UserOut>> {
        Ok(Vec::new())
    }
}

pub use admin_router::router;
```

```rust [src/main.rs]
use tork::App;

use crate::observability::audit::{alert_on_error, audit_request};
use crate::routers::{admin, public};

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        // Router-level hooks: every route in admin (and any nested router).
        .include_router(
            admin::router()
                .on_request(audit_request)
                .on_error(alert_on_error),
        )
        .include_router(public::router()) // unaffected
        .serve("0.0.0.0:8000")
        .await
}
```

::

Scoped hooks fire after routing, so a route-scoped `on_request` does not run when a
middleware short-circuits or no route matches. They compose with the app-global hooks in
this order:

```
app on_request                         (before middleware)
  -> middleware
       scoped on_request               (outer router -> inner router -> route)
       handler
       scoped on_error / on_validation_error  (after the app-global ones)
       scoped on_response              (route -> inner -> outer router)
  <- middleware unwind
app on_response                        (after middleware)
```

## Typed errors and exception handlers

A handler often calls into code that fails with its own error type. Derive `AppError` on
that type to convert it into a framework error through `?`, and register an
`exception_handler` to control the response. The handler receives the recovered typed value
and an `ErrorContext` with the request metadata (`method`, `path`, `route`, `request_id`).

::code-group

```rust [src/core/errors.rs]
#[derive(Debug, tork::AppError)]
#[status(503)] // default status when no handler is registered
pub enum RepoError {
    Unavailable,
}

impl std::fmt::Display for RepoError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            RepoError::Unavailable => f.write_str("the data store is unavailable"),
        }
    }
}

impl std::error::Error for RepoError {}
```

```rust [src/main.rs]
use tork::{App, ErrorContext, IntoResponse, Response};

use crate::core::errors::RepoError;
use crate::routers::users;

async fn on_repo_error(error: RepoError, ctx: ErrorContext) -> Response {
    eprintln!("repo error on {}: {error}", ctx.path());
    tork::Error::service_unavailable("the data store is temporarily unavailable")
        .into_response()
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .exception_handler::<RepoError, _, _>(on_repo_error)
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

::

The handler matches the error's directly attached source, recovered by type. Registering a
handler for the same type again replaces the previous one. With no handler registered, the
error renders with its declared `#[status(...)]` using the standard error body. As always,
a `5xx` body is redacted to a generic message while the detail is logged server-side. See
[Error Handling and Exceptions](/docs/framework/getting-started/error-handling-and-exceptions)
for the full error model and the `#[status(...)]` options.

## The panic boundary

The panic boundary is **on by default**: a panic in a handler is caught and turned into a
generic `500` response instead of dropping the connection. Observe caught panics with
`on_panic`.

```rust
use tork::{App, PanicEvent};

App::new()
    .on_panic(|event: PanicEvent| async move {
        eprintln!("panic on {}: {}", event.path(), event.message());
    })
    .include_router(users::router())
    .serve("0.0.0.0:8000")
    .await
```

A caught panic returns the generic `500` and fires `on_panic` only; it does not run
`on_error` or any exception handler. To opt out and let panics propagate and tear down the
request task instead, call `propagate_panics()`. Either way the boundary has no effect when
the binary is built with `panic = "abort"`.

## Hooks and the application lifecycle

These request hooks are independent of the application lifecycle. You can combine the panic
boundary, the request hooks, and exception handlers with a [lifespan](/docs/framework/getting-started/lifespan)
freely.
