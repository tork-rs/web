# Logging

Tork logs through an injectable, context-aware `Logger` built on `tracing`. In
development the output is a colored, NestJS-style console. In production it is one flat
JSON object per line. HTTP requests, startup, and the application lifecycle are logged
automatically, and a request id correlates every line emitted while handling a request.

## Logging in a service

Add a `Logger` field to a `#[derive(Inject)]` service. The macro gives it the struct's
name as context. Build a record with a level method, attach fields, and call `emit`.
Nothing is logged until `emit` is called.

::code-group

```rust [src/services/payment_service.rs]
use tork::{Inject, Logger};

#[derive(Clone, Inject)]
pub struct PaymentService {
    logger: Logger,
}

impl PaymentService {
    pub async fn charge(&self, user_id: i64) -> tork::Result<()> {
        self.logger
            .info("Charging user")
            .field("user_id", user_id)
            .emit();
        Ok(())
    }
}
```

```rust [src/routers/payments.rs]
use tork::{api_router, post};

use crate::services::payment_service::PaymentService;

#[api_router(prefix = "/payments", tags = ["payments"])]
pub mod payments_router {
    use super::*;

    #[post("/{user_id}/charge")]
    pub async fn charge(user_id: i64, payments: PaymentService) -> tork::Result<&'static str> {
        payments.charge(user_id).await?;
        Ok("ok")
    }
}

pub use payments_router::router;
```

::

Development output:

```
[Tork] 19096 - 06/11/2026, 14:32:11 INFO [PaymentService] Charging user user_id=42 +5ms
```

Production output with `format = json`:

```json
{"timestamp":"2026-06-11T14:32:11Z","level":"INFO","service":"my-api","context":"PaymentService","message":"Charging user","user_id":42,"request_id":"req-..."}
```

The level methods are `trace`, `debug`, `info`, `warn`, and `error`. Each returns a
record you refine with `field` (one structured key/value, repeatable) before `emit`.

### Logging an error

`error(&err)` attaches an error to a record. By default only the error's type name is
recorded; the top-level message and the source chain are opt-in (see
[error detail](#error-detail)).

```rust
self.logger.error("Payment failed").error(&err).field("order_id", id).emit();
```

A logged error is **not** redacted the way a `5xx` client response is, and an error's
`Display` or `source` text can carry sensitive data (a database driver may include a
connection string). Keep secrets out of error messages unless you have deliberately
enabled fuller detail in logs.

### Context overrides

The context is the struct name by default. Override it per field or per struct.

```rust
#[derive(Clone, Inject)]
pub struct OrderService {
    #[logger(context = "Orders")]
    logger: Logger,
}

#[derive(Clone, Inject)]
#[inject(context = "Billing")]
pub struct Invoices {
    logger: Logger,
}
```

A `Logger` injected directly into a handler has the default context. Derive one with a
context or a base field that rides on every record it emits.

```rust
let logger = base.for_context("Checkout").with_field("tenant", tenant_id);
logger.info("Checkout started").emit();
```

## Automatic logs

With no configuration, logging is on. The framework logs:

- startup: a line per mapped route and the bound address;
- each request after it completes, with method, path, route, status, duration, and the
  request id;
- lifecycle and error events.

Every log emitted while handling a request carries that request's id, so a service log
and the HTTP log share one `request_id`. Turn the request-completed log off with
`request_logs(false)`.

## Configuration

Logging is on by default: a developer console on a terminal, JSON otherwise, with the
level taken from `RUST_LOG` or `info`. Configure it with `App::logger`, typically driven
from settings so the same binary behaves differently per environment.

::code-group

```rust [src/config/logging.rs]
use tork::{settings, LogFormat, LoggerConfig};

#[settings]
pub struct LoggingConfig {
    #[setting(default = "info")] pub level: String,
    #[setting(default)]          pub format: LogFormat, // auto | pretty | compact | json
    #[setting(default = true)]   pub request_logs: bool,
}

impl LoggingConfig {
    pub fn to_logger(&self) -> LoggerConfig {
        LoggerConfig::new()
            .level(self.level.clone())
            .format(self.format)
            .service_name("my-api")
            .request_logs(self.request_logs)
    }
}
```

```rust [src/main.rs]
use tork::App;

use crate::config::Config;

#[tork::main]
async fn main() -> tork::Result<()> {
    let config = Config::load()?;
    App::new()
        .logger(config.logging.to_logger())
        .include_router(payments::router())
        .serve("0.0.0.0:8000")
        .await
}
```

::

`LoggerConfig` carries every logging option:

| Method | Effect |
| --- | --- |
| `level(directive)` | Maximum level (`trace`/`debug`/`info`/`warn`/`error`) or any `RUST_LOG`-style directive. |
| `format(LogFormat)` | Output format: `Auto` (default), `Pretty`, `Compact`, `Json`. |
| `color(bool)` | ANSI color in the console format. |
| `service_name(name)` | Service name reported in structured logs. |
| `error_detail(ErrorLogDetail)` | How much of a logged error is recorded. |
| `request_logs(bool)` | The automatic request-completed log. |
| `include_source(bool)` | Add the source file and line to each record. |
| `include_thread_ids(bool)` | Add the thread id to each record. |
| `non_blocking(bool)` | Write through a background worker. |
| `file(FileLogConfig)` | Add a rolling file sink. |
| `telemetry(TelemetryConfig)` | Add OpenTelemetry export (with the `otel` feature). |

`LogFormat::Auto` renders the console when standard output is a terminal and JSON
otherwise, so the same binary is readable in development and structured in production.

### Error detail

`error_detail` controls how much of an attached error reaches the log.

| Variant | Recorded |
| --- | --- |
| `ErrorLogDetail::TypeOnly` | The error's concrete type name only (the default). |
| `ErrorLogDetail::MessageOnly` | The type name and the top-level message. |
| `ErrorLogDetail::FullChain` | The type name, message, and the `source()` chain (capped at 16 entries). |

```rust
use tork::{ErrorLogDetail, LoggerConfig};

LoggerConfig::new().error_detail(ErrorLogDetail::MessageOnly);
```

The chain is bounded so a pathologically deep error cannot produce an unbounded record.

## Spans

Group an operation's logs and carry its fields with a span. Enter one for a synchronous
scope, or wrap a future so the span stays active across `await`.

```rust
// Synchronous scope: the guard exits the span when dropped.
let _scope = self.logger.span("import").field("batch", id).enter();

// Around a future: every log inside is grouped under the span.
let order = self
    .logger
    .instrument("create_order")
    .field("user_id", user_id)
    .run(self.orders.create(input))
    .await?;
```

## File output

Add a rolling file sink alongside the console. Files are always structured JSON.

```rust
use tork::{FileLogConfig, LoggerConfig, Rotation};

LoggerConfig::new().file(
    FileLogConfig::new("./logs")
        .prefix("api")
        .rotation(Rotation::Daily) // Never | Hourly | Daily
        .non_blocking(true),
);
```

## OpenTelemetry

With the `otel` feature enabled (`tork = { version = "0.1", features = ["otel"] }`),
export request spans and logs over OTLP to a collector.

```rust
use tork::{LoggerConfig, TelemetryConfig};

LoggerConfig::new()
    .telemetry(TelemetryConfig::new("http://localhost:4317").service_name("my-api"));
```

## Testing

Capture logs in tests with a `LogRecorder` on the test client and assert on them.

```rust
use tork::testing::{LogRecorder, TestClient};
use tork::assert_logs;

let recorder = LogRecorder::new();
let client = TestClient::builder(app).logger(recorder.clone()).build().await?;

client.get("/orders").send().await?;

assert!(recorder.contains_context("OrderService"));
assert_logs!(recorder, context = "OrderService", message = "Listing orders");
```

See [Testing](/docs/framework/fundamentals/testing) for the rest of the test client.
