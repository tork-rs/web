# Configuration

Most applications need configuration: a database URL, a port, feature flags, secrets.
Tork loads configuration once at startup into a typed, validated struct and injects it
like any other resource. There is no global singleton and no re-parsing per request. If a
value is missing or invalid, the application refuses to start with a clear message.

## Defining configuration

Annotate a struct with `#[settings]`. Each field becomes a setting, and `#[setting(...)]`
adds defaults and validation. The macro derives `Debug`, `Clone`, deserialization, and
validation, and generates a `load()` method.

```rust [src/config.rs]
use tork::{settings, SecretString};

#[settings(prefix = "APP", env_file = ".env")]
pub struct Config {
    #[setting(default = "Awesome API")]
    pub app_name: String,

    #[setting(email)]
    pub admin_email: String,

    #[setting(default = 50, ge = 1, le = 500)]
    pub items_per_user: u32,

    #[setting(secret)]
    pub api_key: SecretString,
}
```

`load()` reads the configured sources, fills in defaults, validates the result, and
returns `tork::Result<Config>`.

```rust
let config = Config::load()?;
```

## Loading and injecting

Load the configuration in `main`, before building the app, so it can also configure
things the app needs at startup (such as logging). Register it as `Arc<Config>` with
`App::state`, and every handler or service can then take `Arc<Config>` by type. Each
injection clones only a pointer.

::code-group

```rust [src/config.rs]
use tork::{settings, SecretString};

#[settings(prefix = "APP", env_file = ".env")]
pub struct Config {
    #[setting(default = "Awesome API")]
    pub app_name: String,

    #[setting(default = 50, ge = 1, le = 500)]
    pub items_per_user: u32,

    #[setting(secret, default = "dev-placeholder-key")]
    pub api_key: SecretString,
}
```

```rust [src/schemas/info.rs]
use tork::api_model;

#[api_model]
pub struct InfoOut {
    pub app_name: String,
    pub items_per_user: u32,
}
```

```rust [src/routers/info.rs]
use std::sync::Arc;

use tork::{api_router, get};

use crate::config::Config;
use crate::schemas::info::InfoOut;

#[api_router(tags = ["meta"])]
pub mod info_router {
    use super::*;

    #[get("/info", response_model = InfoOut)]
    pub async fn info(config: Arc<Config>) -> tork::Result<InfoOut> {
        Ok(InfoOut {
            app_name: config.app_name.clone(),
            items_per_user: config.items_per_user,
        })
    }
}

pub use info_router::router;
```

```rust [src/main.rs]
use std::sync::Arc;

use tork::App;

use crate::config::Config;
use crate::routers::info;

#[tork::main]
async fn main() -> tork::Result<()> {
    let config = Arc::new(Config::load()?);

    App::new()
        .state(config)
        .include_router(info::router())
        .serve("0.0.0.0:8000")
        .await
}
```

::

`Arc<Config>` injects through a built-in extractor, so a handler reads settings by type
and never reaches through a state object. A lifespan that needs configuration to build
other resources can take `Arc<Config>` the same way.

## Sources and precedence

`#[settings]` merges several sources. From lowest to highest precedence:

| # | Source | Set with |
| --- | --- | --- |
| 1 | Struct defaults | `#[setting(default = ...)]` |
| 2 | Base config file (TOML) | `config_file = "config/default.toml"` |
| 3 | Environment-specific files (TOML) | `files = ["config/{env}.toml"]` |
| 4 | `.env` file | `env_file = ".env"` |
| 5 | Environment variables | `prefix = "APP"` |
| 6 | Secrets directory | `secrets_dir = "secrets"` |
| 7 | Explicit overrides | `SettingsLoader::override_value(...)` |

A higher source overrides a lower one for any key it sets. The `.env` file only fills
variables that are not already set in the real environment, so a real environment
variable always wins over `.env`. A missing file is skipped, so config files are
optional; an absent value falls back to its default.

```rust
#[settings(
    prefix = "APP",
    env_file = ".env",
    config_file = "config/default.toml",
    files = ["config/{env}.toml"]
)]
pub struct Config { /* ... */ }
```

File paths are relative to the process's working directory. Under `cargo run -p <crate>`
that directory is the workspace root, not the crate, so run the binary from the crate's
own directory (or pass absolute paths) when you rely on a relative config file or `.env`.

### Environment variables and nesting

With `prefix = "APP"`, the loader reads variables that start with `APP_`. Levels of a
nested setting are separated by a double underscore, while a single underscore stays
inside a field name, so a field name that itself contains underscores is unambiguous.

```bash
APP_APP_NAME=Awesome API              # app_name
APP_ITEMS_PER_USER=100                # items_per_user
APP_SERVER__PORT=8080                 # server.port
APP_DATABASE__MAX_CONNECTIONS=20      # database.max_connections
```

Config files express the same nesting as tables:

```toml
[server]
port = 8080
```

### Environment-specific files

The `{env}` token in a `files` path is replaced with the current environment name, read
from `{PREFIX}_ENV` (here `APP_ENV`) and defaulting to `development`. So `APP_ENV=production`
loads `config/production.toml` on top of `config/default.toml`. With no prefix, the
environment name comes from `ENV`.

## Attribute reference

`#[settings(...)]` configures where values come from:

| Option | Effect |
| --- | --- |
| `prefix = "APP"` | Read environment variables beginning with `APP_`; nesting uses `__`. |
| `env_file = ".env"` | Load this dotenv file before reading the environment. |
| `config_file = "..."` | Merge a base TOML file. |
| `files = ["a.toml", "b.toml"]` | Merge environment-specific TOML files; `{env}` is substituted. |
| `secrets_dir = "secrets"` | Read one secret per file from a directory (file name is the key). |

`#[setting(...)]` configures one field:

| Option | Effect |
| --- | --- |
| `default = <expr>` | Value used when no source provides one (a string literal converts via `Into`). |
| `default` (bare flag) | Fill an absent value from the type's `Default`. |
| `ge` / `le` / `gt` / `lt` | Numeric bounds. |
| `min_length` / `max_length` | String length bounds. |
| `email` | Validate the value as an email address. |
| `custom = path` | Run a custom validator (repeatable). |
| `nested` | Validate a field that is itself a `#[settings]` struct, recursively. |
| `secret` | Hold the value as a `SecretString` (never logged, never validated). |

## Validation

`#[setting(...)]` accepts the same constraints as `#[api_model]`. Validation runs at the
end of `load()`, so a violation aborts the boot rather than failing a request later.

```rust
#[setting(default = 50, ge = 1, le = 500)]
pub items_per_user: u32,
```

A failed load returns an error with code `CONFIG_VALIDATION_ERROR` (listing the offending
fields), or `CONFIG_LOAD_FAILED` when a source cannot be parsed into the struct at all.
Returning it from `main` or a lifespan stops the application before it serves traffic.

## Secrets

A field marked `#[setting(secret)]` uses the `SecretString` type. Its `Debug` and
`Display` render a fixed mask, so a secret never lands in a log line by accident. The
value is readable only through `expose()`, which keeps every read site easy to audit.

```rust
use tork::{settings, SecretString};

#[settings(prefix = "APP")]
pub struct Config {
    #[setting(secret)]
    pub database_url: SecretString,
}

// config.database_url.expose()         -> the real value
// format!("{:?}", config.database_url)  -> SecretString("********")
```

A secret may still carry a default for local development, for example
`#[setting(secret, default = "dev-placeholder-key")]`, with the real value supplied
through the environment in production. Secrets are never validated and never serialized,
so a configuration holding them cannot be written back out.

## Nested configuration

Group related settings into their own `#[settings]` structs and mark the parent field
`#[setting(nested)]`. The whole tree still injects as one `Arc<Config>`.

```rust
#[settings]
pub struct ServerConfig {
    #[setting(default = "0.0.0.0")] pub host: String,
    #[setting(default = 8000)]      pub port: u16,
}

#[settings(prefix = "APP")]
pub struct Config {
    #[setting(default = "Awesome API")] pub app_name: String,
    #[setting(nested, default)]         pub server: ServerConfig,
}

// config.server.port
```

A bare `#[setting(nested)]` group is required: if no source supplies it, loading fails.
Add the `default` flag (`#[setting(nested, default)]`) to fill an absent group from its
own field defaults, which works when every field of the nested struct has a default (so
the group is itself default-constructible). Keep a group required when it holds a value
with no sensible default, such as a database URL.

## Loading without the macro

`#[settings]` builds on `SettingsLoader`, which you can use directly with any
`DeserializeOwned` and validatable type. This is handy in tests: skip the files and
environment and set values through `override_value`, the highest-priority source. The key
uses `.` between nesting levels.

```rust
use std::sync::Arc;

use tork::{App, SettingsLoader};

let config = SettingsLoader::<Config>::new()
    .prefix("APP")
    .override_value("items_per_user", 5u32)
    .override_value("server.port", 0u16)
    .load()?;

let app = App::new()
    .state(Arc::new(config))
    .include_router(info::router())
    .build()?;
```

Because the configuration is a resource rather than a global, a test supplies its own
value without touching process state, and several apps in one process can each load their
own.
