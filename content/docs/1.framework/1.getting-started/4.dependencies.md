# Dependencies

A handler declares what it needs as arguments, and Tork provides them. Path
parameters are matched by name, and everything else is resolved by type: a shared
resource, a service object, or a value you compute from the request. This keeps
handlers small and means a function never reaches into a global to find what it
needs.

## Resources

A resource is a value that lives for the whole life of the application, such as a
database pool or an in-memory store. You create it once at startup and Tork hands a
clone to any handler that asks for it.

Define the resource type. It must be `Clone`, so wrap anything expensive in an
`Arc` to keep cloning cheap.

```rust
use std::collections::HashMap;
use std::sync::Arc;

use tork::{api_model, Error};

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}

#[derive(Clone)]
pub struct UserStore {
    users: Arc<HashMap<i64, UserOut>>,
}

impl UserStore {
    pub fn seed() -> Self {
        let users = HashMap::from([
            (1, UserOut { id: 1, name: "Ada".to_owned() }),
        ]);
        Self { users: Arc::new(users) }
    }

    pub fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.users
            .get(&id)
            .cloned()
            .ok_or_else(|| Error::not_found("user not found"))
    }
}
```

Group your resources in a struct that derives `Resources`, and mark each resource
field with `#[resource]`. A `#[tork::lifespan]` block builds the resources at
startup and releases them at shutdown.

```rust
use tork::{Resources, LifespanContext, Logger, Result};

#[derive(Clone, Resources)]
pub struct AppState {
    #[resource]
    pub store: UserStore,
}

#[tork::lifespan]
impl AppState {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        Ok(AppState { store: UserStore::seed() })
    }

    async fn shutdown(self) -> Result<()> {
        Logger::new("Lifecycle").info("application stopped").emit();
        Ok(())
    }
}
```

Register the lifespan on the application, then ask for the resource by type in a
handler.

```rust
use tork::{api_router, get, App};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}", response_model = UserOut)]
    pub async fn get_one(user_id: i64, store: UserStore) -> tork::Result<UserOut> {
        store.get(user_id)
    }
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .lifespan::<AppState>()
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

The `store: UserStore` argument is filled in on every request. The handler does not
know or care where the store came from.

## Services

Most applications put their logic in services rather than in the handler. A service
that derives `Inject` can take other dependencies as fields, and Tork builds it for
you when a handler asks for it.

```rust
use tork::Inject;

#[derive(Inject)]
pub struct UserService {
    store: UserStore,
}

impl UserService {
    pub async fn get_user(&self, id: i64) -> tork::Result<UserOut> {
        self.store.get(id)
    }
}
```

Now a handler can take the service directly. Tork sees that `UserService` needs a
`UserStore`, resolves the store from the resources, builds the service, and passes
it in.

```rust
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_one(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get_user(user_id).await
}
```

This works to any depth. A service can inject a repository, which injects the store,
and you only ask for the service.

## Custom dependencies

Sometimes a dependency is a value you compute from the request, such as the
authenticated user behind a bearer token. Write a `#[tork::dependency]` block with a
`resolve` function. Its arguments are themselves dependencies, so you can combine a
header extractor with a service.

```rust
use tork::{BearerToken, Error};

#[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
}

#[tork::dependency]
impl CurrentUser {
    async fn resolve(token: BearerToken, service: UserService) -> tork::Result<Self> {
        let user = service.get_user(1).await?;
        // A real service would verify the token; here we just check it is present.
        let _ = token.token();
        Ok(CurrentUser { id: user.id })
    }
}
```

`CurrentUser` is now a dependency like any other. If the `Authorization` header is
missing, `BearerToken` fails first and the handler never runs.

```rust
#[get("/me", response_model = UserOut)]
pub async fn me(current: CurrentUser, store: UserStore) -> tork::Result<UserOut> {
    store.get(current.id)
}
```
