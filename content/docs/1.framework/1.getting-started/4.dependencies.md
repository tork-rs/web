# Dependencies

A handler lists what it needs as arguments, and Tork provides each one. Instead of
building the current user, opening a service, or reading a header inside every
function, you name the type you want and Tork resolves it before the handler runs.

There is no special marker to wrap a dependency. The type is the dependency. If a
type knows how to build itself from a request, you can ask for it in any handler,
and you can ask for as many as you need.

## How arguments are resolved

A handler has two kinds of arguments:

- **Path parameters** are matched by name to the `{...}` parts of the path, like
  `user_id` below.
- **Everything else** is resolved by type. Tork looks at the type, builds it from
  the request, and passes it in.

```rust
use tork::{api_router, get, BearerToken};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64, token: BearerToken) -> tork::Result<i64> {
        // `user_id` comes from the path, `token` is resolved by type.
        let _ = token.token();
        Ok(user_id)
    }
}
```

## Built-in dependencies

Tork ships with several dependencies you can use right away. `BearerToken` reads the
`Authorization: Bearer ...` header, and `Logger` gives you a logger that already
knows about the current request.

```rust
use tork::{api_router, get, BearerToken, Logger};

#[api_router(prefix = "/me", tags = ["me"])]
pub mod me {
    use super::*;

    #[get("/")]
    pub async fn whoami(token: BearerToken, logger: Logger) -> tork::Result<&'static str> {
        logger.info("checking the caller").emit();
        let _ = token.token();
        Ok("ok")
    }
}
```

If the `Authorization` header is missing, `BearerToken` fails first and the request
is answered with `401` before your function body runs.

## Writing your own dependency

This is where dependencies earn their keep. Write a piece of request logic once,
give it a type, and reuse it across every route that needs it. Use
`#[tork::dependency]` with a `resolve` function.

```rust
use tork::{BearerToken, Error};

/// The authenticated user for the current request.
#[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
}

#[tork::dependency]
impl CurrentUser {
    async fn resolve(token: BearerToken) -> tork::Result<Self> {
        // A real app verifies a signed token here. This example maps the token
        // text to a user id.
        let id = token
            .token()
            .parse::<i64>()
            .map_err(|_| Error::unauthorized("invalid token"))?;
        Ok(CurrentUser { id })
    }
}
```

`CurrentUser` is now a dependency like any other. Ask for it in a handler, and the
`resolve` function runs first.

```rust
#[get("/profile")]
pub async fn profile(current: CurrentUser) -> tork::Result<i64> {
    Ok(current.id)
}
```

Write the rule once, use it in as many routes as you like. If the token logic
changes, you change it in one place.

## Dependencies that use other dependencies

The arguments of `resolve` are themselves dependencies, and Tork resolves them
first. So a dependency can build on other dependencies, to any depth.

Here `AdminUser` depends on `CurrentUser`, which depends on `BearerToken`.

```rust
use tork::Error;

#[derive(Clone)]
pub struct AdminUser {
    pub id: i64,
}

#[tork::dependency]
impl AdminUser {
    async fn resolve(current: CurrentUser) -> tork::Result<Self> {
        if current.id != 1 {
            return Err(Error::forbidden("admins only"));
        }
        Ok(AdminUser { id: current.id })
    }
}
```

When a handler asks for `AdminUser`, Tork resolves the chain in order:
`BearerToken`, then `CurrentUser`, then `AdminUser`. You get all three steps from a
single argument.

```rust
#[get("/admin/stats")]
pub async fn stats(admin: AdminUser) -> tork::Result<i64> {
    Ok(admin.id)
}
```

## Dependencies as guards

Because `resolve` returns a `Result`, a dependency can stop the request. If it
returns an error, the handler never runs and the error becomes the response.
`AdminUser` above is a guard: any route that asks for it is admin only, and the
check lives in the dependency rather than in each handler. Authentication, access
checks, and feature flags all fit this shape.

## Services as dependencies

Plain values fit `#[tork::dependency]`. For a service object that holds logic and
its own dependencies, derive `Inject` instead. Each field of the service is a
dependency, and Tork builds the whole thing for you.

The service, its repository, and the handler each live in their own file. The
repository derives `Inject` too and depends on a long-lived resource, such as a
database handle or an in-memory store, which you set up in the
[Lifespan](/docs/framework/getting-started/lifespan) guide.

::code-group

```rust [src/services/user_service.rs]
use tork::Inject;

#[derive(Inject)]
pub struct UserService {
    repo: UserRepository,
}

impl UserService {
    pub async fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.repo.find(id).await
    }
}
```

```rust [src/repositories/user_repository.rs]
use tork::Inject;

#[derive(Inject)]
pub struct UserRepository {
    store: UserStore,
}

impl UserRepository {
    pub async fn find(&self, id: i64) -> tork::Result<UserOut> {
        self.store.get(id)
    }
}
```

```rust [src/routers/users.rs]
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_user(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get(user_id).await
}
```

::

A handler asks only for the service. Tork resolves the store, builds the repository,
builds the service, and passes it in. It is the same resolution you already saw,
applied to your own types: ask for the top of the chain and Tork assembles the rest.

## Reusing a dependency

Because a dependency is just a type, using it in several routes is nothing special.
The shared logic stays in one place.

```rust
#[get("/orders")]
pub async fn orders(current: CurrentUser) -> tork::Result<&'static str> {
    let _ = current.id;
    Ok("orders")
}

#[get("/settings")]
pub async fn settings(current: CurrentUser) -> tork::Result<&'static str> {
    let _ = current.id;
    Ok("settings")
}
```

Both routes get the same authenticated `CurrentUser`, and the token logic is written
once.
