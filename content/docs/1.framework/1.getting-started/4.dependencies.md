# Dependencies

A handler lists what it needs as arguments, and Tork provides each one. Instead of
building the current user, opening a service, or reading a header inside every
function, you name the type you want and Tork resolves it before the handler runs.

There is no special marker to wrap a dependency. The type is the dependency. If a
type knows how to build itself from a request, you can ask for it in any handler,
and you can ask for as many as you need.

## How arguments are resolved

A handler has two kinds of arguments:

- **Path parameters** are matched by name to the `{...}` parts of the path, like
  `user_id` below.
- **Everything else** is resolved by type. Tork looks at the type, builds it from
  the request, and passes it in.

```rust
use tork::{api_router, get, BearerToken};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64, token: BearerToken) -> tork::Result<i64> {
        // `user_id` comes from the path, `token` is resolved by type.
        let _ = token.token();
        Ok(user_id)
    }
}
```

## Built-in dependencies

Tork ships with several dependencies you can use right away. `BearerToken` reads the
`Authorization: Bearer ...` header, and `Logger` gives you a logger that already
knows about the current request.

```rust
use tork::{api_router, get, BearerToken, Logger};

#[api_router(prefix = "/me", tags = ["me"])]
pub mod me {
    use super::*;

    #[get("/")]
    pub async fn whoami(token: BearerToken, logger: Logger) -> tork::Result<&'static str> {
        logger.info("checking the caller").emit();
        let _ = token.token();
        Ok("ok")
    }
}
```

If the `Authorization` header is missing, `BearerToken` fails first and the request
is answered with `401` before your function body runs.

## Writing your own dependency

This is where dependencies earn their keep. Write a piece of request logic once,
give it a type, and reuse it across every route that needs it. Use
`#[tork::dependency]` with a `resolve` function.

```rust
use tork::{BearerToken, Error};

/// The authenticated user for the current request.
#[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
}

#[tork::dependency]
impl CurrentUser {
    async fn resolve(token: BearerToken) -> tork::Result<Self> {
        // A real app verifies a signed token here. This example maps the token
        // text to a user id.
        let id = token
            .token()
            .parse::<i64>()
            .map_err(|_| Error::unauthorized("invalid token"))?;
        Ok(CurrentUser { id })
    }
}
```

`CurrentUser` is now a dependency like any other. Ask for it in a handler, and the
`resolve` function runs first.

```rust
#[get("/profile")]
pub async fn profile(current: CurrentUser) -> tork::Result<i64> {
    Ok(current.id)
}
```

Write the rule once, use it in as many routes as you like. If the token logic
changes, you change it in one place.

## Dependencies that use other dependencies

The arguments of `resolve` are themselves dependencies, and Tork resolves them
first. So a dependency can build on other dependencies, to any depth.

Here `AdminUser` depends on `CurrentUser`, which depends on `BearerToken`.

```rust
use tork::Error;

#[derive(Clone)]
pub struct AdminUser {
    pub id: i64,
}

#[tork::dependency]
impl AdminUser {
    async fn resolve(current: CurrentUser) -> tork::Result<Self> {
        if current.id != 1 {
            return Err(Error::forbidden("admins only"));
        }
        Ok(AdminUser { id: current.id })
    }
}
```

When a handler asks for `AdminUser`, Tork resolves the chain in order:
`BearerToken`, then `CurrentUser`, then `AdminUser`. You get all three steps from a
single argument.

```rust
#[get("/admin/stats")]
pub async fn stats(admin: AdminUser) -> tork::Result<i64> {
    Ok(admin.id)
}
```

## Dependencies as guards

Because `resolve` returns a `Result`, a dependency can stop the request. If it
returns an error, the handler never runs and the error becomes the response.
`AdminUser` above is a guard: any route that asks for it is admin only, and the
check lives in the dependency rather than in each handler. Authentication, access
checks, and feature flags all fit this shape.

## Services as dependencies

Plain values fit `#[tork::dependency]`. For a service object that holds logic and
its own dependencies, derive `Inject` instead. Each field of the service is a
dependency, and Tork builds the whole thing for you.

```rust
use tork::Inject;

#[derive(Inject)]
pub struct UserService {
    repo: UserRepository,
}

impl UserService {
    pub async fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.repo.find(id).await
    }
}
```

The repository can derive `Inject` too and depend on a long-lived resource, such as
a database handle or an in-memory store, which you set up in the
[Lifespan](/docs/framework/getting-started/lifespan) guide.

```rust
use tork::Inject;

#[derive(Inject)]
pub struct UserRepository {
    store: UserStore,
}

impl UserRepository {
    pub async fn find(&self, id: i64) -> tork::Result<UserOut> {
        self.store.get(id)
    }
}
```

A handler asks only for the service. Tork resolves the store, builds the repository,
builds the service, and passes it in.

```rust
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_user(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get(user_id).await
}
```

It is the same resolution you already saw, applied to your own types: ask for the
top of the chain and Tork assembles the rest.

## Reusing a dependency

Because a dependency is just a type, using it in several routes is nothing special.
The shared logic stays in one place.

```rust
#[get("/orders")]
pub async fn orders(current: CurrentUser) -> tork::Result<&'static str> {
    let _ = current.id;
    Ok("orders")
}

#[get("/settings")]
pub async fn settings(current: CurrentUser) -> tork::Result<&'static str> {
    let _ = current.id;
    Ok("settings")
}
```

Both routes get the same authenticated `CurrentUser`, and the token logic is written
once.

Services and stores have to come from somewhere. The

long-lived resources at startup and inject them like any other dependency.

## 4. Extractors and dependency injection

Every handler parameter that is not a path parameter is resolved from the request
through the `FromRequest` trait. Tork resolves them in declaration order before
calling the handler.

### Built-in extractors

- `State<S>`: a clone of a registered application state value.
- `BearerToken`: the token from an `Authorization: Bearer ...` header.
- `Json<T>`: the request body parsed as JSON into `T`.
- `Valid<T>`: the request body parsed as JSON and validated (see chapter 5).

At most one parameter may consume the request body.

### Application state

State is registered on the `App` and read with `State<S>`. The value is cloned per
request, so it should be cheap to clone (hold pools and handles behind `Arc`):

```rust
use std::collections::HashMap;
use std::sync::Arc;

use tork::{Error, Result, State, api_router, get};

use crate::models::user::UserOut;

##[derive(Clone)]
pub struct AppState {
    users: Arc<HashMap<i64, UserOut>>,
}

impl AppState {
    pub fn get_user(&self, id: i64) -> Result<UserOut> {
        self.users
            .get(&id)
            .cloned()
            .ok_or_else(|| Error::not_found("user not found"))
    }
}

##[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("/{user_id}", response_model = UserOut, summary = "Get user by id")]
    pub async fn get_user(user_id: i64, state: State<AppState>) -> Result<UserOut> {
        state.0.get_user(user_id)
    }
}
```

Register the value when building the app: `App::new().state(AppState::boot().await?)`.

### Custom dependencies

A dependency is any type that knows how to build itself from the request. Annotate
an `impl` block that has an async `resolve` function with `#[tork::dependency]`:

```rust
use tork::{State, dependency};

use crate::core::app_state::AppState;
use crate::models::user::UserOut;

##[derive(Clone)]
pub struct UserRepository {
    state: AppState,
}

##[tork::dependency]
impl UserRepository {
    pub async fn resolve(state: State<AppState>) -> tork::Result<Self> {
        Ok(Self { state: state.0 })
    }
}

impl UserRepository {
    pub async fn find_by_id(&self, id: i64) -> tork::Result<UserOut> {
        self.state.get_user(id)
    }
}
```

`resolve` may take other dependencies as parameters. They are resolved first, then
`resolve` runs. This makes dependency graphs compose:

```rust
pub struct UserService {
    users: UserRepository,
}

##[tork::dependency]
impl UserService {
    pub async fn resolve(users: UserRepository) -> tork::Result<Self> {
        Ok(Self { users })
    }
}
```

Now a handler can simply ask for the service:

```rust
##[get("/{user_id}", response_model = UserOut)]
pub async fn get_user(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get_user(user_id).await
}
```

Resolving `UserService` builds a `UserRepository`, which reads `State<AppState>`.

### Authentication example

`BearerToken` and a dependency combine into a current-user guard:

```rust
use tork::{BearerToken, Error, dependency};

use crate::repositories::user_repository::UserRepository;

##[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
    pub email: String,
}

##[tork::dependency]
impl CurrentUser {
    pub async fn resolve(token: BearerToken, users: UserRepository) -> tork::Result<Self> {
        let user_id = users.authenticate(token.token())?;
        let user = users.find_by_id(user_id).await?;
        Ok(CurrentUser { id: user.id, email: user.email })
    }
}

impl CurrentUser {
    pub fn ensure_can_access_user(&self, user_id: i64) -> tork::Result<()> {
        if self.id != user_id {
            return Err(Error::forbidden("access denied"));
        }
        Ok(())
    }
}
```

A handler that lists a user's orders can now require authentication and authorize
the path:

```rust
##[get("/", response_model = Vec<OrderOut>, summary = "List orders for a user")]
pub async fn list_user_orders(
    user_id: i64,
    current_user: CurrentUser,
    service: OrderService,
) -> tork::Result<Vec<OrderOut>> {
    current_user.ensure_can_access_user(user_id)?;
    service.list_orders_for_user(user_id).await
}
```

A missing or malformed token yields `401`, a forbidden access yields `403`.
