# Error Handling and Exceptions

Tork has no exceptions in the throwing sense. An error is a value: a handler returns
`tork::Result<T>`, and on failure you return a `tork::Error` that the framework turns
into an HTTP response with the right status code and a JSON body. Because the return
type is a `Result`, errors flow up with the `?` operator, and a typed *exception handler*
can map a specific error into a response of your choosing.

## Raising an error

Build an error with the constructor that fits the situation, pass a message, and return
it. You can return it directly with `Err(...)`, or let it propagate from a fallible call
with `?`.

```rust [src/routers/users.rs]
use tork::{api_router, get, Error};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64) -> tork::Result<i64> {
        if user_id < 1 {
            // Return an error directly.
            return Err(Error::bad_request("user id must be positive"));
        }
        if user_id > 100 {
            return Err(Error::not_found("user not found"));
        }
        Ok(user_id)
    }
}
```

Any function that returns `tork::Result` composes with `?`, so you can move a check into
a helper and let the error surface on its own.

```rust [src/routers/search.rs]
use tork::{api_router, get, Error};

fn parse_page(raw: &str) -> tork::Result<u32> {
    raw.parse::<u32>()
        .map_err(|_| Error::bad_request("page must be a number"))
}

#[api_router(tags = ["search"])]
pub mod search_router {
    use super::*;

    #[get("/page/{raw}")]
    pub async fn page(raw: String) -> tork::Result<u32> {
        let page = parse_page(&raw)?; // a bad value becomes a 400 here
        Ok(page)
    }
}
```

## The error kinds

Each constructor maps to one status, a stable machine `code`, and the standard reason
phrase used as the `title`. These are every kind Tork ships.

| Constructor | Status | `code` | `title` | Use it when |
| --- | --- | --- | --- | --- |
| `Error::bad_request` | 400 | `BAD_REQUEST` | Bad Request | the request is malformed or a value cannot be parsed |
| `Error::unauthorized` | 401 | `UNAUTHORIZED` | Unauthorized | authentication is missing or invalid |
| `Error::forbidden` | 403 | `FORBIDDEN` | Forbidden | the caller is authenticated but not allowed |
| `Error::not_found` | 404 | `NOT_FOUND` | Not Found | the resource does not exist |
| `Error::method_not_allowed` | 405 | `METHOD_NOT_ALLOWED` | Method Not Allowed | the path exists but not for this method |
| `Error::conflict` | 409 | `CONFLICT` | Conflict | the request conflicts with current state (a duplicate, a version clash) |
| `Error::payload_too_large` | 413 | `PAYLOAD_TOO_LARGE` | Payload Too Large | the request body exceeds the allowed size |
| `Error::unprocessable` | 422 | `UNPROCESSABLE_ENTITY` | Unprocessable Entity | the body is well-formed but semantically invalid |
| `Error::too_many_requests` | 429 | `TOO_MANY_REQUESTS` | Too Many Requests | the caller is being rate limited |
| `Error::internal` | 500 | `INTERNAL_SERVER_ERROR` | Internal Server Error | an unexpected server-side failure |
| `Error::service_unavailable` | 503 | `SERVICE_UNAVAILABLE` | Service Unavailable | a dependency is down or the server is overloaded |
| `Error::gateway_timeout` | 504 | `GATEWAY_TIMEOUT` | Gateway Timeout | an upstream call timed out |

A failed body validation is produced for you by `Valid<T>`: it is a `422` with the code
`VALIDATION_ERROR` and one `details` entry per offending field. See
[Request Data](/docs/framework/getting-started/request-data) for how that is generated.

## Every other status

Every other standard HTTP error status has a named constructor too, used exactly like
the common ones above. Each sets its status and reason phrase, and a machine `code` that
matches it.

| Constructor | Status | Constructor | Status |
| --- | --- | --- | --- |
| `Error::payment_required` | 402 | `Error::too_early` | 425 |
| `Error::not_acceptable` | 406 | `Error::upgrade_required` | 426 |
| `Error::proxy_authentication_required` | 407 | `Error::precondition_required` | 428 |
| `Error::request_timeout` | 408 | `Error::request_header_fields_too_large` | 431 |
| `Error::gone` | 410 | `Error::unavailable_for_legal_reasons` | 451 |
| `Error::length_required` | 411 | `Error::not_implemented` | 501 |
| `Error::precondition_failed` | 412 | `Error::bad_gateway` | 502 |
| `Error::uri_too_long` | 414 | `Error::http_version_not_supported` | 505 |
| `Error::unsupported_media_type` | 415 | `Error::variant_also_negotiates` | 506 |
| `Error::range_not_satisfiable` | 416 | `Error::insufficient_storage` | 507 |
| `Error::expectation_failed` | 417 | `Error::loop_detected` | 508 |
| `Error::im_a_teapot` | 418 | `Error::not_extended` | 510 |
| `Error::misdirected_request` | 421 | `Error::network_authentication_required` | 511 |
| `Error::locked` | 423 | | |
| `Error::failed_dependency` | 424 | | |

```rust
return Err(Error::im_a_teapot("this endpoint only brews tea"));
```

For a non-standard code, or when you would rather pass the number directly, use
`Error::http` with the numeric status and a message. The status sets the response status
and reason phrase, while the machine `code` defaults from the nearest kind and can be set
with `with_code`.

```rust
use tork::Error;

return Err(Error::http(499, "client closed request").with_code("CLIENT_CLOSED"));
```

A `5xx` status (whether from a named constructor or `Error::http`) is still redacted in
the response body, the same as `Error::internal`. An out-of-range number falls back to
`500`.

## The error response

Every error renders to a flat JSON body with a consistent shape:

```json
{
  "status": 404,
  "code": "NOT_FOUND",
  "title": "Not Found",
  "message": "user not found",
  "traceId": "req-d01618f2-f39c-439e-93fd-21db43ec1cbd",
  "timestamp": "2026-06-10T11:08:07Z"
}
```

- `status` is the numeric HTTP status, and `title` is its reason phrase.
- `code` is the stable identifier from the table above. Override it per error with
  `Error::with_code("USER_NOT_FOUND")` when you want a more specific machine code.
- `details` appears only when there are field-level errors. You can attach your own.
- `traceId` is generated for each error response and is also logged for server errors,
  so an id a client reports can be matched against the logs.
- `timestamp` is the time of the response in RFC 3339 UTC.

Error responses are sent with `Cache-Control: no-store`, so a proxy or browser never
caches a `401` or `500`.

## Server errors are redacted

For any `5xx`, the body never exposes the internal message. The client receives a generic
`"Internal server error"` while the real message and any attached cause are logged with
the trace id. Attach the underlying cause with `with_source` so it reaches the logs.

```rust
// The client sees "Internal server error"; the log records the real detail and the trace id.
Err(Error::internal("database password is wrong").with_source(db_error))
```

Field `details` are likewise omitted from `5xx` bodies. This keeps internal failures from
leaking to clients while preserving what you need to debug them.

## Adding detail to an error

Three builder methods refine an error before you return it, and they chain.

| Method | Purpose |
| --- | --- |
| `with_code(code)` | Replace the machine `code` with your own stable identifier. |
| `with_details(details)` | Attach field-level entries (surfaced in `4xx` bodies). |
| `with_source(cause)` | Attach the underlying error for the logs (never serialized). |

```rust
use tork::{Error, ErrorDetail};

Err(Error::unprocessable("the order could not be created")
    .with_code("ORDER_INVALID")
    .with_details(vec![
        ErrorDetail::new("quantity", "TOO_SMALL", "must be at least 1"),
    ]))
```

An `ErrorDetail` carries a `field`, an `issue` code, and a `message`.

## Your own error types

Most applications have their own error type in the data or service layer. Derive
`AppError` on it and set a default status with `#[status(...)]`. This generates a
conversion into `tork::Error`, so a function can surface your error through `?`. The
`#[status(...)]` value is either an HTTP status code (such as `503`) or an error kind
name (such as `ServiceUnavailable`), and the original value is kept as the error's cause
so it can be mapped later.

`AppError` requires the type to implement `Display` (its text becomes the error message)
and `std::error::Error`.

::code-group

```rust [src/core/errors.rs]
#[derive(Debug, tork::AppError)]
#[status(503)]
pub enum RepoError {
    Unavailable,
}

impl std::fmt::Display for RepoError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            RepoError::Unavailable => f.write_str("the data store is unavailable"),
        }
    }
}

impl std::error::Error for RepoError {}
```

```rust [src/routers/demo.rs]
use tork::{api_router, get};

use crate::core::errors::RepoError;

#[api_router(prefix = "/demo", tags = ["demo"])]
pub mod demo_router {
    use super::*;

    #[get("/db")]
    pub async fn db() -> tork::Result<i64> {
        let outcome: Result<i64, RepoError> = Err(RepoError::Unavailable);
        // `?` converts RepoError into a 503 tork::Error.
        Ok(outcome?)
    }
}
```

## Mapping an error to a custom response

Sometimes you want full control over how a specific error type becomes a response, for
example to hide internal detail or to log it. Register an *exception handler* for the type
on the application. It receives the recovered error value and an `ErrorContext`, and
returns a response. The handler fires whenever an error carrying a source of that type
reaches the boundary, which is exactly what the `AppError` conversion attaches.

```rust [src/main.rs]
use tork::{App, ErrorContext, IntoResponse, Logger, Response};

use crate::core::errors::RepoError;

async fn handle_repo_error(error: RepoError, ctx: ErrorContext) -> Response {
    Logger::new("ExceptionHandler")
        .error("repository error")
        .field("path", ctx.path().to_owned())
        .error(&error)
        .emit();
    tork::Error::service_unavailable("the service is temporarily unavailable")
        .into_response()
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .include_router(demo::router())
        .exception_handler::<RepoError, _, _>(handle_repo_error)
        .serve("0.0.0.0:8000")
        .await
}
```

The `ErrorContext` exposes the request `method`, `path`, matched `route`, and
`request_id`. Registering a handler for a type again replaces the previous one. Now any
`RepoError` that reaches the boundary is logged and answered with a clean message, while
the internal detail stays in your logs.
