# Request Data

Handlers read input through their arguments. A path parameter comes from the URL, a
typed body comes from `Valid<T>` (or `Json<T>` when you do not need validation), and a
form comes from `Form<T>` or `Multipart<T>`. This page covers bodies, validation, and
forms.

## Defining a model

A model is a struct marked with `#[api_model]`. The attribute derives `Debug`,
`Clone`, serialization, validation, and a JSON Schema in one place, so the same type
works for both input and output and shows up in the OpenAPI document.

```rust [src/schemas/user.rs]
use tork::api_model;

#[api_model]
pub struct CreateUser {
    pub name: String,
    pub email: String,
}
```

By default the JSON field names match the Rust field names. To use a different casing
in JSON, add `rename_all` on the model.

```rust [src/schemas/user.rs]
#[api_model(rename_all = "camelCase")]
pub struct UserOut {
    pub id: i64,
    pub display_name: String,
}
```

Here `display_name` is sent as `displayName` in the response, in the schema, and when
reading input.

## Reading a JSON body

Ask for the body as `Valid<T>`. Tork reads the JSON, turns it into your model, runs
validation, and gives you the value through `into_inner`.

::code-group

```rust [src/schemas/user.rs]
use tork::api_model;

#[api_model]
pub struct CreateUser {
    pub name: String,
    pub email: String,
}

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}
```

```rust [src/routers/users.rs]
use tork::{api_router, post, Valid};

use crate::schemas::user::{CreateUser, UserOut};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[post("/", response_model = UserOut, status_code = 201)]
    pub async fn create(body: Valid<CreateUser>) -> tork::Result<UserOut> {
        let input = body.into_inner();
        Ok(UserOut { id: 1, name: input.name })
    }
}

pub use users_router::router;
```

::

A request with a body that is not valid JSON, or that is missing a field, is rejected
with `422` before the handler runs. At most one argument may read the body, since the
body is consumed once.

When you only need the parsed body and no validation, use `Json<T>` instead. It
deserializes the body into `T` but skips the validation step.

```rust [src/routers/users.rs]
use tork::{post, Json};

use crate::schemas::user::CreateUser;

#[post("/raw")]
pub async fn create_raw(body: Json<CreateUser>) -> tork::Result<&'static str> {
    let _input = body.0;
    Ok("ok")
}
```

To guard against memory-exhaustion, the body is buffered with a size cap (set with
`App::max_request_body_size`) and rejected if it nests too deeply. Both cases return a
`400` before deserialization.

## Validating fields

Add rules to a field with `#[field(...)]`. The body is checked against these rules, and
a value that breaks one is rejected with `422` and a message that names the field.

```rust [src/schemas/order.rs]
use tork::api_model;

#[api_model]
pub struct CreateOrder {
    /// Must be between 1 and 120 characters.
    #[field(min_length = 1, max_length = 120)]
    pub name: String,

    /// Optional, at most 300 characters when present.
    #[field(max_length = 300)]
    pub description: Option<String>,

    /// Must be greater than zero.
    #[field(gt = 0)]
    pub price: f64,

    /// Must not be negative when present.
    #[field(ge = 0)]
    pub tax: Option<f64>,
}
```

The built-in rules are `min_length` and `max_length` for strings, and `gt`, `ge`,
`lt`, `le` for numbers. `title` and `description` add documentation to the schema. An
`Option<T>` field is only checked when a value is present, so optional fields stay
optional.

A failing body produces `422` with one entry per offending field:

```json
{
  "status": 422,
  "code": "VALIDATION_ERROR",
  "title": "Unprocessable Entity",
  "message": "The submitted data failed validation.",
  "details": [
    { "field": "name", "issue": "TOO_SHORT", "message": "length is lower than 1" },
    { "field": "price", "issue": "TOO_SMALL", "message": "must be greater than 0" }
  ],
  "traceId": "req-...",
  "timestamp": "2026-06-10T11:08:07Z"
}
```

The full error shape is covered in
[Error Handling](/docs/framework/getting-started/error-handling-and-exceptions).

### Custom validators

For a rule the built-in constraints do not cover, write a function and point `custom`
at it. The function performs the check and supplies its own message. It uses `garde`,
the validation crate behind `#[api_model]`. You can repeat `custom` for several checks.

```rust [src/schemas/tag.rs]
use tork::api_model;

fn not_blank(value: &str, _ctx: &()) -> garde::Result {
    if value.trim().is_empty() {
        Err(garde::Error::new("must not be blank"))
    } else {
        Ok(())
    }
}

#[api_model]
pub struct CreateTag {
    #[field(min_length = 1, custom = not_blank)]
    pub label: String,
}
```

### Nested models and defaults

A field that is itself a model is serialized and documented automatically. To also
validate it recursively, mark it `nested`. A field marked `default` falls back to the
type's `Default` when it is absent from the body.

```rust [src/schemas/item.rs]
use std::collections::HashSet;

use tork::api_model;

#[api_model]
pub struct Image {
    #[field(min_length = 1)]
    pub url: String,
    pub name: String,
}

#[api_model(rename_all = "camelCase")]
pub struct Item {
    #[field(min_length = 1)]
    pub name: String,

    pub price: f64,

    #[field(default)]
    pub tags: HashSet<String>,

    #[field(nested)]
    pub image: Option<Image>,
}
```

Validating an `Item` now also validates the nested `Image`, and an absent `tags`
becomes an empty set.

## Path parameters

Path parameters are read straight from the URL by matching the argument name to the
name in braces. There is no extractor type to wrap them.

```rust [src/routers/users.rs]
#[get("/{user_id}")]
pub async fn get_one(user_id: i64) -> tork::Result<i64> {
    Ok(user_id)
}
```

A value that does not parse into the declared type is rejected with `422`.

## Forms

For an HTML form submitted as `application/x-www-form-urlencoded`, mark the model with
`#[api_model]` and read it with `Form<T>`. The same field rules apply and run before
the handler.

::code-group

```rust [src/schemas/auth.rs]
use tork::api_model;

#[api_model]
pub struct LoginForm {
    pub username: String,
    pub password: String,
}
```

```rust [src/routers/auth.rs]
use tork::{api_router, post, Form};

use crate::schemas::auth::LoginForm;

#[api_router(prefix = "/auth", tags = ["auth"])]
pub mod auth_router {
    use super::*;

    #[post("/login")]
    pub async fn login(form: Form<LoginForm>) -> tork::Result<&'static str> {
        let form = form.into_inner();
        let _ = (form.username, form.password);
        Ok("ok")
    }
}

pub use auth_router::router;
```

::

## File uploads

For uploads sent as `multipart/form-data`, use a model that derives `FormModel` and
read it with `Multipart<T>`. File fields are marked with `#[file]`, where you can cap
the size and limit the content types; text fields keep their `#[field]` rules.

::code-group

```rust [src/schemas/profile.rs]
use tork::{FileBytes, FormModel};

#[derive(FormModel)]
pub struct ProfileForm {
    #[file(max_size = "2MB", content_types = ["image/png", "image/jpeg"], sniff = true)]
    pub avatar: FileBytes,

    #[field(min_length = 2, max_length = 40)]
    pub display_name: String,
}
```

```rust [src/routers/profile.rs]
use tork::{api_router, post, Multipart};

use crate::schemas::profile::ProfileForm;

#[api_router(prefix = "/profile", tags = ["profile"])]
pub mod profile_router {
    use super::*;

    #[post("/")]
    pub async fn upload(form: Multipart<ProfileForm>) -> tork::Result<usize> {
        let form = form.into_inner();
        Ok(form.avatar.len())
    }
}

pub use profile_router::router;
```

::

`sniff = true` checks the real bytes of the file against the declared content types, so
a file cannot lie about its type through its name alone.

A `FileBytes` field is held in memory, which suits small files. For large uploads, use
`UploadFile`, which is spooled and streams to disk with `save_to`. A field can also be
optional or repeated: use `Option<FileBytes>` for an optional file and `Vec<FileBytes>`
for several under the same name.

Limits can also be set per route on the upload attributes, such as `max_file_size` and
`max_files`, so a single endpoint can be stricter than the application default.
