# Introduction

Tork is an async web framework for Rust. You use it to build HTTP APIs: a function
becomes a route, the arguments of that function become the data you pull out of the
request, and the return value becomes the response.

The framework is built on Hyper and Tokio, so it runs on the same async stack that
most production Rust services already use. The pieces you reach for every day
(routing, request parsing, validation, dependency injection, OpenAPI) are part of
the framework, not extra crates you have to wire together.

Here is a complete application. It answers `GET /` with a short message.

```rust
use tork::{api_router, get, App};

#[api_router]
pub mod hello {
    use super::*;

    #[get("/")]
    pub async fn index() -> tork::Result<&'static str> {
        Ok("Hello from Tork")
    }
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .include_router(hello::router())
        .serve("0.0.0.0:8000")
        .await
}
```

## What you get

- **Routing from attributes.** A route is a function with a `#[get]` or `#[post]`
  attribute. The path and method live next to the code that handles them.
- **Typed request data.** A handler asks for what it needs in its arguments. A path
  like `/users/{id}` gives you an `id: i64`, and a JSON body becomes a typed struct.
  Bad input is rejected before your function runs.
- **Validation built in.** Mark the fields of a model with rules like `min_length`
  or `gt`, and an invalid body returns `422` with the field errors filled in.
- **Dependency injection.** Services and shared resources are passed into handlers
  by type. You do not reach into a global object to find them.
- **OpenAPI for free.** Your routes and models produce an OpenAPI document and a
  documentation page, so the API is browsable from the first endpoint.

## 1. Introduction

Tork helps you build HTTP APIs in Rust without giving up control or performance.
It is opinionated about developer experience and unopinionated about your domain.

### What Tork gives you

- **Routers from annotations.** Group handlers in a module, annotate them with
  `#[get]`, `#[post]`, and friends, and Tork builds the router for you.
- **Dependency injection.** Handlers declare what they need (a service, the
  current user, the parsed body) as parameters. Tork resolves each one.
- **Validation and models.** `#[api_model]` turns a struct into a validated,
  documented request or response type.
- **Errors as data.** Every error renders to a consistent JSON body with a status,
  a machine code, a message, optional field details, a trace id, and a timestamp.
- **OpenAPI out of the box.** Routes and models produce an OpenAPI document and a
  documentation page with no extra work.
- **Middleware.** Wrap request handling with built-in or custom layers.

### How it fits together

A Tork application is assembled with the `App` builder and run with `serve`:

```rust
use tork::App;

use my_api::core::app_state::AppState;
use my_api::routers::users;

##[tork::main]
async fn main() -> tork::Result<()> {
    let state = AppState::boot().await?;

    App::new()
        .state(state)
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

`#[tork::main]` sets up the asynchronous runtime, `state` registers shared
application state, `include_router` mounts a group of routes, and `serve` binds
an address and runs until the process receives a shutdown signal.

The chapters that follow introduce each piece in turn. The next chapter gets a

    fn from_request(
        ctx: &RequestContext,
    ) -> impl std::future::Future<Output = tork::Result<Self>> + Send {
        let headers = ctx.headers().clone();
        async move { Ok(Headers(headers)) }
    }
}

#[get("/headers")]
async fn read_headers(headers: Headers) -> tork::Result<serde_json::Value> {
    let token = headers.0.get("x-token").and_then(|v| v.to_str().ok());

    Ok(json!({ "token": token, "cookie": cookie }))
}
