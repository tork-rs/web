# Error Handling

A handler returns `tork::Result<T>`. On success you return `Ok(value)`, and on
failure you return an error that Tork turns into an HTTP response with the right
status code and a JSON body. Because the return type is a `Result`, you can use the
`?` operator throughout your code.

## Returning an error

The `tork::Error` type has a constructor for each common status. Pick the one that
fits the situation and pass a message.

```rust
use tork::{api_router, get, Error};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64) -> tork::Result<i64> {
        if user_id < 1 {
            return Err(Error::bad_request("user id must be positive"));
        }
        if user_id > 100 {
            return Err(Error::not_found("user not found"));
        }
        Ok(user_id)
    }
}
```

A `not_found` error answers with `404` and a JSON body describing the problem. The
constructors map to the statuses you reach for most:

| Constructor | Status |
| --- | --- |
| `Error::bad_request` | 400 |
| `Error::unauthorized` | 401 |
| `Error::forbidden` | 403 |
| `Error::not_found` | 404 |
| `Error::conflict` | 409 |
| `Error::unprocessable` | 422 |
| `Error::too_many_requests` | 429 |
| `Error::internal` | 500 |
| `Error::service_unavailable` | 503 |

## Using the question mark operator

Because handlers return a `Result`, you can let errors flow up with `?`. Any
function that returns a `tork::Result` composes cleanly.

```rust
use tork::{api_router, get, Error};

fn parse_page(raw: &str) -> tork::Result<u32> {
    raw.parse::<u32>()
        .map_err(|_| Error::bad_request("page must be a number"))
}

#[api_router(tags = ["search"])]
pub mod search {
    use super::*;

    #[get("/page/{raw}")]
    pub async fn page(raw: String) -> tork::Result<u32> {
        let page = parse_page(&raw)?;
        Ok(page)
    }
}
```

## Your own error types

Most applications have their own error type in the data or service layer. Derive
`AppError` on it and set a default status with `#[status(...)]`. This generates a
conversion into `tork::Error`, so a function can surface your error through `?`.

```rust
#[derive(Debug, tork::AppError)]
#[status(503)]
pub enum RepoError {
    Unavailable,
}

impl std::fmt::Display for RepoError {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        match self {
            RepoError::Unavailable => f.write_str("the data store is unavailable"),
        }
    }
}

impl std::error::Error for RepoError {}
```

Now a handler can return your error through `?`, and it becomes a `503` response.

```rust
use tork::{api_router, get};

#[api_router(prefix = "/demo", tags = ["demo"])]
pub mod demo {
    use super::*;

    #[get("/db")]
    pub async fn db() -> tork::Result<i64> {
        let outcome: Result<i64, RepoError> = Err(RepoError::Unavailable);
        Ok(outcome?)
    }
}
```

## Mapping an error to a custom response

Sometimes you want full control over how a specific error type becomes a response,
for example to hide internal detail or to log it. Register an exception handler for
the type on the application. It receives the error and a context, and returns a
response.

```rust
use tork::{App, ErrorContext, IntoResponse, Logger, Response};

async fn handle_repo_error(error: RepoError, ctx: ErrorContext) -> Response {
    Logger::new("ExceptionHandler")
        .error("repository error")
        .field("path", ctx.path().to_owned())
        .error(&error)
        .emit();
    tork::Error::service_unavailable("the service is temporarily unavailable")
        .into_response()
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .include_router(demo::router())
        .exception_handler::<RepoError, _, _>(handle_repo_error)
        .serve("0.0.0.0:8000")
        .await
}
```

Now any `RepoError` that reaches the boundary is logged and answered with a clean
message, while the internal detail stays in your logs.
