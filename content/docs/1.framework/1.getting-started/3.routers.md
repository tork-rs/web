# Routers

A router is a module that holds a set of related routes. You mark the module with
`#[api_router]`, write one function per route inside it, and the macro builds a
`router()` function you pass to the application.

## A router with a prefix

Give a router a `prefix` so every route inside it starts from the same base path.
The `tags` are used to group the routes in the OpenAPI documentation.

```rust
use tork::{api_router, get};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/")]
    pub async fn list() -> tork::Result<Vec<i64>> {
        Ok(vec![1, 2, 3])
    }
}
```

With the prefix `/users` and the route path `/`, this handler answers `GET /users`.
The prefix is joined with each route's local path, and a trailing slash is dropped
(except for the root path).

## Path parameters

Write a parameter in the path with braces, then add an argument with the same name
to the handler. Tork parses the value into the type you declare using its `FromStr`
implementation, so `{user_id}` becomes a real `i64`.

```rust
#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64) -> tork::Result<i64> {
        Ok(user_id)
    }
}
```

A request to `GET /users/42` gives you `user_id = 42`. If the value does not parse
as an `i64`, the request is rejected with `422` before your function runs. Every
other handler parameter is resolved through dependency injection, which the next
chapter covers.

## HTTP methods

There is an attribute for each method: `get`, `post`, `put`, `patch`, and `delete`.
They take the same options.

```rust
use tork::{api_router, get, post, delete};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64) -> tork::Result<i64> {
        Ok(user_id)
    }

    #[delete("/{user_id}")]
    pub async fn remove(user_id: i64) -> tork::Result<()> {
        // Returning the unit type sends an empty 200 response.
        Ok(())
    }
}
```

Routes are matched by method and path. A request to a known path with an
unsupported method returns `405 Method Not Allowed`, while an unknown path returns
`404 Not Found`.

## Documenting a route

A few options make a route clearer in the generated documentation:

- `response_model` declares the type of the success response. When it differs from
  the handler's return type, the returned value is converted into it.
- `summary` is a short one line description, and `description` is a longer note.
- `status_code` sets the success status when a route should answer with something
  other than `200`, such as `201` for a created resource.
- `tags` adds extra tags to a single route, on top of the router tags.

::code-group

```rust [src/schemas/user.rs]
use tork::api_model;

/// A user as returned by the API.
#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}

/// The body accepted when creating a user.
#[api_model]
pub struct CreateUser {
    pub name: String,
}
```

```rust [src/routers/users.rs]
use tork::{api_router, get, post, Valid};

use crate::schemas::user::{CreateUser, UserOut};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get(
        "/{user_id}",
        response_model = UserOut,
        summary = "Get a user by id",
        description = "Looks up a single user by their numeric id."
    )]
    pub async fn get_one(user_id: i64) -> tork::Result<UserOut> {
        Ok(UserOut { id: user_id, name: "Ada".to_owned() })
    }

    #[post("/", response_model = UserOut, status_code = 201, summary = "Create a user")]
    pub async fn create(body: Valid<CreateUser>) -> tork::Result<UserOut> {
        let input = body.into_inner();
        Ok(UserOut { id: 1, name: input.name })
    }
}

// Re-export the generated router so callers write `users::router()`.
pub use users_router::router;
```

::

The `Valid<CreateUser>` argument reads and validates the JSON body. Request bodies
and validation are covered in [Request Data](/docs/framework/getting-started/request-data).

## Nesting routers

Larger APIs are easier to read when each area has its own router. A parent router
can include a child with `include`, and the child keeps its own prefix. Prefixes
compose and tags are merged.

Each file under `src/routers/` is one router and re-exports its `router()` function.
The `src/routers/mod.rs` file declares those submodules so the rest of the crate can
reach them as `crate::routers::<name>`.

::code-group

```rust [src/routers/mod.rs]
pub mod orders;
pub mod users;
```

```rust [src/routers/orders.rs]
use tork::{api_router, get};

#[api_router(prefix = "/{user_id}/orders", tags = ["orders"])]
pub mod orders_router {
    use super::*;

    #[get("/")]
    pub async fn list(user_id: i64) -> tork::Result<Vec<i64>> {
        Ok(vec![user_id])
    }
}

// Re-export the generated router so callers write `orders::router()`.
pub use orders_router::router;
```

```rust [src/routers/users.rs]
use tork::{api_router, get};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("/{user_id}")]
    pub async fn get_one(user_id: i64) -> tork::Result<i64> {
        Ok(user_id)
    }
}

pub fn router() -> tork::Router {
    users_router::router().include(crate::routers::orders::router())
}
```

::

Mounting that combined router exposes both `GET /users/{user_id}` and
`GET /users/{user_id}/orders`. A path parameter declared in a parent prefix, like
`{user_id}`, is available to the nested handlers as an ordinary parameter.

## Mounting on the application

Each router is mounted with `include_router`. You can call it more than once to
mount several routers.

```rust
use tork::App;

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```
