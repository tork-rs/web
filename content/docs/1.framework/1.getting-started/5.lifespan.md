# Lifespan

Some things in an application are built once and shared by every request: a database
pool, an HTTP client, an in-memory store, a broadcast channel. The lifespan is where
you create them at startup and release them at shutdown. Once a resource is
registered, any handler or service can ask for it by type, exactly like the
dependencies in the [previous guide](/docs/framework/getting-started/dependencies).

## A resource

A resource is any value you want to share. It has to be `Clone`, because Tork hands a
clone to each request, so wrap anything expensive in an `Arc` to keep cloning cheap.

```rust
use std::collections::HashMap;
use std::sync::Arc;

use tork::{api_model, Error};

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}

/// An in-memory store. A real app would hold a database pool here.
#[derive(Clone)]
pub struct UserStore {
    users: Arc<HashMap<i64, UserOut>>,
}

impl UserStore {
    pub fn seed() -> Self {
        let users = HashMap::from([
            (1, UserOut { id: 1, name: "Ada".to_owned() }),
            (2, UserOut { id: 2, name: "Alan".to_owned() }),
        ]);
        Self { users: Arc::new(users) }
    }

    pub fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.users
            .get(&id)
            .cloned()
            .ok_or_else(|| Error::not_found("user not found"))
    }
}
```

## The resource container

Group your resources in a struct that derives `Resources`, and mark each shared field
with `#[resource]`. This struct is what the lifespan builds.

```rust
use tork::Resources;

#[derive(Clone, Resources)]
pub struct AppState {
    #[resource]
    pub store: UserStore,
}
```

## Startup and shutdown

Add a `#[tork::lifespan]` block to the container. The `startup` function builds it,
and the value it returns has its resources registered for injection. The `shutdown`
function releases them and is optional, so you can leave it out when there is nothing
to clean up.

```rust
use tork::{LifespanContext, Logger, Result};

#[tork::lifespan]
impl AppState {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        Ok(AppState { store: UserStore::seed() })
    }

    async fn shutdown(self) -> Result<()> {
        Logger::new("Lifecycle").info("application stopped").emit();
        Ok(())
    }
}
```

`startup` returns a `Result`, so it runs first and fails fast. If it returns an
error, for example because a database is unreachable, the application does not start
and serves no traffic. This is the right place to open connections and verify they
work.

The `LifespanContext` can read required environment variables, which is handy for a
value you need before the app is up.

```rust
async fn startup(ctx: LifespanContext) -> Result<Self> {
    let database_url = ctx.env("DATABASE_URL")?;
    let store = UserStore::connect(&database_url).await?;
    Ok(AppState { store })
}
```

## Registering the lifespan

Tell the application about the lifespan with `lifespan`. Startup runs before the
server binds, and shutdown runs when the server stops.

```rust
use tork::{api_router, get, App};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users {
    use super::*;

    #[get("/{user_id}", response_model = UserOut)]
    pub async fn get_one(user_id: i64, store: UserStore) -> tork::Result<UserOut> {
        store.get(user_id)
    }
}

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .lifespan::<AppState>()
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

The handler asks for `store: UserStore`, and it is filled in on every request. The
handler does not know or care how the store was created.

## Using resources in services

Resources are not only for handlers. A service that derives `Inject` can take a
resource as a field, so your logic stays out of the handler.

```rust
use tork::Inject;

#[derive(Inject)]
pub struct UserService {
    store: UserStore,
}

impl UserService {
    pub async fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.store.get(id)
    }
}
```

```rust
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_one(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get(user_id).await
}
```

## More than one resource

A container can hold as many resources as you need. Each `#[resource]` field becomes
injectable by its own type.

```rust
use tork::{Hub, Resources};

#[derive(Clone)]
pub struct Events(pub Hub<String>);

#[derive(Clone, Resources)]
pub struct AppState {
    #[resource]
    pub store: UserStore,
    #[resource]
    pub events: Events,
}

#[tork::lifespan]
impl AppState {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        Ok(AppState {
            store: UserStore::seed(),
            events: Events(Hub::new()),
        })
    }
}
```

A handler can now ask for either resource, or both.

```rust
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_one(
    user_id: i64,
    store: UserStore,
    events: Events,
) -> tork::Result<UserOut> {
    let user = store.get(user_id)?;
    events.0.room("audit").broadcast(format!("read user {user_id}"));
    Ok(user)
}
```

## Registering a single value

Not every shared value needs a container. When you already have one value, such as a
loaded configuration, register it directly with `state`. It is then injectable by its
type. Wrapping it in an `Arc` keeps each injection cheap.

```rust
use std::sync::Arc;

struct Config {
    app_name: String,
}

let config = Arc::new(Config { app_name: "my_api".to_owned() });

App::new()
    .state(config)
    .include_router(users::router());
```

```rust
#[get("/info")]
pub async fn info(config: Arc<Config>) -> tork::Result<String> {
    Ok(config.app_name.clone())
}
```

This is how loaded settings reach your handlers. The lifespan and `state` work
together: build a container for the resources you set up at startup, and register
single values like configuration directly.

### Real repo example 1: lifespan.rs
```rust
//! End-to-end test of `#[tork::lifespan]`, resources, and the serve lifecycle.

use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::{Arc, Mutex};

use tokio::io::{AsyncReadExt, AsyncWriteExt};
use tokio::net::TcpStream;
use tokio::sync::oneshot;
use tork::{get, App, LifespanContext, Resources, Router};

static STOPPED: AtomicBool = AtomicBool::new(false);

/// A resource produced by the lifespan.
#[derive(Clone)]
struct Marker(i64);

#[derive(Clone, Resources)]
struct BootState {
    #[resource]
    marker: Marker,
}

#[tork::lifespan]
impl BootState {
    async fn startup(_ctx: LifespanContext) -> tork::Result<Self> {
        Ok(BootState {
            marker: Marker(123),
        })
    }

    async fn shutdown(self) -> tork::Result<()> {
        STOPPED.store(true, Ordering::SeqCst);
        Ok(())
    }
}

#[get("/marker")]
async fn read_marker(marker: Marker) -> tork::Result<i64> {
    Ok(marker.0)
}

#[tokio::test]
async fn lifespan_registers_resources_and_shuts_down() {
    let (addr_tx, addr_rx) = oneshot::channel();
    let (shutdown_tx, shutdown_rx) = oneshot::channel();
    let sender = Arc::new(Mutex::new(Some(addr_tx)));

    let app = App::new()
        .lifespan::<BootState>()
        .on_ready(move |ctx| {
            let sender = sender.clone();
            async move {
                if let Some(tx) = sender.lock().unwrap().take() {
                    let _ = tx.send(ctx.addr());
                }
                Ok(())
            }
        })
        .include_router(Router::new().route(__tork_route_read_marker()));

    let server = tokio::spawn(app.serve_with_shutdown("127.0.0.1:0", async move {
        let _ = shutdown_rx.await;
    }));

    let addr = addr_rx.await.unwrap();
    let mut stream = TcpStream::connect(addr).await.unwrap();
    stream
        .write_all(b"GET /marker HTTP/1.1\r\nHost: localhost\r\nConnection: close\r\n\r\n")
        .await
        .unwrap();
    let mut response = String::new();
    stream.read_to_string(&mut response).await.unwrap();

    // The lifespan registered the Marker resource; the handler injected it.
    assert!(response.contains("123"), "unexpected response: {response}");

    let _ = shutdown_tx.send(());
    let _ = server.await;

    assert!(STOPPED.load(Ordering::SeqCst), "shutdown should have run");
}

// --- Multiple lifespans: startup in order, shutdown in reverse ---

static ORDER: Mutex<Vec<&'static str>> = Mutex::new(Vec::new());

#[derive(Clone, Resources)]
struct First;

#[derive(Clone, Resources)]
struct Second;

#[tork::lifespan]
impl First {
    async fn startup(_ctx: LifespanContext) -> tork::Result<Self> {
        ORDER.lock().unwrap().push("first");
        Ok(First)
    }
    async fn shutdown(self) -> tork::Result<()> {
        ORDER.lock().unwrap().push("first-stop");
        Ok(())
    }
}

#[tork::lifespan]
impl Second {
    async fn startup(_ctx: LifespanContext) -> tork::Result<Self> {
        ORDER.lock().unwrap().push("second");
        Ok(Second)
    }
    async fn shutdown(self) -> tork::Result<()> {
        ORDER.lock().unwrap().push("second-stop");
        Ok(())
    }
}

#[tokio::test]
async fn multiple_lifespans_start_in_order_and_stop_in_reverse() {
    ORDER.lock().unwrap().clear();

    App::new()
        .lifespan::<First>()
        .lifespan::<Second>()
        .serve_with_shutdown("127.0.0.1:0", async {})
        .await
        .unwrap();

    assert_eq!(
        *ORDER.lock().unwrap(),
        ["first", "second", "second-stop", "first-stop"]
    );
}

#[test]
fn lifespan_with_event_hooks_is_rejected() {
    let error = App::new()
        .lifespan::<First>()
        .on_shutdown(|| async { Ok(()) })
        .build()
        .err()
        .expect("lifespan plus on_shutdown should conflict");
    assert_eq!(error.code(), "LIFECYCLE_CONFLICT");
}
```
