# Lifespan

Some things in an application are built once and shared by every request: a database
pool, an HTTP client, an in-memory store, a broadcast channel. The lifespan is where
you create them at startup and release them at shutdown. Once a resource is
registered, any handler or service can ask for it by type, exactly like the
dependencies in the [previous guide](/docs/framework/getting-started/dependencies).

## A resource

A resource is any value you want to share. It has to be `Clone`, because Tork hands a
clone to each request, so wrap anything expensive in an `Arc` to keep cloning cheap.

The store and the schema it returns live in their own files: the schema is part of the
API contract, the store is infrastructure.

::code-group

```rust [src/schemas/user.rs]
use tork::api_model;

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}
```

```rust [src/core/store.rs]
use std::collections::HashMap;
use std::sync::Arc;

use tork::Error;

use crate::schemas::user::UserOut;

/// An in-memory store. A real app would hold a database pool here.
#[derive(Clone)]
pub struct UserStore {
    users: Arc<HashMap<i64, UserOut>>,
}

impl UserStore {
    pub fn seed() -> Self {
        let users = HashMap::from([
            (1, UserOut { id: 1, name: "Ada".to_owned() }),
            (2, UserOut { id: 2, name: "Alan".to_owned() }),
        ]);
        Self { users: Arc::new(users) }
    }

    pub fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.users
            .get(&id)
            .cloned()
            .ok_or_else(|| Error::not_found("user not found"))
    }
}
```

::

## The resource container

Group your resources in a struct that derives `Resources`, and mark each shared field
with `#[resource]`. The derive registers each marked field by type and makes it
injectable. This struct, in `src/core/state.rs`, is what the lifespan builds.

```rust [src/core/state.rs]
use tork::Resources;

use crate::core::store::UserStore;

#[derive(Clone, Resources)]
pub struct AppState {
    #[resource]
    pub store: UserStore,
}
```

## Startup and shutdown

Add a `#[tork::lifespan]` block to the same type. The `startup` function builds the
container, and the value it returns has its resources registered for injection. The
`shutdown` function releases them and is optional, so you can leave it out when there
is nothing to clean up. The block may only hold `startup` and `shutdown`; put any
helper methods in a separate `impl` block.

```rust [src/core/state.rs]
use tork::{LifespanContext, Logger, Result};

#[tork::lifespan]
impl AppState {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        Ok(AppState { store: UserStore::seed() })
    }

    async fn shutdown(self) -> Result<()> {
        Logger::new("Lifecycle").info("application stopped").emit();
        Ok(())
    }
}
```

`startup` returns a `Result`, so it runs first and fails fast. If it returns an
error, for example because a database is unreachable, the application does not start
and serves no traffic. This is the right place to open connections and verify they
work.

When a resource needs a value from the environment, read it from the context with
`ctx.env`. It returns an error (so startup fails fast) when the variable is not set.

```rust
async fn startup(ctx: LifespanContext) -> Result<Self> {
    let database_url = ctx.env("DATABASE_URL")?;
    let store = UserStore::connect(&database_url).await?;
    Ok(AppState { store })
}
```

## Registering the lifespan

Tell the application about the lifespan with `lifespan`. Startup runs before the
server binds, and shutdown runs when the server stops. The handler asks for
`store: UserStore` and Tork fills it in on every request. Each directory has a
`mod.rs` that declares its files, as shown in the [Routers](/docs/framework/getting-started/routers)
guide.

::code-group

```rust [src/routers/users.rs]
use tork::{api_router, get};

use crate::core::store::UserStore;
use crate::schemas::user::UserOut;

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("/{user_id}", response_model = UserOut)]
    pub async fn get_one(user_id: i64, store: UserStore) -> tork::Result<UserOut> {
        store.get(user_id)
    }
}

pub use users_router::router;
```

```rust [src/main.rs]
mod core;
mod routers;
mod schemas;

use tork::App;

use crate::core::state::AppState;
use crate::routers::users;

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .lifespan::<AppState>()
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

::

The handler does not know or care how the store was created. It asks for the type,
and the lifespan made sure that type is available.

## Using resources in services

Resources are not only for handlers. A service that derives `Inject` can take a
resource as a field, so your logic stays out of the handler.

::code-group

```rust [src/services/user_service.rs]
use tork::Inject;

use crate::core::store::UserStore;
use crate::schemas::user::UserOut;

#[derive(Inject)]
pub struct UserService {
    store: UserStore,
}

impl UserService {
    pub async fn get(&self, id: i64) -> tork::Result<UserOut> {
        self.store.get(id)
    }
}
```

```rust [src/routers/users.rs]
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_one(user_id: i64, service: UserService) -> tork::Result<UserOut> {
    service.get(user_id).await
}
```

## More than one resource

A container can hold as many resources as you need. Each `#[resource]` field becomes
injectable by its own type.

::code-group

```rust [src/core/state.rs]
use tork::{Hub, LifespanContext, Resources, Result};

use crate::core::store::UserStore;

#[derive(Clone)]
pub struct Events(pub Hub<String>);

#[derive(Clone, Resources)]
pub struct AppState {
    #[resource]
    pub store: UserStore,
    #[resource]
    pub events: Events,
}

#[tork::lifespan]
impl AppState {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        Ok(AppState {
            store: UserStore::seed(),
            events: Events(Hub::new()),
        })
    }
}
```

```rust [src/routers/users.rs]
#[get("/{user_id}", response_model = UserOut)]
pub async fn get_one(
    user_id: i64,
    store: UserStore,
    events: Events,
) -> tork::Result<UserOut> {
    let user = store.get(user_id)?;
    events.0.room("audit").broadcast(format!("read user {user_id}"));
    Ok(user)
}
```

::

## More than one lifespan

You can register several lifespans by calling `lifespan` more than once. They start in
registration order and stop in reverse order, so a lifespan can safely depend on one
registered before it.

```rust [src/main.rs]
App::new()
    .lifespan::<Database>()
    .lifespan::<AppState>()
    .include_router(users::router())
    .serve("0.0.0.0:8000")
    .await
```

## Startup without a lifespan

For a small task that does not produce a resource, register a plain hook instead.
`on_startup` runs before the server binds and `on_shutdown` runs when it stops.

```rust [src/main.rs]
App::new()
    .on_startup(|| async {
        // Warm a cache, run a one-off check, and so on.
        Ok(())
    })
    .on_shutdown(|| async {
        Logger::new("Lifecycle").info("goodbye").emit();
        Ok(())
    })
    .include_router(users::router());
```

Use one approach or the other: combining `lifespan` with `on_startup` or `on_shutdown`
is a configuration error and fails the build with the code `LIFECYCLE_CONFLICT`.

If you only need to know when the server is up, for example to print the bound
address, use `on_ready`.

```rust [src/main.rs]
App::new()
    .on_ready(|ctx| async move {
        Logger::new("Lifecycle")
            .info("listening")
            .field("addr", ctx.addr().to_string())
            .emit();
        Ok(())
    })
    .include_router(users::router());
```

## Registering a single value

Not every shared value needs a container. When you already have one value, such as a
loaded configuration, register it directly with `state`. Wrap it in an `Arc` and a
handler can take it by `Arc<T>`, cloning only the pointer per request.

::code-group

```rust [src/main.rs]
use std::sync::Arc;

use tork::App;

pub struct Config {
    pub app_name: String,
}

#[tork::main]
async fn main() -> tork::Result<()> {
    let config = Arc::new(Config { app_name: "my_api".to_owned() });

    App::new()
        .state(config)
        .include_router(users::router())
        .serve("0.0.0.0:8000")
        .await
}
```

```rust [src/routers/info.rs]
use std::sync::Arc;

use tork::get;

use crate::Config;

#[get("/info")]
pub async fn info(config: Arc<Config>) -> tork::Result<String> {
    Ok(config.app_name.clone())
}
```

::

This is how loaded settings reach your handlers. The lifespan and `state` work
together: build a container for the resources you set up at startup, and register
single values like configuration directly.
