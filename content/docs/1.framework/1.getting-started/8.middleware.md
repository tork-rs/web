# Middleware

A middleware is a function that wraps request handling. It gets the request
before your handler runs, can modify it or reject it, passes it along, and then
gets the response back from your handler before it goes to the client.

You stack middlewares on an app with `.middleware()`. They run in the order you
add them, outermost first.

```rust
use tork::{middleware::{RequestId, Cors, Compression}, App};

App::new()
    .middleware(RequestId::new())
    .middleware(Cors::new().allow_origin("*"))
    .middleware(Compression::new().gzip())
    .include_router(my_router::router())
    .serve("0.0.0.0:8000")
    .await
```

## Custom middleware

You can write your own middleware with the `#[middleware]` attribute. The
function takes a `Request` and a `Next` handle. Call `next.run(req)` to pass
the request to the rest of the chain. You can inspect or change the request
before calling `next`, and the response after.

```rust
use tork::{middleware, App, Next, Request, Response, Result};

#[middleware]
async fn log_method(req: Request, next: Next) -> Result<Response> {
    println!("got a {}", req.method());
    let response = next.run(req).await?;
    Ok(response)
}
```

Register it the same way.

```rust
App::new()
    .middleware(log_method)
    .include_router(my_router::router())
    .serve("0.0.0.0:8000")
    .await
```

### Reading request data

You can read headers or the URI before the handler runs.

```rust
#[middleware]
async fn check_user_agent(req: Request, next: Next) -> Result<Response> {
    let ua = req
        .headers()
        .get("user-agent")
        .and_then(|v| v.to_str().ok())
        .unwrap_or("unknown");
    println!("user agent: {ua}");
    next.run(req).await
}
```

### Modifying the request

You can add headers before the handler sees the request.

```rust
#[middleware]
async fn add_timestamp(mut req: Request, next: Next) -> Result<Response> {
    let now = std::time::SystemTime::now()
        .duration_since(std::time::UNIX_EPOCH)
        .unwrap()
        .as_secs();
    if let Ok(value) = http::HeaderValue::from_str(&now.to_string()) {
        req.headers_mut().insert(
            http::header::HeaderName::from_static("x-timestamp"),
            value,
        );
    }
    next.run(req).await
}
```

### Modifying the response

You can add response headers after the handler runs.

```rust
#[middleware]
async fn add_server_header(req: Request, next: Next) -> Result<Response> {
    let mut res = next.run(req).await?;
    res.headers_mut().insert(
        http::header::HeaderName::from_static("x-powered-by"),
        http::HeaderValue::from_static("tork"),
    );
    Ok(res)
}
```

### Short-circuiting

You can return a response without calling `next`. The rest of the chain and
the handler never run.

```rust
#[middleware]
async fn block_maintenance(req: Request, next: Next) -> Result<Response> {
    if req.uri().path() == "/maintenance" {
        return Ok(tork::response::empty(http::StatusCode::SERVICE_UNAVAILABLE));
    }
    next.run(req).await
}
```

### Measuring time

A common pattern is to time how long the handler took.

```rust
#[middleware]
async fn add_process_time(req: Request, next: Next) -> Result<Response> {
    let start = std::time::Instant::now();
    let mut res = next.run(req).await?;
    let elapsed = start.elapsed().as_secs_f64();
    if let Ok(value) = http::HeaderValue::from_str(&format!("{elapsed:.6}")) {
        res.headers_mut().insert(
            http::header::HeaderName::from_static("x-process-time"),
            value,
        );
    }
    Ok(res)
}
```

## Built-in middleware types

Tork includes middleware for the most common needs. Every built-in middleware
can only be registered once. If you try to add a second copy, the app will
refuse to start with a clear error.

### RequestId

Assigns a unique id to every request and echoes it back on the response. If
the incoming request already carries the id header, the existing value is
reused.

```rust
use tork::middleware::RequestId;

App::new()
    .middleware(RequestId::new())
    .include_router(my_router::router())
    .serve("0.0.0.0:8000")
    .await
```

The default header is `x-request-id`. You can change it.

```rust
App::new()
    .middleware(RequestId::new().header_name("x-correlation-id"))
```

### Trace

Logs every request's method, path, status, and how long it took. Unlike the
built-in server logs that only cover matched routes, this middleware logs
before routing, so short-circuited and unmatched requests are also recorded.

```rust
use tork::middleware::Trace;

App::new()
    .middleware(Trace::new())
    .include_router(my_router::router())
```

### Timeout

Fails a request with `504 Gateway Timeout` if it takes longer than the
allowed time.

```rust
use std::time::Duration;
use tork::middleware::Timeout;

App::new()
    .middleware(Timeout::new(Duration::from_secs(10)))
```

You can also use seconds or milliseconds directly.

```rust
// five second timeout
.middleware(Timeout::seconds(5))

// 500 millisecond timeout for fast endpoints
.middleware(Timeout::millis(500))
```

### BodyLimit

Rejects requests whose body exceeds a size limit. The `Content-Length` header
is checked before the handler runs. Chunked bodies are also checked as they
are read.

```rust
use tork::middleware::BodyLimit;

App::new()
    .middleware(BodyLimit::mb(5))   // 5 mebibytes
    .middleware(BodyLimit::kb(500)) // 500 kibibytes
    .middleware(BodyLimit::bytes(1024)) // 1024 bytes
```

A request that goes over the limit gets `413 Payload Too Large`.

### Cors

Handles Cross-Origin Resource Sharing. It answers preflight `OPTIONS`
requests directly and annotates actual responses with the right headers.

```rust
use tork::middleware::Cors;

App::new()
    .middleware(
        Cors::new()
            .allow_origin("*")
            .allow_methods(["GET", "POST"])
            .allow_headers(["Authorization"]),
    )
```

You can control each part of CORS.

```rust
Cors::new()
    .allow_origin("https://app.example.com")
    .allow_origin("https://admin.example.com")
    .allow_methods(["GET", "POST", "PUT", "DELETE"])
    .allow_headers(["Authorization", "Content-Type"])
    .expose_headers(["X-Request-Id"])
    .allow_credentials(true)
    .max_age(3600)
```

Some notes on CORS:
- Wildcard `*` origin cannot be used with credentials. The middleware rejects
  that configuration at runtime by not returning the wildcard.
- Preflight requests return `204 No Content`.
- The `Vary` header is set correctly for both preflight and actual requests.

### SecurityHeaders

Adds baseline security headers to every response. The defaults are sensible
for most APIs.

```rust
use tork::middleware::SecurityHeaders;

App::new()
    .middleware(SecurityHeaders::new())
```

Default headers set on every response:

| Header | Value |
| --- | --- |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` |
| `X-Content-Type-Options` | `nosniff` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `no-referrer` |

`Content-Security-Policy` is off by default. You can turn it on.

```rust
SecurityHeaders::new()
    .content_security_policy("default-src 'self'")
```

You can customize individual headers or remove them.

```rust
SecurityHeaders::new()
    .frame_options("SAMEORIGIN")
    .hsts_max_age(60, false)
    .without_content_type_options()
    .without_referrer_policy()
```

If a handler sets one of these headers, the handler's value stays. The
middleware only adds headers that are not already present.

### Compression

Compresses response bodies with gzip when the client supports it.

```rust
use tork::middleware::Compression;

App::new()
    .middleware(Compression::new().gzip())
```

You can set a minimum size to avoid compressing tiny responses.

```rust
Compression::new()
    .gzip()
    .minimum_size(256) // only compress bodies over 256 bytes
```

There is also a maximum size cap (default 8 MiB) to keep memory bounded. You
can raise or remove it.

```rust
Compression::new()
    .gzip()
    .maximum_size(usize::MAX) // allow any size
```

Compression middleware only enables gzip at the moment. The method is named
`gzip()` so that other algorithms can be added later without breaking
existing code.

The middleware also sets `Vary: Accept-Encoding` on every response so that
caches do not serve a compressed body to a client that does not support it.

### TrustedHost

Rejects requests whose `Host` header is not in an allow-list. Patterns can
be exact or wildcard suffixes with `*.`.

```rust
use tork::middleware::TrustedHost;

App::new()
    .middleware(TrustedHost::new(["example.com", "*.example.com"]))
```

A request with a host that does not match any pattern gets `400 Bad Request`.

The middleware strips port numbers before matching, so `example.com:8080`
matches `example.com` in the list.

### HttpsRedirect

Redirects plain HTTP requests to HTTPS with a `308 Permanent Redirect`.

```rust
use tork::middleware::HttpsRedirect;

App::new()
    .middleware(HttpsRedirect::new())
```

This middleware reads the scheme from trusted proxy headers if
`ProxyHeaders` is registered before it. Otherwise it uses the URI scheme.

The redirect preserves the path and query string.

### ProxyHeaders

Normalizes headers set by a reverse proxy. When the request comes through a
trusted proxy, this middleware rewrites the `Host` header from
`X-Forwarded-Host` and stores the original scheme from `X-Forwarded-Proto`.

```rust
use std::net::IpAddr;
use tork::middleware::ProxyHeaders;

App::new()
    .middleware(
        ProxyHeaders::new()
            .trust_proxy(IpAddr::from([10, 0, 0, 1]))
            .trust_loopback(),
    )
```

You can trust individual IPs, whole CIDR networks, or both.

```rust
ProxyHeaders::new()
    .trust_proxy(IpAddr::from([10, 0, 0, 1]))
    .trust_cidr("10.0.0.0/24".parse().unwrap())
```

When used together with `HttpsRedirect` or `TrustedHost`, put `ProxyHeaders`
first so they see the normalized values.

```rust
App::new()
    .middleware(ProxyHeaders::new().trust_loopback())
    .middleware(HttpsRedirect::new())
    .middleware(TrustedHost::new(["example.com"]))
```

## Registration order

Middlewares run in the order you add them. The first middleware you register
runs first. It gets the original request, does its work, then calls
`next.run()` which triggers the next middleware, and so on down to the
handler. The response then bubbles back up through each middleware in reverse
order.

```rust
App::new()
    .middleware(A) // runs first
    .middleware(B) // runs second
    .middleware(C) // runs third, then the handler
```

The diagram is: request goes `A -> B -> C -> handler -> C -> B -> A`, then
back to the client.

## Duplicate policy

Every built-in middleware can only be registered once. If you try to register
a second `Cors`, the app will fail to build.

```rust
// This will return an error at build time:
App::new()
    .middleware(Cors::new())
    .middleware(Cors::new())
    .build()
```

Custom middleware using `#[middleware]` uses the default policy which allows
duplicates. You can control this by implementing `Middleware` directly.
