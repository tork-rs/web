# Authorization

[Authentication](/docs/framework/security/authentication) proves who a request is from. Authorization decides what that user is allowed to do. The two map to two status codes:

- `401 Unauthorized` (`Error::unauthorized`): the request is not authenticated. The token is missing, invalid, or expired.
- `403 Forbidden` (`Error::forbidden`): the request is authenticated, but this user may not do this.

This guide adds authorization to the `CurrentUser` guard from the previous chapter. The code is the same `auth_jwt` example.

## The current user carries what you check against

The guard holds the identity plus the things you authorize on: the user's role and the scopes signed into their token.

```rust
#[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
    pub username: String,
    pub role: String,
    pub scopes: Vec<String>,
}
```

Because the scopes come from the token's claims (set when the token was issued), a token can carry a subset of a user's permissions. A handler reads them through the guard with no extra database lookup.

## Ownership: act only on your own data

The simplest rule is "you may only touch your own records". Compare the current user's id to the id in the path.

```rust
use tork::Error;

impl CurrentUser {
    pub fn ensure_can_access_user(&self, user_id: i64) -> tork::Result<()> {
        if self.id != user_id {
            return Err(Error::forbidden("you can only access your own account"));
        }
        Ok(())
    }
}
```

A handler takes both the path id and the guard, then checks before doing any work:

```rust
use tork::get;

#[get("/users/{user_id}/profile", response_model = UserOut, security = ["bearerAuth"])]
pub async fn read_profile(user_id: i64, current_user: CurrentUser) -> tork::Result<UserOut> {
    current_user.ensure_can_access_user(user_id)?;
    // ... load and return the profile ...
}
```

A request for someone else's profile is authenticated but not allowed, so it returns `403`.

## Roles

Give each user a role and gate a handler on it.

```rust
impl CurrentUser {
    pub fn require_role(&self, role: &str) -> tork::Result<()> {
        if self.role != role {
            return Err(Error::forbidden(format!("requires the `{role}` role")));
        }
        Ok(())
    }
}

#[get("/admin/settings", response_model = UserOut, security = ["bearerAuth"])]
pub async fn admin_settings(current_user: CurrentUser) -> tork::Result<UserOut> {
    current_user.require_role("admin")?;
    // ... admin-only work ...
}
```

## Scopes

Scopes are fine-grained permissions, like `users:read` or `users:write`, carried in the token. They are the OAuth2 model for "this token may do these specific things", and they are independent of the user's role. Check one with a guard method, and declare it on the route so the docs show it.

```rust
impl CurrentUser {
    pub fn require_scope(&self, scope: &str) -> tork::Result<()> {
        if !self.scopes.iter().any(|granted| granted == scope) {
            return Err(Error::forbidden(format!("missing required scope `{scope}`")));
        }
        Ok(())
    }
}

#[get(
    "/admin/overview",
    response_model = UserOut,
    security = [bearerAuth(scopes = ["users:write"])],
)]
pub async fn admin_overview(current_user: CurrentUser) -> tork::Result<UserOut> {
    current_user.require_scope("users:write")?;
    Ok(UserOut {
        id: current_user.id,
        username: current_user.username,
        role: current_user.role,
    })
}
```

Declaring `security = [bearerAuth(scopes = ["users:write"])]` records the requirement in the OpenAPI document as `"security": [{ "bearerAuth": ["users:write"] }]`, so the docs show which scopes an operation needs. The runtime enforcement is still the `require_scope` call; the declaration is documentation, not a check.

## Where the checks go

Put a rule in the guard when it applies to every protected route, such as rejecting a disabled account (the `CurrentUser` guard already does this with `403`). Put a rule in the handler when it is specific to that route, like the ownership and scope checks above. A guard rejects before the handler runs; a handler check lets one route allow more than another while sharing the same `CurrentUser`.

## Try it

The seeded `ada` has the `users:write` scope and `alan` does not, so the scoped route accepts one and forbids the other:

```bash
# ada (has users:write) -> 200
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8000/admin/overview \
  -H 'Authorization: Bearer <ada-token>'

# alan (only users:read) -> 403
curl -s -o /dev/null -w "%{http_code}\n" http://localhost:8000/admin/overview \
  -H 'Authorization: Bearer <alan-token>'
```

## Test it

```rust
#[tokio::test]
async fn scope_protected_route_checks_the_token_scopes() -> tork::Result<()> {
    let client = client().await;

    // ada has `users:write`.
    let ada = login(&client, "ada", "secret").await;
    let ok = client
        .get("/admin/overview")
        .header("authorization", &format!("Bearer {ada}"))
        .send()
        .await?;
    assert_eq!(ok.status(), 200);

    // alan only has `users:read`, so the same route is forbidden.
    let alan = login(&client, "alan", "secret2").await;
    let denied = client
        .get("/admin/overview")
        .header("authorization", &format!("Bearer {alan}"))
        .send()
        .await?;
    assert_eq!(denied.status(), 403);
    Ok(())
}
```

## Recap

| Need | Check | On failure |
| --- | --- | --- |
| Own data only | `current_user.ensure_can_access_user(user_id)?` | `403` |
| A role | `current_user.require_role("admin")?` | `403` |
| A scope | `current_user.require_scope("users:write")?` | `403` |
| Show a scope in the docs | `security = [bearerAuth(scopes = ["users:write"])]` | documentation only |

Authentication and authorization together give you a complete, standards-based security flow: a signed token proves identity, and these checks enforce permissions. See [Authentication](/docs/framework/security/authentication) for issuing and reading the token.
