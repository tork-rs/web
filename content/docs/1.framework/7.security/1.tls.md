# TLS and HTTP/2

Tork can terminate TLS itself and tune the HTTP/1 and HTTP/2 connection layers. TLS lives behind the `tls` feature, which is off by default so the base build stays lean.

```toml
tork = { version = "0.1", features = ["tls"] }
```

It uses rustls with the ring provider, so there is no OpenSSL and no C toolchain.

## Enabling TLS

Pass a `TlsConfig` to `App::tls`. Build the config from PEM file paths with `from_pem_files`, or from PEM contents with `from_pem`.

```rust
use tork::{App, TlsConfig};

App::new()
    .include(index)
    .tls(TlsConfig::from_pem_files("./cert.pem", "./key.pem")?)
    .serve("0.0.0.0:8443")
    .await?;
```

`cert.pem` is a certificate chain (leaf first), and `key.pem` is a PKCS#8, PKCS#1, or SEC1 private key. A malformed certificate or key fails fast at boot, not on the first connection, and the startup log then shows `https://`.

HTTP/2 is negotiated automatically: ALPN advertises `h2` then `http/1.1`, so a modern client gets HTTP/2 and older clients fall back to HTTP/1.1. Call `.http1_only()` on the config to advertise only HTTP/1.1, or `.alpn(protocols)` to set the list yourself.

## Trying it locally

Generate a self-signed certificate for `localhost`:

```bash
openssl req -x509 -newkey rsa:2048 -nodes -days 365 \
  -keyout key.pem -out cert.pem -subj "/CN=localhost"
```

Run the app, then call it (with `-k`, since the certificate is self-signed):

```bash
# Confirm the connection is HTTP/2 over TLS.
curl -k --http2 -I https://localhost:8443/
```

## SNI: multiple certificates

Serve a different certificate per host name. The certificate from `from_pem*` is the default, served when no name matches. Add more with `add_sni_cert_pem_files` (or `add_sni_cert_pem` for contents), keyed by the server name.

```rust
let tls = TlsConfig::from_pem_files("./default-cert.pem", "./default-key.pem")?
    .add_sni_cert_pem_files("api.example.com", "./api-cert.pem", "./api-key.pem")?
    .add_sni_cert_pem_files("admin.example.com", "./admin-cert.pem", "./admin-key.pem")?;
```

## Mutual TLS

Verify a client certificate signed by a CA you trust. The CA is given as PEM contents. Use one of the two modes, not both.

```rust
let ca = std::fs::read("./client-ca.pem")?;

// Require a valid client certificate, rejecting connections without one.
let tls = TlsConfig::from_pem_files("./cert.pem", "./key.pem")?
    .client_auth_required_pem(&ca)?;

// Or verify a client certificate when present, but still allow clients with none:
//   .client_auth_optional_pem(&ca)?;
```

## HTTP/2 and HTTP/1 tuning

Tune the protocol layers. Unset fields keep hyper's defaults. These apply with or without TLS, and HTTP/2 is served over cleartext too (detected automatically), not only over TLS.

```rust
use std::time::Duration;
use tork::{App, Http1Config, Http2Config};

App::new()
    .http2(
        Http2Config::new()
            .max_concurrent_streams(256)
            .keep_alive_interval(Duration::from_secs(20))
            .keep_alive_timeout(Duration::from_secs(10))
            .initial_stream_window_size(1 << 20)
            .initial_connection_window_size(1 << 21)
            .max_frame_size(1 << 14)
            .max_header_list_size(16 * 1024),
    )
    .http1(Http1Config::new().keep_alive(true).max_headers(100));
```

`Http2Config` has exactly those builder methods. `Http1Config` has `keep_alive(bool)` and `max_headers(usize)`. The window, frame, and header sizes on `Http2Config` are `u32`.

## Server options

```rust
use std::time::Duration;

App::new()
    .max_request_body_size(8 * 1024 * 1024) // cap buffered bodies (default 2 MiB)
    .header_read_timeout(Duration::from_secs(30)) // slowloris guard (on by default)
    .idle_timeout(Duration::from_secs(60))  // drop inactive connections (off by default)
    .reuse_port(true);                      // SO_REUSEPORT on Unix
```

- `max_request_body_size` bounds the buffered body extractors (`Json`, `Valid<T>`, and urlencoded `Form<T>`), rejecting an oversized body with `400` before it is fully buffered. The default is 2 MiB. Multipart uploads use their own [`UploadConfig`](/docs/framework/data/file-uploads) limits.
- `header_read_timeout` (default 30 seconds, on by default) caps how long a client may take to send the request head, so a slow-header connection cannot tie up a worker. It applies to HTTP/1. Disable it with `without_header_read_timeout()`, only behind a trusted proxy.
- `idle_timeout` drops a connection that has been silent too long. It is off by default, because a quiet WebSocket or SSE stream is normal. Set it well above your SSE heartbeat if you enable it.
- `reuse_port` sets `SO_REUSEPORT` on Unix, so several processes can listen on the same address and the kernel load-balances across them.

## Unix-domain sockets

On Unix, serve over a socket file instead of TCP, which is useful behind a same-host reverse proxy.

```rust
App::new().include(index).serve_unix("/tmp/my_api.sock").await?;
```

There is also `serve_unix_with_shutdown(path, future)` for a caller-driven shutdown, the Unix counterpart to `serve_with_shutdown(addr, future)`.

## Behind a reverse proxy

Many deployments terminate TLS at a reverse proxy or load balancer and forward plain HTTP to Tork on a private network. In that setup, leave TLS off here and add two middleware so the forwarded host and scheme are honored only from a proxy you trust.

```rust
use tork::App;
use tork::middleware::{ProxyHeaders, TrustedHost};

App::new()
    // Read X-Forwarded-Host and X-Forwarded-Proto, but only from a trusted peer.
    .middleware(ProxyHeaders::new().trust_loopback())
    // Reject any request whose Host is not in this allowlist.
    .middleware(TrustedHost::new(["example.com", "*.example.com"]));
```

`ProxyHeaders` ignores forwarded headers from any peer not in its trust list. Add trusted proxies with `trust_loopback()`, `trust_proxy(ip)`, or `trust_cidr(net)`. `TrustedHost` matches exact names or a single leading wildcard like `*.example.com`, and rejects anything else with `400`. Register `ProxyHeaders` before `TrustedHost` so the rewritten host is the one checked. See [Middleware](/docs/framework/getting-started/middleware) for the registration order rules.

## HTTP/3

HTTP/3 (QUIC) is not supported. It needs a separate UDP/QUIC stack. For now, use HTTP/2 over TLS.
