# Server Configuration

The `App` builder configures how the server runs: where it listens, how it shuts down, and the connection-level limits. TLS and protocol negotiation live in [TLS and HTTP/2](/docs/framework/security/tls).

## Serving

`serve` takes the listen address as a string and runs until a shutdown signal.

```rust
App::new()
    .include_router(api::router())
    .serve("0.0.0.0:8000")
    .await
```

The lifecycle runs in order: startup, bind, the `on_ready` hooks, the accept loop, drain, then shutdown. The socket binds only after startup finishes, so a client cannot connect before the app is ready. A bind failure (for example the port is in use) is returned as an error from `serve`.

## Graceful shutdown

`serve` listens for `SIGINT` (Ctrl-C) and, on Unix, `SIGTERM`. On Windows only Ctrl-C applies. When a signal arrives, the server stops accepting new connections and drains the in-flight ones, with a 15 second timeout. In-flight WebSocket connections are told to close cleanly so they are not dropped mid-message.

To drive shutdown yourself instead of waiting for a signal, use `serve_with_shutdown`. It takes a future that, when it resolves, begins the same graceful shutdown. This is also how tests stop the server.

```rust
use tokio::sync::oneshot;

let (stop_tx, stop_rx) = oneshot::channel::<()>();

let server = tokio::spawn(async move {
    App::new()
        .include_router(api::router())
        .serve_with_shutdown("0.0.0.0:8000", async move {
            let _ = stop_rx.await;
        })
        .await
});

// ... later, trigger a clean shutdown:
let _ = stop_tx.send(());
let _ = server.await;
```
