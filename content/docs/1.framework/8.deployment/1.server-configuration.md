# Server Configuration

The `App` builder configures how the server runs: where it listens, how it shuts down, and the connection-level limits. TLS and protocol negotiation live in [TLS and HTTP/2](/docs/framework/security/tls).

## Serving

`serve` takes the listen address as a string and runs until a shutdown signal.

```rust
App::new()
    .include_router(api::router())
    .serve("0.0.0.0:8000")
    .await
```

The lifecycle runs in order: startup, bind, the `on_ready` hooks, the accept loop, drain, then shutdown. The socket binds only after startup finishes, so a client cannot connect before the app is ready. A bind failure (for example the port is in use) is returned as an error from `serve`.

## Graceful shutdown

`serve` listens for `SIGINT` (Ctrl-C) and, on Unix, `SIGTERM`. On Windows only Ctrl-C applies. When a signal arrives, the server stops accepting new connections and drains the in-flight ones, with a 15 second timeout. In-flight WebSocket connections are told to close cleanly so they are not dropped mid-message.

To drive shutdown yourself instead of waiting for a signal, use `serve_with_shutdown`. It takes a future that, when it resolves, begins the same graceful shutdown. This is also how tests stop the server.

```rust
use tokio::sync::oneshot;

let (stop_tx, stop_rx) = oneshot::channel::<()>();

let server = tokio::spawn(async move {
    App::new()
        .include_router(api::router())
        .serve_with_shutdown("0.0.0.0:8000", async move {
            let _ = stop_rx.await;
        })
        .await
});

// ... later, trigger a clean shutdown:
let _ = stop_tx.send(());
let _ = server.await;
```

## Readiness

`on_ready` runs a hook once the socket is bound, just before the accept loop starts. The hook receives a `ReadyContext` whose `addr()` is the bound local address, which is handy when you bind to port `0` and need to know the port the OS chose.

```rust
use tork::{App, ReadyContext};

App::new()
    .include_router(api::router())
    .on_ready(|ctx: ReadyContext| async move {
        tracing::info!("listening on {}", ctx.addr());
        Ok(())
    })
    .serve("0.0.0.0:8000")
    .await
```

The hook returns `tork::Result<()>`, and returning an error aborts startup before any connection is served. Use `on_ready` for readiness signaling and logging. For work that sets up and tears down resources, use a lifespan or the `on_startup` and `on_shutdown` hooks, covered in [Lifespan](/docs/framework/getting-started/lifespan).

## Request body size

Cap the size of buffered request bodies with `max_request_body_size`. It applies to the `Json`, `Valid<T>`, and urlencoded `Form<T>` extractors, which enforce the cap as the body arrives, so an oversized body is rejected with `400` before it is fully buffered.

```rust
App::new().max_request_body_size(8 * 1024 * 1024); // default is 2 MiB
```

Multipart uploads have their own limits through `UploadConfig`, in [File Uploads](/docs/framework/data/file-uploads).

## Timeouts

```rust
use std::time::Duration;

App::new()
    .header_read_timeout(Duration::from_secs(30)) // slowloris guard, on by default
    .idle_timeout(Duration::from_secs(60));        // drop idle connections, off by default
```

`header_read_timeout` caps how long a client may take to send the request head, which bounds slowloris-style attacks. It defaults to 30 seconds and applies to HTTP/1. Disable it with `without_header_read_timeout()`, only behind a trusted proxy that already bounds slow clients.

`idle_timeout` closes a connection that has had no activity for too long. It is off by default, because an idle WebSocket or a quiet Server-Sent Events stream is normal. If you enable it, set it well above your SSE heartbeat.

## Connection tuning

HTTP/1 keep-alive is on by default. Tune the protocol layers with `Http1Config` and `Http2Config`.

```rust
use tork::{App, Http1Config, Http2Config};

App::new()
    .http1(Http1Config::new().keep_alive(true))
    .http2(Http2Config::new().max_concurrent_streams(256));
```

The full option list for both is in [TLS and HTTP/2](/docs/framework/security/tls).
