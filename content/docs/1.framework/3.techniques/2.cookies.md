# Handling Cookies

Tork works with cookies through the standard HTTP headers: you read the `Cookie` request
header to see what the client sent, and set `Set-Cookie` response headers to store a
cookie. There is no separate cookie type to learn, and no global cookie state.

## Reading cookies

There is no built-in cookie extractor, so read the `Cookie` header from the
`RequestContext` and pick out the value you want. The header is a `;`-separated list of
`name=value` pairs.

```rust
use tork::{get, RequestContext};

#[get("/whoami")]
pub async fn whoami(ctx: RequestContext) -> tork::Result<String> {
    let session = ctx
        .headers()
        .get(tork::header::COOKIE)
        .and_then(|value| value.to_str().ok())
        .unwrap_or("")
        .split(';')
        .map(str::trim)
        .find_map(|pair| pair.strip_prefix("session="))
        .unwrap_or("none");

    Ok(format!("session is: {session}"))
}
```

## Writing cookies

Set a cookie by adding a `Set-Cookie` header to the response. Return a `Response`, build it
with `json_response` to keep a body, and **`append`** the header. Use `append` rather than
`insert` so that setting several cookies produces one `Set-Cookie` line each; `insert` would
replace any previous one. The value is computed at runtime, so build it with
`HeaderValue::from_str` and handle the error.

::code-group

```rust [src/schemas/session.rs]
use tork::api_model;

#[api_model]
pub struct SessionOut {
    pub status: String,
}
```

```rust [src/routers/session.rs]
use tork::{api_router, post, header::SET_COOKIE, json_response, HeaderValue, Response, StatusCode};

use crate::schemas::session::SessionOut;

#[api_router(prefix = "/session", tags = ["session"])]
pub mod session_router {
    use super::*;

    #[post("/login")]
    pub async fn login() -> tork::Result<Response> {
        let token = "abc123"; // issued by your auth code
        let cookie = format!("session={token}; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=3600");

        let mut response = json_response(StatusCode::OK, &SessionOut { status: "ok".to_owned() });
        if let Ok(value) = HeaderValue::from_str(&cookie) {
            response.headers_mut().append(SET_COOKIE, value);
        }
        Ok(response)
    }
}

pub use session_router::router;
```

::

Set the attributes that match how the cookie is used:

- `Path=/` scopes the cookie to the whole site.
- `HttpOnly` keeps it out of JavaScript, which blunts XSS theft of a session.
- `Secure` sends it only over HTTPS.
- `SameSite=Lax` (or `Strict`) limits cross-site sending, which helps against CSRF.
- `Max-Age` (or `Expires`) controls its lifetime; omit both for a session cookie.

To delete a cookie, set it again with the same name, an empty value, and `Max-Age=0`.

## Testing with the cookie jar

The `TestClient` keeps a cookie jar: it records every `Set-Cookie` from a response and
replays it on later requests, so a login flow carries its session automatically across
calls. Seed a cookie with `.cookie(name, value)` on the builder.

```rust
use tork::testing::TestClient;
use tork::App;

#[tokio::test]
async fn login_sets_a_session_cookie() -> tork::Result<()> {
    let client = TestClient::builder(App::new().include_router(session::router()))
        .cookie("session", "seed-value") // seeded, sent on the first request
        .build()
        .await?;

    // The response's Set-Cookie is stored in the jar...
    let response = client.post("/session/login").send().await?;
    assert_eq!(response.status(), 200);

    // ...and replayed automatically on the next request.
    client.get("/whoami").send().await?;
    Ok(())
}
```

The jar keeps only the `name=value` pair and ignores cookie attributes (path, expiry,
flags), which is enough to carry a session through a test. See
[Testing](/docs/framework/fundamentals/testing) for the rest of the test client.
