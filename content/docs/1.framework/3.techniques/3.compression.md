# Response Compression

Tork ships a `Compression` middleware that gzip-compresses response bodies before they are
sent, trading a little CPU for less bandwidth. It only acts when it is safe and worthwhile:
the client must support gzip, the response must not already be encoded, and the body must
fall inside a size window.

## Enabling compression

Add the `Compression` middleware and turn on gzip with `.gzip()`. On its own,
`Compression::new()` enables no algorithm and does nothing, so the `.gzip()` call is what
actually switches compression on.

```rust
use tork::{middleware::Compression, App};

App::new()
    .middleware(Compression::new().gzip())
    .include_router(api::router())
    .serve("0.0.0.0:8000")
    .await
```

gzip is the only algorithm at the moment. The method is named `gzip()` (rather than a
generic `enable()`) so other codings can be added later without breaking this call.

## The size window

Compression buffers the whole body in memory and produces a second buffer for the gzip
output, so very small bodies are not worth it and very large ones cost too much memory. The
`minimum_size` and `maximum_size` bounds (in bytes) define the eligible window.

```rust
use tork::middleware::Compression;

Compression::new()
    .gzip()
    .minimum_size(256)            // skip bodies smaller than 256 bytes
    .maximum_size(2 * 1024 * 1024); // cap at 2 MiB (default is 8 MiB)
```

`minimum_size` defaults to `0` and `maximum_size` to 8 MiB. When a response advertises a
`Content-Length` outside the window, it is passed through without being buffered at all,
which keeps peak memory bounded for large payloads. Pass `usize::MAX` to lift the upper
cap.

## When a response is compressed

For an enabled middleware, a response is gzip-compressed only when all of these hold:

- the request's `Accept-Encoding` includes gzip;
- the response has no existing `Content-Encoding`;
- the response is not a Server-Sent Events stream (`text/event-stream`);
- the body size is within `[minimum_size, maximum_size]`.

When it compresses, it sets `Content-Encoding: gzip` and recomputes `Content-Length` for
the smaller body. If gzip encoding somehow fails, the original body is sent uncompressed
rather than erroring.

## The Vary header

Whenever gzip is enabled, every eligible (non-stream) response gets `Vary: Accept-Encoding`,
even the ones that end up uncompressed. This tells caching proxies to key the compressed and
uncompressed variants separately, so a cache never hands a gzip body to a client that did
not ask for one.

## Notes

- **Register once.** Like other built-in middleware, `Compression` can be added only once;
  a second registration makes the app fail to build.
- **Streams bypass it.** SSE responses are never buffered or compressed, so long-lived
  streams stay responsive.

See [Middleware](/docs/framework/getting-started/middleware) for how middleware is ordered
and for the other built-ins.
