# Request and Response Headers

Headers are a standard part of any API. In Tork you read request headers from the
`RequestContext` and set response headers on a `Response`, either in a handler or in
middleware.

## Reading request headers

Take a `RequestContext` parameter in your handler and call `headers()`, which returns the
request's `HeaderMap`. A header value is bytes, so convert it to a string with `to_str`
before using it.

```rust
use tork::{get, RequestContext};

#[get("/user-agent")]
pub async fn user_agent(ctx: RequestContext) -> tork::Result<String> {
    let ua = ctx
        .headers()
        .get("user-agent")
        .and_then(|value| value.to_str().ok())
        .unwrap_or("unknown");

    Ok(format!("Your client is: {ua}"))
}
```

The same context exposes the rest of the request line when you need it: `method()`,
`uri()` (with the query string), `peer_addr()` for the remote address, and `scheme()` for
the effective scheme after trusted-proxy normalization.

## Pulling a bearer token

For the common `Authorization: Bearer <token>` case, take the built-in `BearerToken`
extractor instead of reading the header by hand. It parses the header and rejects the
request with `401 Unauthorized` when the header is missing or is not a valid bearer token,
so the handler only runs for a well-formed token. Call `token()` for the raw value (the
part after `Bearer `).

```rust
use tork::{get, BearerToken};

#[get("/secure")]
pub async fn secure(token: BearerToken) -> tork::Result<String> {
    Ok(format!("authenticated with: {}", token.token()))
}
```

`BearerToken` only parses the header. Verifying the token (signature, expiry, claims) is
layered on top in application code, typically a `#[tork::dependency]` that itself takes a
`BearerToken`.

## Setting response headers

A handler that needs to set a header returns a `Response` and inserts into its
`headers_mut()`. Build the response with `json_response` to keep a JSON body (returning a
bare status code would send an empty body). `HeaderName` and `HeaderValue` are re-exported
from `tork`; use `from_static` for constant strings and `from_str` for computed values.

::code-group

```rust [src/schemas/info.rs]
use tork::api_model;

#[api_model]
pub struct Info {
    pub name: String,
    pub version: String,
}
```

```rust [src/routers/meta.rs]
use tork::{api_router, get, header::HeaderName, json_response, HeaderValue, Response, StatusCode};

use crate::schemas::info::Info;

#[api_router(tags = ["meta"])]
pub mod meta_router {
    use super::*;

    #[get("/info")]
    pub async fn info() -> tork::Result<Response> {
        let payload = Info { name: "my-api".to_owned(), version: "1.0.0".to_owned() };

        let mut response = json_response(StatusCode::OK, &payload);
        response.headers_mut().insert(
            HeaderName::from_static("x-api-version"),
            HeaderValue::from_static("1.0.0"),
        );
        Ok(response)
    }
}

pub use meta_router::router;
```

::

`insert` replaces any existing value for that header; use `append` to add another value
under the same name. A `HeaderValue::from_str` call returns a `Result`, since an arbitrary
string may not be a legal header value, so handle the error rather than unwrapping on
untrusted input.

## Setting headers in middleware

To touch headers for many routes at once, do it in middleware. A middleware can add request
headers before the handler sees them, and response headers after the handler runs. This
example records how long handling took.

```rust
use tork::{middleware, header::HeaderName, HeaderValue, Next, Request, Response};

#[middleware]
pub async fn add_process_time(req: Request, next: Next) -> tork::Result<Response> {
    let start = std::time::Instant::now();
    let mut response = next.run(req).await?;
    let elapsed = start.elapsed().as_secs_f64();

    if let Ok(value) = HeaderValue::from_str(&format!("{elapsed:.6}")) {
        response.headers_mut().insert(
            HeaderName::from_static("x-process-time"),
            value,
        );
    }
    Ok(response)
}
```

## Built-in middleware for headers

You rarely need to set common headers by hand. Tork ships middleware that manage them:
`SecurityHeaders` adds a baseline of security headers, `Cors` writes the CORS headers,
`Compression` sets `Content-Encoding` and `Vary`, and `RequestId` echoes a request id. See
[Middleware](/docs/framework/getting-started/middleware) for the full list and their
options.
