# Streaming Files and Responses

Tork buffers response bodies by default: a handler returns a value and the framework
serializes it into one contiguous body. For a large download or a body generated on the fly,
stream it instead, sending frames over time so the whole payload never sits in memory at
once.

The response body type, `RespBody`, has three constructors:

| Constructor | Use |
| --- | --- |
| `RespBody::new(bytes)` | A fully-buffered body (what handlers normally produce). |
| `RespBody::stream(body)` | A streaming body that yields frames over time. |
| `RespBody::stream_capped(body, max_bytes)` | A streaming body that errors once it emits more than `max_bytes`. |

## Streaming a response

To stream, build a frame body and wrap it with `RespBody::stream`, then return a `Response`.
A streaming body is any `http_body::Body` whose data is `Bytes` and whose error is
`tork::BoxError`; `http_body_util::StreamBody` adapts an ordinary stream of frames into one.

```rust
use bytes::Bytes;
use http_body::Frame;
use http_body_util::StreamBody;
use tork::{get, BoxError, RespBody, Response, StatusCode};

#[get("/stream")]
pub async fn stream_numbers() -> tork::Result<Response> {
    let frames = futures_util::stream::iter(vec![
        Ok::<_, BoxError>(Frame::data(Bytes::from("chunk one\n"))),
        Ok(Frame::data(Bytes::from("chunk two\n"))),
    ]);

    let mut response = Response::new(RespBody::stream(StreamBody::new(frames)));
    *response.status_mut() = StatusCode::OK;
    Ok(response)
}
```

`Frame`, `StreamBody`, and the stream combinators come from the standard ecosystem crates
(`http-body`, `http-body-util`, `futures-util`); add the ones you use to your `Cargo.toml`.

## Sending a file download

There is no dedicated file-download helper, so a download is a streamed `Response` with two
headers set: `Content-Type` for the media type, and `Content-Disposition: attachment` to
tell the browser to save it under a name rather than render it. Stream the bytes from disk so
a large file is never fully buffered.

```rust
use futures_util::StreamExt;
use http_body::Frame;
use http_body_util::StreamBody;
use tokio_util::io::ReaderStream;
use tork::{get, header, BoxError, HeaderValue, RespBody, Response, StatusCode};

#[get("/download/report")]
pub async fn download_report() -> tork::Result<Response> {
    let file = tokio::fs::File::open("/var/data/report.pdf")
        .await
        .map_err(|_| tork::Error::not_found("report not found"))?;

    // Each filesystem chunk becomes a body frame.
    let frames = ReaderStream::new(file)
        .map(|chunk| chunk.map(Frame::data).map_err(BoxError::from));

    let mut response = Response::new(RespBody::stream(StreamBody::new(frames)));
    *response.status_mut() = StatusCode::OK;
    let headers = response.headers_mut();
    headers.insert(header::CONTENT_TYPE, HeaderValue::from_static("application/pdf"));
    headers.insert(
        header::CONTENT_DISPOSITION,
        HeaderValue::from_static("attachment; filename=\"report.pdf\""),
    );
    Ok(response)
}
```

When the filename is dynamic, build the `Content-Disposition` value with `HeaderValue::from_str`
and handle the error, and keep user-supplied names out of the header unless you have
sanitized them.

## Capping a runaway stream

An unbounded stream (a download backed by a generator, say) has no inherent size limit, so a
buggy or runaway producer could stream without end. `RespBody::stream_capped` wraps the body
and errors the response once it has emitted more than the given number of bytes.

```rust
use tork::RespBody;

// Abort the stream if it tries to emit more than 10 MiB.
let body = RespBody::stream_capped(StreamBody::new(frames), 10 * 1024 * 1024);
```

Frames within the limit are delivered as normal; the frame that pushes the total over the cap
fails the stream. Server-Sent Events are intentionally open-ended and use `RespBody::stream`
instead, so do not cap them this way.

## Reading an uploaded file in chunks

When you already hold an `UploadFile` (from a multipart upload), you can stream through its
contents without loading it all into memory. Its read and save operations are async and run
off the runtime.

```rust
use tork::{post, UploadFile};

#[post("/process")]
pub async fn process(mut file: UploadFile) -> tork::Result<()> {
    while let Some(chunk) = file.read_chunk(64 * 1024).await? {
        // process each chunk...
        let _ = chunk;
    }

    file.seek_start().await?;                       // rewind before reusing
    file.save_to_dir("/var/uploads", "cv.pdf").await?;
    Ok(())
}
```

Use `read()` for the whole file at once, `read_chunk(n)` to stream through it, `seek_start()`
to rewind, and `save_to`/`save_to_dir` to persist it under a safe path. See
[File Uploads](/docs/framework/techniques/file-uploads) for the upload types, limits, and the
path-safety guarantees of the save methods.

## Server-Sent Events

For a long-lived event stream rather than a one-shot download, use the higher-level
Server-Sent Events API (`Sse`/`SseEvent`), which builds on `RespBody::stream` and adds event
framing, heartbeats, and resumption. See [Server-Sent Events](/docs/framework/realtime/server-sent-events).
