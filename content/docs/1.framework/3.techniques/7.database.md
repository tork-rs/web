# Database

Tork has no built-in ORM. Database access lives in `tork-orm`, a standalone crate, and the framework wires it in through the same resource and dependency injection system used for everything else. You open a connection pool once at startup, register it as `Arc<Database>`, and inject it wherever you need it. With the `tork` feature enabled, ORM errors turn into HTTP responses on their own, so a missing row answers with a 404 and you write no extra code for it.

This guide covers connecting, injecting the pool, error mapping, transactions, migrations, and tests. For the full query, mutation, and migration API, see the [ORM docs](/docs/orm/getting-started/introduction).

## Setup

Add the framework and the ORM, with the integration feature enabled.

```toml
[dependencies]
tork = { git = "https://github.com/tork-rs/tork-framework.git" }
tork-orm = { git = "https://github.com/tork-rs/tork-orm.git", default-features = false, features = ["sqlite", "tork", "migrations"] }
tokio = { version = "1", features = ["rt-multi-thread", "macros"] }
```

Three features are in play:

- `sqlite` picks the backend. Swap it for `postgres` or `mysql` when you target those.
- `migrations` enables the migration engine.
- `tork` maps ORM errors to framework responses and lets the pool flow through dependency injection.

The example reads its connection settings from the environment:

```bash
DB_URL=sqlite::memory:
DB_MAX_CONNECTIONS=5
```

`sqlite::memory:` keeps the example self-contained. Point `DB_URL` at `sqlite://app.db` to store data on disk.

## Project layout

The code splits into one file per concern: the connection, the model, the request and response schemas, a repository, the routes, and the entrypoint.

::code-group

```rust [src/db.rs]
use std::sync::Arc;

use tork::{settings, LifespanContext, Resources, Result};
use tork_orm::migration::FileMigrator;
use tork_orm::prelude::*;

#[settings(prefix = "DB")]
pub struct DatabaseConfig {
    #[setting(default = "sqlite::memory:")]
    pub url: String,
    #[setting(default = 5, ge = 1, le = 64)]
    pub max_connections: u32,
}

// The struct registered with the app. Every `#[resource]` field becomes
// injectable by its type, so handlers can ask for `Arc<Database>`.
#[derive(Clone, Resources)]
pub struct Db {
    #[resource]
    pub database: Arc<Database>,
}

#[tork::lifespan]
impl Db {
    // Runs once before the server accepts requests.
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        let config = DatabaseConfig::load()?;
        let database = Database::connect(&config.url, config.max_connections).await?;

        // Apply any pending migration files before serving traffic.
        // `up()` returns the migrations it ran this time, so you can log them.
        let applied = FileMigrator::new(
            database.clone(),
            concat!(env!("CARGO_MANIFEST_DIR"), "/migrations"),
        )
        .up()
        .await?;
        if !applied.is_empty() {
            tork::Logger::new("Database")
                .info(format!("Applied {} migration(s)", applied.len()))
                .emit();
        }

        Ok(Db {
            database: Arc::new(database),
        })
    }

    // Runs once during shutdown.
    async fn shutdown(self) -> Result<()> {
        self.database.close().await;
        Ok(())
    }
}
```

```rust [src/schemas.rs]
use tork::api_model;

use crate::models::User;

// What the API returns. Keeping it separate from the model means database
// columns and the public shape can change independently.
#[api_model]
pub struct UserOut {
    pub id: i64,
    pub username: String,
    pub email: String,
    pub is_active: bool,
}

impl From<User> for UserOut {
    fn from(user: User) -> Self {
        Self {
            id: user.id,
            username: user.username,
            email: user.email,
            is_active: user.is_active,
        }
    }
}

// What the API accepts. The validation rules run before the handler.
#[api_model]
pub struct CreateUserInput {
    #[field(min_length = 2, max_length = 50)]
    pub username: String,
    #[field(min_length = 3, max_length = 255)]
    pub email: String,
}
```

```rust [src/repositories.rs]
use std::sync::Arc;

use tork::Inject;
use tork_orm::prelude::*;

use crate::models::User;
use crate::schemas::{CreateUserInput, UserOut};

// A repository groups the queries for one model. `#[derive(Inject)]` pulls
// `Arc<Database>` from the app, so handlers ask for `UserRepository` by type.
#[derive(Inject)]
pub struct UserRepository {
    db: Arc<Database>,
}

impl UserRepository {
    pub async fn list_active(&self) -> Result<Vec<UserOut>> {
        list_active_users(&self.db).await
    }

    pub async fn create(&self, input: CreateUserInput) -> Result<UserOut> {
        let user = User::create(
            &self.db,
            &User {
                id: 0,
                username: input.username,
                email: input.email,
                is_active: true,
            },
        )
        .await?;

        Ok(UserOut::from(user))
    }
}

// A free function that takes any `Executor` runs the same way on the pool or
// inside a transaction. See the Executor section below.
pub async fn list_active_users(executor: impl Executor) -> Result<Vec<UserOut>> {
    let users = User::query()
        .filter(User::is_active.eq(true))
        .order_by(User::username.asc())
        .all(executor)
        .await?;

    Ok(users.into_iter().map(UserOut::from).collect())
}
```

```rust [src/routers.rs]
use std::sync::Arc;

use tork::{api_router, get, post, Valid};
use tork_orm::prelude::Database;

use crate::models::User;
use crate::repositories::UserRepository;
use crate::schemas::{CreateUserInput, UserOut};

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("", response_model = Vec<UserOut>, summary = "List users")]
    pub async fn list_users(repo: UserRepository) -> tork::Result<Vec<UserOut>> {
        Ok(repo.list_active().await?)
    }

    #[get("/{user_id}", response_model = UserOut, summary = "Get user by id")]
    pub async fn get_user(user_id: i64, db: Arc<Database>) -> tork::Result<UserOut> {
        // A missing row returns 404 on its own. See "Errors become responses".
        let user = User::find(&*db, user_id).await?;
        Ok(UserOut::from(user))
    }

    #[post("", response_model = UserOut, status_code = 201, summary = "Create user")]
    pub async fn create_user(
        repo: UserRepository,
        body: Valid<CreateUserInput>,
    ) -> tork::Result<UserOut> {
        Ok(repo.create(body.into_inner()).await?)
    }
}

pub use users_router::router;
```

## Injecting the pool

`Arc<Database>` is injectable by type, so any handler or service can ask for it directly:

```rust
use std::sync::Arc;

use tork::get;
use tork_orm::prelude::*;

use crate::models::User;

#[get("/users/count")]
pub async fn count_users(db: Arc<Database>) -> tork::Result<i64> {
    let total = User::query().count(&*db).await?;
    Ok(total)
}
```

Cloning the handle copies a pointer, not the pool, so passing it around is cheap. Inject it straight into handlers for one-off queries, or wrap it in a repository with `#[derive(Inject)]` (as `UserRepository` does above) when you want to group queries behind named methods.

## Errors become responses

The `tork` feature adds a conversion from the ORM error type into a framework error. Any query you follow with `?` flows through the handler and becomes an HTTP response, so the common cases need no handling code:

| ORM outcome | HTTP status |
| --- | --- |
| Connection failure | `503 Service Unavailable` |
| Row not found (a `find` or `one` miss) | `404 Not Found` |
| Optimistic-lock conflict | `409 Conflict` |
| Anything else (query, conversion, config) | `500 Internal Server Error` |

That is why `get_user` above needs no branch for the missing case:

```rust
#[get("/{user_id}", response_model = UserOut, summary = "Get user by id")]
pub async fn get_user(user_id: i64, db: Arc<Database>) -> tork::Result<UserOut> {
    let user = User::find(&*db, user_id).await?; // missing id answers 404
    Ok(UserOut::from(user))
}
```

The original ORM error stays attached as the source, so you can take over the response for one error type. Register an exception handler for `OrmError` to log it or change the body:

```rust
use tork::{App, ErrorContext, IntoResponse, Response};
use tork_orm::OrmError;

async fn handle_orm_error(error: OrmError, _ctx: ErrorContext) -> Response {
    tork::Error::service_unavailable("the database is busy, try again")
        .into_response()
}

App::new()
    .lifespan::<Db>()
    .include_router(routers::router())
    .exception_handler::<OrmError, _, _>(handle_orm_error);
```

The full pattern, including `ErrorContext`, lives in [Error Handling](/docs/framework/getting-started/error-handling-and-exceptions).

## One query, two callers: the Executor

`Database`, `&Database`, `Arc<Database>`, `Transaction`, and `&Transaction` all implement `Executor`. A query helper that accepts `impl Executor` therefore runs the same way whether you call it on the pool or inside a transaction. That is why `list_active_users` in `src/repositories.rs` is a free function:

```rust
pub async fn list_active_users(executor: impl Executor) -> Result<Vec<UserOut>> {
    let users = User::query()
        .filter(User::is_active.eq(true))
        .order_by(User::username.asc())
        .all(executor)
        .await?;

    Ok(users.into_iter().map(UserOut::from).collect())
}
```

Call it with `&self.db` in a handler, or with the transaction handle inside a `transaction!` block. You write the query once.

## Transactions

For writes that must succeed or fail together, wrap them in `transaction!`. The closure receives a transaction handle. Return `Ok` and the work commits, return `Err` and it rolls back:

```rust
use tork_orm::{prelude::*, transaction};

let created = transaction!(db, |tx| async move {
    let user = User::create(
        tx,
        &User {
            id: 0,
            username: "alice".into(),
            email: "alice@example.com".into(),
            is_active: true,
        },
    )
    .await?;

    // `tx` is an Executor, so the shared helper works here too.
    let active = list_active_users(tx).await?;

    Ok((user, active))
})
.await?;
```

The full transaction API, including isolation levels, lives in [Transactions](/docs/orm/querying/transactions).

## Running migrations

`FileMigrator::new(database.clone(), dir).up()` applies the `.sql` files in `dir` that have not run yet, in order, and records each one so it runs only once. Running it in `startup` keeps the schema current every time the app boots.

If you would rather migrate before a deploy instead of at boot, the same files run from the command line with `tork migrate up`. Leave the `FileMigrator` call in for local development, or drop it once the deploy step owns migrations. The file format and the CLI are covered in [Migrations](/docs/orm/schema/migrations).

## Trying it out

Run the app:

```bash
cargo run
```

Then create a user and read it back:

```bash
# Create a user
curl -X POST http://localhost:8000/users \
  -H 'content-type: application/json' \
  -d '{"username": "alice", "email": "alice@example.com"}'

# List active users
curl http://localhost:8000/users

# A missing id returns 404, no extra code in the handler
curl -i http://localhost:8000/users/9999
```

## Testing

Point the app at a fresh in-memory database in tests instead of your real one. Build the app, register a migrated test database as a resource override, and send requests with `TestClient`. The override replaces `Arc<Database>` for the test, so no handler changes are needed.

```rust [tests/users.rs]
use std::sync::Arc;

use serde_json::json;
use tork::testing::TestClient;
use tork::App;
use tork_orm::migration::FileMigrator;
use tork_orm::prelude::*;

use myapp::routers;

// A clean database for each test: connect, migrate, hand back the pool.
async fn test_db() -> tork::Result<Arc<Database>> {
    let db = Database::connect("sqlite::memory:", 1).await?;
    FileMigrator::new(db.clone(), concat!(env!("CARGO_MANIFEST_DIR"), "/migrations"))
        .up()
        .await?;
    Ok(Arc::new(db))
}

#[tokio::test]
async fn create_then_list() -> tork::Result<()> {
    let app = App::new().include_router(routers::router());
    let client = TestClient::builder(app)
        .resource(test_db().await?)
        .build()
        .await?;

    let created = client
        .post("/users")
        .json(&json!({ "username": "alice", "email": "alice@example.com" }))
        .send()
        .await?;
    assert_eq!(created.status(), 201);

    let listed = client.get("/users").send().await?;
    assert_eq!(listed.status(), 200);

    Ok(())
}
```

The full testing API, including dependency overrides and log assertions, is in [Testing](/docs/framework/fundamentals/testing).
