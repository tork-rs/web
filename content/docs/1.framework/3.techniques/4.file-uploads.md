# Handling File Uploads

File uploads arrive as `multipart/form-data`. Tork parses the body once and hands each part
to a typed field: a file binds to `FileBytes` or `UploadFile`, and a text field binds to a
normal value. You read a form with a parameter style for quick handlers or a model style for
anything you want validated and documented.

## The two file types

A file field binds to one of two types, depending on how large the file is and what you do
with it.

- **`FileBytes`** holds the whole file in memory. Use it for small files you read in one go:
  an avatar, a CSV, a thumbnail.
- **`UploadFile`** is spooled: it stays in memory up to a threshold, then spills to a
  temporary file on disk. Use it for large or unbounded uploads you stream to their final
  destination.

`FileBytes` exposes its bytes directly:

```rust
file.len();            // usize, size in bytes
file.is_empty();       // bool
file.bytes();          // &[u8]
file.into_bytes();     // Bytes, by value
file.filename();       // Option<&str>, the client-supplied name
file.content_type();   // Option<&Mime>, the declared type
```

`UploadFile` carries metadata up front and reads its contents through async methods, so the
blocking disk IO runs off the runtime:

```rust
file.filename();       // Option<&str>
file.content_type();   // Option<&Mime>
file.size();           // u64, total bytes

let all = file.read().await?;                       // read it all into Bytes
while let Some(chunk) = file.read_chunk(64 * 1024).await? { /* ... */ }
file.seek_start().await?;                            // rewind
```

`Mime` is re-exported as `tork::mime::Mime`, so you can name it without depending on the
`mime` crate directly.

## Reading a form: two styles

A handler declares exactly one body. Mixing a JSON body (`Valid`/`Json`) with form or file
parameters, or combining `Form` and `Multipart`, is a compile error.

### Parameter style

Annotate each parameter with `#[file]` for a file field or `#[form]` for a text field; the
field name defaults to the parameter name. A bare `FileBytes` or `UploadFile` parameter is
treated as a file field even without the attribute. Use `name = "..."` when the form field
name differs from the parameter. `Option<T>` makes a field optional and `Vec<T>` accepts it
repeated.

```rust
use tork::{post, FileBytes, UploadFile};

#[post("/files")]
pub async fn create(
    #[file(max_size = "2MB", content_types = ["image/png"])] avatar: FileBytes,
    #[file] documents: Vec<UploadFile>, // zero or more under "documents"
    #[form] caption: Option<String>,    // optional text field
) -> tork::Result<String> {
    Ok(format!("{} bytes, {} docs", avatar.len(), documents.len()))
}
```

### Model style

For a real endpoint, describe the form once with `#[derive(FormModel)]` and take it as
`Multipart<T>`. Each field becomes a form field, validated and documented. `#[file(...)]`
configures a file field and `#[field(...)]` adds the text constraints from
[Request Data](/docs/framework/getting-started/request-data).

::code-group

```rust [src/schemas/profile.rs]
use tork::{FileBytes, FormModel, UploadFile};

#[derive(FormModel)]
pub struct ProfileForm {
    #[file(max_size = "2MB", content_types = ["image/png", "image/jpeg"], sniff = true)]
    pub avatar: FileBytes,

    #[file(max_size = "20MB")]
    pub attachment: Option<UploadFile>,

    #[field(min_length = 2, max_length = 40)]
    pub display_name: String,

    pub tags: Vec<String>,
}
```

```rust [src/routers/profile.rs]
use tork::{api_router, post, Multipart};

use crate::schemas::profile::ProfileForm;

#[api_router(prefix = "/profile", tags = ["profile"])]
pub mod profile_router {
    use super::*;

    #[post("/")]
    pub async fn save(form: Multipart<ProfileForm>) -> tork::Result<usize> {
        let form = form.into_inner();
        // form.avatar, form.attachment, form.display_name, form.tags
        Ok(form.avatar.len())
    }
}

pub use profile_router::router;
```

## Validating a file

`#[file(...)]` accepts the same options on a `FormModel` field and on a `#[file]` parameter:

- `max_size = "5MB"` — reject a larger file with `413`. Sizes accept `B`, `KB`/`KiB`, and
  `MB`/`MiB` suffixes (case-insensitive; `MB` means mebibytes), or a plain byte count.
- `content_types = ["image/png", "application/pdf"]` — allow only these declared content
  types; anything else returns `422`.
- `sniff = true` — inspect the file's leading bytes and reject it when the real type does not
  match an allowed one. This catches a GIF renamed to `.png`. It complements `content_types`
  rather than replacing it; for an `UploadFile` it sniffs the first 512 bytes.

```rust
#[file(max_size = "5MB", content_types = ["image/png", "image/jpeg"], sniff = true)]
pub avatar: FileBytes,
```

A failed text `#[field(...)]` constraint returns `422` with a `VALIDATION_ERROR` body, the
same as a JSON model.

## Saving an upload to disk

`UploadFile` writes its contents to disk through two async helpers that guard against path
traversal, so a client-supplied filename cannot escape your target directory.

```rust
// Into a directory under a single, safe file name. Returns the final path.
let path = file.save_to_dir("/var/uploads", "photo.jpg").await?;

// To an explicit relative path (absolute paths and `..` are rejected).
file.save_to("uploads/photo.jpg").await?;
```

`save_to_dir` requires `file_name` to be a single path component (no separators, no `..`) and
refuses to overwrite a symlink; `save_to` rejects an absolute path or any `..` component.
Both run on a blocking thread so a large write does not stall other requests. Never feed a
raw `file.filename()` from the client straight into a path — pass it through `save_to_dir`,
which sanitizes it.

## Upload limits

Uploads are bounded so a large or malicious request cannot exhaust memory or disk. Set
application-wide defaults with `App::upload_config`.

```rust
use tork::{App, UploadConfig};

App::new()
    .upload_config(
        UploadConfig::new()
            .max_body_size_mb(32)       // whole request
            .max_file_size_mb(16)       // per file
            .max_text_field_size_kb(64) // per text field
            .memory_threshold(1 << 20)  // spool to disk past 1 MiB
            .max_files(16)              // file parts
            .max_fields(1000)           // total parts (text + file)
            .temp_dir("/var/tmp"),      // where spilled files land
    )
    .include_router(profile::router())
```

The defaults, and what exceeding each limit returns:

| Setting | Default | Exceeding it returns |
| --- | --- | --- |
| `max_body_size` | 16 MiB | `413 Payload Too Large` |
| `max_file_size` | 8 MiB | `413 Payload Too Large` (`FILE_TOO_LARGE`) |
| `max_text_field_size` | 1 MiB | `413 Payload Too Large` (`FIELD_TOO_LARGE`) |
| `max_files` | 32 | `400 Bad Request` (`TOO_MANY_FILES`) |
| `max_fields` | 1000 | `422 Unprocessable Entity` (`TOO_MANY_FIELDS`) |
| `memory_threshold` | 1 MiB | spools to disk past this size |
| `temp_dir` | system temp | — |

Override the limits for a single route with `upload(...)` on the route attribute; route
values merge over the application defaults.

```rust
#[post("/upload", upload(max_file_size = "25MB", memory_threshold = "2MB"))]
pub async fn upload(form: Multipart<ProfileForm>) -> tork::Result<()> {
    Ok(())
}
```

Set `temp_dir` to a real disk-backed directory in containers where the default `/tmp` is a
memory-backed `tmpfs`, otherwise large uploads that "spill to disk" actually stay in RAM and
can trigger an out-of-memory kill.

## OpenAPI

Upload endpoints appear in the generated document with the right media type: a `Multipart<T>`
or parameter-based file handler is documented as `multipart/form-data` with file fields
marked `format: binary`, and a urlencoded `Form<T>` as `application/x-www-form-urlencoded`.
See [OpenAPI and Docs](/docs/framework/fundamentals/openapi). For plain forms without files,
see the form section in [Request Data](/docs/framework/getting-started/request-data).
