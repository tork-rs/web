# Rate Limiting

Tork has a built-in throttler. You define named policies once on the app, then apply them with a `throttle` attribute on routes and routers. A request over the limit gets back `429 Too Many Requests` with a `Retry-After` header and the error code `RATE_LIMITED`.

Counts are tracked per route and per key. The key is the client IP by default, so each caller gets their own budget on each endpoint. Without `App::throttle(...)`, every `throttle` attribute is a no-op, so you can annotate routes first and turn limiting on later.

## Defining policies

Configure the throttler on the app. A policy has a name, a request limit, and a window in seconds. Mark one policy as the default to apply it to every route that does not set its own.

```rust
use tork::{App, Throttle};

App::new().throttle(
    Throttle::new()
        .policy("default", 100, 60) // 100 requests per 60s
        .policy("strict", 5, 60)    // 5 requests per 60s
        .default("default"),
);
```

The limit is a `u32` and the window is in whole seconds. A window of `0` is treated as `1`.

## Applying a policy

Add the `throttle` attribute to a route. It takes a named policy, an inline limit, or `"skip"`.

```rust
use tork::get;

// Use a named policy defined on the app.
#[get("/feed", throttle = "strict")]
pub async fn feed() -> tork::Result<&'static str> {
    Ok("feed")
}

// Set the limit and window inline, no named policy needed.
#[get("/search", throttle(limit = 3, ttl = 60))]
pub async fn search() -> tork::Result<&'static str> {
    Ok("results")
}

// Opt a route out of limiting.
#[get("/health", throttle = "skip")]
pub async fn health() -> tork::Result<&'static str> {
    Ok("ok")
}
```

`ttl` is the window in seconds. A named policy and an inline limit cannot be mixed in the same attribute: use one form or the other.
