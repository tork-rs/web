# Redis Integration

Redis is more than a cache: idempotency keys, distributed locks, atomic counters, Lua
scripts, pub/sub. Tork does not wrap those. It manages the connection and dependency
injection and hands you the raw [`redis`](https://docs.rs/redis) client for everything else.
The same handle can also back the application cache.

Everything on this page is behind the `redis` feature.

```toml
tork = { version = "0.1", features = ["redis"] }
```

## Connecting and registering

Connect once at startup and register the handle with `App::redis`. A `Redis` is a
cheap-to-clone handle over one multiplexed, auto-reconnecting connection, so cloning it (which
happens on every injection) shares that single connection.

```rust
use tork::{App, Redis};

#[tork::main]
async fn main() -> tork::Result<()> {
    let redis = Redis::connect("redis://127.0.0.1:6379").await?;

    App::new()
        .redis(redis)
        .include_router(stats::router())
        .serve("0.0.0.0:8000")
        .await
}
```

## Running commands

Inject `Redis` into any handler or service by type. `query` runs a prepared command and
decodes its reply; the `redis` crate is re-exported as `tork::redis`, so you build commands
without adding your own dependency (and stay on the same client version).

::code-group

```rust [src/routers/stats.rs]
use tork::{api_router, get, redis, Redis};

#[api_router(tags = ["stats"])]
pub mod stats_router {
    use super::*;

    #[get("/incr/{key}")]
    pub async fn increment(key: String, redis: Redis) -> tork::Result<i64> {
        // `redis::cmd(...).arg(...)` is a `&mut Cmd`, which `query` takes as `&Cmd`.
        let count: i64 = redis.query(redis::cmd("INCR").arg(&key)).await?;
        Ok(count)
    }
}

pub use stats_router::router;
```

```rust [src/main.rs]
use tork::{App, Redis};

use crate::routers::stats;

#[tork::main]
async fn main() -> tork::Result<()> {
    let redis = Redis::connect("redis://127.0.0.1:6379").await?;

    App::new()
        .redis(redis)
        .include_router(stats::router())
        .serve("0.0.0.0:8000")
        .await
}
```

::

`query` returns `tork::Result<T>` where `T` is any `redis::FromRedisValue`, so an `Option<T>`
captures a command that may return nil. This makes idempotency-style writes natural:

```rust
// SET key value NX: returns the value only the first time, None afterwards.
let first: Option<String> = redis
    .query(redis::cmd("SET").arg("idem:abc").arg("1").arg("NX").arg("EX").arg(60))
    .await?;
let took_the_key = first.is_some();
```

For anything the convenience method does not cover (Lua scripts, pipelines, pub/sub), reach
for `connection()`, which returns a cloned multiplexed connection to use with the full `redis`
API.

```rust
let count: i64 = redis::Script::new("return redis.call('INCR', KEYS[1])")
    .key("hits")
    .invoke_async(&mut redis.connection())
    .await
    .map_err(|error| tork::Error::internal(error.to_string()))?;
```

If no `Redis` was registered, injecting one fails with a clear internal error pointing you to
`App::redis(...)`.

## Backing the cache with Redis

Tork's `Cache` is a small, store-agnostic key/value cache with TTL. By default it uses an
in-memory store; point it at Redis to share state across processes. `Cache::from_redis` reuses
an existing handle (so the cache and your raw commands share one connection), while
`Cache::redis(url)` opens its own.

```rust
use tork::{App, Cache, Redis};

let redis = Redis::connect("redis://127.0.0.1:6379").await?;

App::new()
    .redis(redis.clone())
    .cache(Cache::from_redis(&redis)) // shares redis's connection
    .include_router(stats::router());
```

Inject `Cache` by type, the same as `Redis`. Values are JSON-serialized, so they need
`Serialize`/`DeserializeOwned`.

```rust
use tork::{get, Cache};

#[get("/profile/{id}")]
pub async fn profile(id: i64, cache: Cache) -> tork::Result<Profile> {
    // Cache-aside: a hit returns immediately; a miss runs the closure once and stores it.
    let profile = cache
        .get_or_set(&format!("profile:{id}"), None, || async move { load_profile(id).await })
        .await?;
    Ok(profile)
}
```

The `Cache` API:

| Method | Effect |
| --- | --- |
| `get::<T>(key)` | Returns the value, or `None` if absent or expired. |
| `set(key, &value)` | Stores the value with the cache's default TTL. |
| `set_ttl(key, &value, ttl)` | Stores the value, expiring it after `ttl`. |
| `get_or_set(key, ttl, init)` | Returns the cached value, or computes, stores, and returns it. |
| `delete(key)` | Removes one entry. |
| `clear()` | Removes every entry. |

Configure the fallback TTL with `Cache::...default_ttl(ttl)`; a zero duration means "never
expire". Build the constructors `Cache::in_memory()` for tests or single-process apps, or
`Cache::new(store)` to supply a custom `CacheStore`.

## Rate limiting on Redis

The rate limiter can use the same Redis server, so limits are shared across processes rather
than per-instance. See [Rate Limiting](/docs/framework/techniques/rate-limiting).
