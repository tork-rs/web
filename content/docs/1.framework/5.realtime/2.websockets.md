# WebSockets

A WebSocket endpoint upgrades an HTTP request to a long-lived, bidirectional connection. Where Server-Sent Events push one way, a WebSocket lets both sides send messages until either closes. It fits chat, collaborative editing, live dashboards, and anything where the client talks back.

## A first socket

Declare an endpoint with `#[websocket]`. The handler takes a `WebSocket`, the upgrade handle, and returns `tork::Result<()>`. Call `accept` to complete the upgrade and get a `WebSocketConn`, then loop over incoming messages.

```rust
use tork::{websocket, WebSocket, WsMessage};

#[websocket("/ws")]
pub async fn echo(socket: WebSocket) -> tork::Result<()> {
    let mut socket = socket.accept().await?;
    while let Some(message) = socket.recv().await? {
        match message {
            WsMessage::Text(text) => socket.send_text(text).await?,
            WsMessage::Binary(bytes) => socket.send_binary(bytes).await?,
            WsMessage::Close(_) => break,
            _ => {} // ping frames are answered for you
        }
    }
    Ok(())
}
```

`recv` returns `Ok(None)` when the peer closes. It also returns `Ok(None)` when the server begins a graceful shutdown: Tork first sends the client a `1001 Going Away` close frame, so the loop ends cleanly instead of the connection dropping mid-flight. `send_text` and `send_binary` write messages back, and the wire framing is handled for you.

## Typed messages

For structured messages, exchange JSON. `receive_json` reads the next message and deserializes it; `send_json` serializes and sends.

```rust
use tork::{websocket, WebSocket};

#[websocket("/chat/{room_id}")]
pub async fn chat(socket: WebSocket, room_id: String, user: CurrentUser) -> tork::Result<()> {
    let mut socket = socket.accept().await?;
    while let Some(input) = socket.receive_json::<ChatIn>().await? {
        socket
            .send_json(&ChatOut { from: user.name.clone(), message: input.message })
            .await?;
    }
    Ok(())
}
```

`receive_json` returns `Ok(None)` when the connection ends, and a malformed JSON message is a `400` error.

## Validating messages

`receive_valid::<T>()` deserializes the next message and then runs its `#[field(...)]` validation, so each message is checked the way a request body is.

```rust
#[api_model]
pub struct ChatIn {
    #[field(min_length = 1, max_length = 500)]
    pub message: String,
}

while let Some(input) = socket.receive_valid::<ChatIn>().await? {
    // input is deserialized and validated
}
```

A message that fails to parse or fails a rule is a `422` error. There is also `receive_text`, which returns the next message as a plain `String`.

## Limits and timeouts

Set limits for the whole app with `App::websocket_config`, or per route with `#[websocket(...)]` attributes. A route value overrides the app default.

```rust
use std::time::Duration;
use tork::WebSocketConfig;

App::new().websocket_config(
    WebSocketConfig::new()
        .max_message_size_kb(64)
        .idle_timeout_secs(120)
        .handshake_timeout(Duration::from_secs(10))
        .max_connections_per_ip(20),
);

#[websocket("/chat/{room}", max_message_size = "64KB", idle_timeout = "60s")]
pub async fn chat(socket: WebSocket /* ... */) -> tork::Result<()> { /* ... */ }
```

In the route attribute, sizes accept suffixes like `"64KB"` and `"1MB"`, and durations accept `"500ms"`, `"60s"`, and `"2m"`.

- `max_message_size` and `max_frame_size` bound incoming data. Both default to 1 MiB, so a peer cannot make the server buffer a huge message. Raise them for routes that legitimately carry larger payloads.
- `idle_timeout` closes a connection that has been silent too long. It is off by default, so set it when you want to drop silent clients.
- `handshake_timeout` (default 10 seconds) bounds how long the upgrade may take, so a client that opens the upgrade and then stalls cannot hold a slot forever.
- `max_connections_per_ip` caps how many sockets one client IP may hold at once. Further connections from that IP are rejected with `429 Too Many Requests` before the upgrade. It is an app-level setting only.

## Lifecycle hooks

Observe connections opening and closing with app-level hooks. `on_ws_connect` fires once the socket opens; `on_ws_disconnect` fires when it closes, carrying how long it was open and the close code.

```rust
App::new()
    .on_ws_connect(|info| async move {
        tracing::info!("ws open {}", info.path());
    })
    .on_ws_disconnect(|info| async move {
        tracing::info!("ws closed after {:?} ({:?})", info.duration(), info.close_code());
    });
```

The connect info exposes `method()`, `path()`, and `request_id()`. The disconnect info adds `duration()` and `close_code()`.
