# Server-Sent Events

Server-Sent Events (SSE) is a one-way streaming protocol over plain HTTP. The server holds the response open and pushes messages as they happen. It fits live feeds, progress updates, notifications, and token-by-token model output. You return an `Sse<T>` built from a stream, and Tork handles the headers, the wire encoding, and the keep-alive.

## A first stream

Declare an endpoint with `#[sse]`. The handler returns `tork::Result<Sse<T>>`, where `T` is the type each event carries. `Sse::new` accepts any stream of `tork::Result<T>`.

```rust
use tork::{sse, Sse};

#[sse("/items/stream", event = "item_update", response_model = ItemOut)]
pub async fn stream_items(service: ItemService) -> tork::Result<Sse<ItemOut>> {
    let items = service.item_stream(); // any Stream<Item = tork::Result<ItemOut>>
    Ok(Sse::new(items))
}
```

The stream can come from anywhere: a channel, a database cursor, or a crate like `async-stream`. Tork does not pick one for you.

The response headers are set automatically:

```
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
X-Accel-Buffering: no
```

Each item is serialized to JSON and written as one event, named by `event = ...`:

```
event: item_update
data: {"id":1,"name":"Keyboard"}

event: item_update
data: {"id":2,"name":"Mouse"}
```

`#[sse]` handlers take extractors, dependencies, and path parameters exactly like a normal route.

## Per-event control

For control over an event's name, id, retry hint, or comment, stream `SseEvent<T>` values and use `Sse::events`.

```rust
use futures_util::StreamExt;
use tork::{sse, Sse, SseEvent};

#[sse("/items/stream")]
pub async fn stream_items(service: ItemService) -> tork::Result<Sse<ItemOut>> {
    let events = service.items().map(|item| {
        let item = item?;
        Ok(SseEvent::new(ItemOut::from(&item))
            .event("item_update") // event name
            .id(item.id)          // SSE event id (any value that is ToString)
            .retry_ms(5000))      // client reconnection delay in ms
    });
    Ok(Sse::events(events))
}
```

`SseEvent::new(value)` carries a typed payload serialized to JSON. `SseEvent::raw("[DONE]")` carries a pre-formatted string instead. An event has either typed data or raw data, never both. `.comment("...")` adds a comment line, which clients ignore.

Tork strips any carriage return or line feed from event names and ids (and NUL from an id) before writing them, so a user-controlled name or id cannot inject extra fields into the client's stream.

## Configuring the stream

`Sse` has builder methods for the response behavior. The defaults shown are the framework defaults.

```rust
use std::time::Duration;

Ok(Sse::new(items)
    .event("item_update")                // default event name for data items
    .heartbeat(Duration::from_secs(15))  // keep-alive comment interval (default 15s)
    .max_event_size(256 * 1024)          // drop events larger than this (default 256 KiB)
    .client_timeout(Duration::from_secs(60)) // end the stream after this long
    .done_event("[DONE]"))               // emit a final data event when the source ends
```

A heartbeat keeps idle connections open by sending a comment line every interval. Call `.no_heartbeat()` to turn it off. An event larger than `max_event_size` is dropped and logged rather than sent, so one event cannot exhaust memory; pass `usize::MAX` to remove the cap. `Cache-Control: no-cache` and `X-Accel-Buffering: no` are on by default so proxies do not buffer the stream, and the compression middleware skips `text/event-stream` responses.

## Limiting concurrent streams

Each open stream holds a pinned stream and its timers for its whole lifetime, which can be hours, so an unbounded number of them can exhaust memory. Cap the concurrent count on the app.

```rust
App::new().max_sse_connections(10_000);
```

Once the cap is reached, further SSE requests get `503 Service Unavailable` until an open stream ends and frees a slot. With no cap set, SSE streams are unbounded.

## Resuming with Last-Event-ID

When a browser reconnects after a drop, it sends the id of the last event it saw in the `Last-Event-ID` header. The `LastEventId` extractor reads it, so you can resume from where the client left off. For this to work, set an id on each event with `SseEvent::new(..).id(..)`.

```rust
use tork::{sse, Sse, LastEventId};

#[sse("/notifications")]
pub async fn notifications(
    last: LastEventId,
    service: NotificationService,
) -> tork::Result<Sse<NotificationOut>> {
    let after = last.as_str().and_then(|id| id.parse::<i64>().ok()).unwrap_or(0);
    Ok(Sse::new(service.after(after)))
}
```

When the id has a known type, `SseResume<T>` parses it for you. A missing or invalid value yields `None`.

```rust
use tork::{sse, Sse, SseResume};

#[sse("/notifications")]
pub async fn notifications(
    resume: SseResume<i64>,
    service: NotificationService,
) -> tork::Result<Sse<NotificationOut>> {
    let after = resume.last_id().copied().unwrap_or(0);
    Ok(Sse::new(service.after(after)))
}
```

## Streaming over POST

SSE is not limited to `GET`. Use `#[post_sse]` (or `#[sse(method = POST, ...)]`) to stream a response to a request that carries a body, for example streaming model tokens for a prompt.

```rust
use tork::{post_sse, Sse, Valid};

#[post_sse("/chat/stream", event = "token")]
pub async fn stream_chat(
    body: Valid<ChatPrompt>,
    service: ChatService,
) -> tork::Result<Sse<TokenOut>> {
    Ok(Sse::new(service.stream_tokens(body.into_inner())))
}
```

Validation and the other extractors work the same as on a normal route.
