# `#[derive(AppError)]` and `#[status(...)]`

Deriving `AppError` lets your own error type convert into a `tork::Error` with `?`, so a
handler can return it directly. The `#[status(...)]` attribute chooses which HTTP status
the error maps to. The original value is kept as the error source, so a typed
`exception_handler` can recover it later.

**Applies to:** a struct or enum that implements `Display` (the `Display` text becomes the
error message).

```rust
use tork::AppError;

#[derive(Debug, thiserror::Error, AppError)]
#[error("rate limited")]
#[status(429)]
pub struct RateLimited;

// fn handler() -> tork::Result<()> { Err(RateLimited)?; Ok(()) }
```

## `#[status(...)]`

The attribute is optional. Without it, the error maps to `500 Internal Server Error`. It
accepts one of two forms:

| Form | Example | Meaning |
| --- | --- | --- |
| An HTTP status code | `#[status(404)]` | Map to that status. |
| An `ErrorKind` variant name | `#[status(NotFound)]` | Map to that kind. |

The supported status codes are `400`, `401`, `403`, `404`, `405`, `409`, `413`, `422`,
`429`, `500`, `503`, and `504`. A code outside this set, or an unknown variant name, is a
compile error. The status is set on the type as a whole, not per enum variant.
