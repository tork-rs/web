# Validation

A model struct with `#[api_model]` gets validation through `garde`. When you read
a body with `Valid<T>`, every field rule is checked before the handler runs. A
violation returns `422` with a list of what went wrong.

## All field rules

Here is every rule you can put in `#[field(...)]`:

| Rule | Applies to | Example | What it does |
|---|---|---|---|
| `min_length = N` | `String` | `#[field(min_length = 1)]` | Minimum string length |
| `max_length = N` | `String` | `#[field(max_length = 255)]` | Maximum string length |
| `ge = E` | numbers | `#[field(ge = 0)]` | Value must be >= E |
| `le = E` | numbers | `#[field(le = 100)]` | Value must be <= E |
| `gt = E` | numbers | `#[field(gt = 0)]` | Value must be > E |
| `lt = E` | numbers | `#[field(lt = 100)]` | Value must be < E |
| `title = "..."` | any | `#[field(title = "User ID")]` | OpenAPI field title |
| `description = "..."` | any | `#[field(description = "The unique identifier")]` | OpenAPI field description |
| `custom = path` | any | `#[field(custom = validate_email)]` | Calls your own validator |
| `nested` | struct | `#[field(nested)]` | Recurse into nested model |
| `default` | any | `#[field(default)]` | Serde default when absent |

Rules work on `Option<T>` fields too. When the value is `None`, no rule runs.
When it is `Some(...)`, the rules are applied.

```rust
use tork::api_model;

#[api_model]
pub struct CreateOrder {
    #[field(min_length = 1, max_length = 120)]
    pub name: String,

    #[field(max_length = 300)]
    pub description: Option<String>,

    #[field(gt = 0)]
    pub price: f64,

    #[field(ge = 0)]
    pub tax: Option<f64>,

    #[field(lt = 1000)]
    pub quantity: u32,
}
```

## Using a model with validation

Pass the model as `Valid<T>` to a handler. The body is read, deserialized, and
validated before your code runs.

```rust
use tork::{post, Valid};

#[post("/orders")]
pub async fn create_order(body: Valid<CreateOrder>) -> tork::Result<OrderOut> {
    let input = body.into_inner();
    // input is validated, use it safely
    Ok(OrderOut { name: input.name, price: input.price })
}
```

## Validation error response

When validation fails, the response has status `422` and a body like this:

```json
{
  "status": 422,
  "code": "VALIDATION_ERROR",
  "title": "Unprocessable Entity",
  "message": "The submitted data failed validation.",
  "details": [
    {
      "field": "name",
      "issue": "TOO_SHORT",
      "message": "length is lower than 1"
    },
    {
      "field": "price",
      "issue": "TOO_SMALL",
      "message": "must be greater than 0"
    }
  ],
  "traceId": "req-abc123",
  "timestamp": "2026-06-14T10:00:00Z"
}
```

The issue codes are:

| Code | When |
|---|---|
| `TOO_SHORT` | string is below `min_length` |
| `TOO_LONG` | string is above `max_length` |
| `TOO_SMALL` | number is below `gt` or `ge` |
| `TOO_LARGE` | number is above `lt` or `le` |
| `INVALID_FORMAT` | email or pattern check failed |
| `INVALID` | any other validation error |

## Custom validators

When the built-in rules are not enough, write your own and reference it with
`#[field(custom = your_fn)]`.

```rust
#[api_model]
pub struct Signup {
    #[field(min_length = 3, custom = validate_username)]
    pub username: String,
}

fn validate_username(value: &str, _: &()) -> garde::Result {
    if value.contains("admin") {
        return Err(garde::Error::new("username cannot contain 'admin'"));
    }
    Ok(())
}
```

The function signature is `fn(&T, &()) -> garde::Result`. It is called after the
built-in rules pass.

```rust
fn validate_email(value: &str, _: &()) -> garde::Result {
    if !value.contains('@') {
        return Err(garde::Error::new("must be a valid email address"));
    }
    Ok(())
}
```

## Nested validation

When a model contains another model, mark the field with `#[field(nested)]`. The
inner model's rules are checked too.

```rust
#[api_model]
pub struct Address {
    #[field(min_length = 1, max_length = 200)]
    pub street: String,

    #[field(min_length = 5, max_length = 10)]
    pub zip: String,
}

#[api_model]
pub struct CreateUser {
    #[field(min_length = 1, max_length = 100)]
    pub name: String,

    #[field(nested)]
    pub address: Address,
}
```

Now a request with an invalid address is also rejected. Nested `Option<T>` fields
are skipped when absent.

## Default values

Mark a field `#[field(default)]` to use the type's default when the JSON key is
missing instead of rejecting the request.

```rust
#[api_model]
pub struct PaginatedQuery {
    #[field(default)]
    pub page: u32,          // defaults to 0
    #[field(default)]
    pub per_page: u32,      // defaults to 0
}
```

Without `default`, a missing key is a deserialization error. `Option<T>` fields
are always optional without needing `default`.

## Using Json directly

When you do not need validation, use `Json<T>` instead of `Valid<T>`. It
deserializes the body but skips all validation rules.

```rust
use tork::{post, Json};

#[post("/raw")]
pub async fn raw(body: Json<CreateOrder>) -> tork::Result<OrderOut> {
    let input = body.0;
    // input is deserialized but not validated
    Ok(OrderOut { name: input.name, price: input.price })
}
```

`Json<T>` is available for any `T: DeserializeOwned`. `Valid<T>` additionally
requires `T: Validate<Context = ()>`.

## 5. Models and validation

`#[api_model]` turns a struct into a request or response model. It derives
serialization, validation, and JSON Schema in one place, and translates concise
field constraints into the underlying rules.

### Declaring a model

```rust
use tork::api_model;

##[api_model(rename_all = "camelCase")]
pub struct CreateOrderInput {
    #[field(min_length = 1, max_length = 120)]
    pub name: String,

    #[field(max_length = 300, title = "The description of the item")]
    pub description: Option<String>,

    #[field(gt = 0, description = "The price must be greater than zero")]
    pub price: f64,

    #[field(ge = 0)]
    pub tax: Option<f64>,
}
```

The macro derives `Debug`, `Clone`, serialization, validation, and JSON Schema.
`rename_all = "camelCase"` applies to the serialized field names and the schema.

### Field constraints

`#[field(...)]` accepts:

- `min_length` / `max_length`: string length bounds.
- `ge` / `le`: inclusive numeric bounds.
- `gt` / `lt`: exclusive numeric bounds.
- `title` / `description`: documentation metadata.
- `custom = path`: a custom validator function (may be repeated).
- `nested`: validate a nested model or collection recursively.
- `default`: use the type's `Default` when the field is absent.

A field with no constraints is allowed; it simply has no rules. `Option<T>` fields
are optional during deserialization (an absent field becomes `None`).

### Validating a request body

Receive a model with the `Valid<T>` extractor. It parses the JSON body and runs
validation before the handler sees it:

```rust
use tork::{Valid, post};

use crate::models::order::{CreateOrderInput, OrderOut};

##[post("/", response_model = OrderOut, status_code = 201, summary = "Create an order")]
pub async fn create_order(
    user_id: i64,
    body: Valid<CreateOrderInput>,
    service: OrderService,
) -> tork::Result<OrderOut> {
    service.create_order(user_id, body.into_inner()).await
}
```

A body that fails validation produces `422 Unprocessable Entity` with one entry
per offending field. For example, posting `{"name":"","price":0}` returns:

```json
{
  "status": 422,
  "code": "VALIDATION_ERROR",
  "title": "Unprocessable Entity",
  "message": "The submitted data failed validation.",
  "details": [
    { "field": "name", "issue": "TOO_SHORT", "message": "length is lower than 1" },
    { "field": "price", "issue": "TOO_SMALL", "message": "must be greater than 0" }
  ],
  "traceId": "req-...",
  "timestamp": "2026-06-10T11:08:07Z"
}
```

### Custom validators

For a rule that the built-in constraints do not cover, write a function and point
`custom` at it. The function performs the check and supplies its own message:

```rust
use tork::api_model;

fn not_blank(value: &str, _ctx: &()) -> garde::Result {
    if value.trim().is_empty() {
        Err(garde::Error::new("must not be blank"))
    } else {
        Ok(())
    }
}

##[api_model]
pub struct CreateTag {
    #[field(min_length = 1, custom = not_blank)]
    pub label: String,
}
```

### Nested models

A field that is itself a model is serialized and documented automatically. To also
validate it recursively, mark it `nested`:

```rust
use tork::api_model;

##[api_model]
pub struct Image {
    #[field(min_length = 1)]
    pub url: String,
    pub name: String,
}

##[api_model(rename_all = "camelCase")]
pub struct Item {
    #[field(min_length = 1)]
    pub name: String,

    pub price: f64,

    #[field(default)]
    pub tags: std::collections::HashSet<String>,

    #[field(nested)]
    pub image: Option<Image>,
}
```

Validating an `Item` now also validates the nested `Image`. The `tags` field uses
`default`, so an absent `tags` becomes an empty set.
