# Validation

A struct with `#[api_model]` derives three things at once: serde serialization, `garde` validation, and a JSON Schema. When you read a body with `Valid<T>`, every field rule runs before the handler. A violation returns `422 Unprocessable Entity` with one entry per offending field.

```rust
use tork::api_model;

#[api_model]
pub struct CreateOrder {
    #[field(min_length = 1, max_length = 120)]
    pub name: String,

    #[field(max_length = 300)]
    pub description: Option<String>,

    #[field(gt = 0)]
    pub price: f64,

    #[field(ge = 0)]
    pub tax: Option<f64>,
}
```

Rules apply to `Option<T>` fields too. When the value is `None`, no rule runs. When it is `Some(...)`, the rules apply.

## Field rules

`#[field(...)]` accepts exactly these keys. Anything else is a compile error.

| Rule | Applies to | What it does |
| --- | --- | --- |
| `min_length = N` | strings | minimum length |
| `max_length = N` | strings | maximum length |
| `ge = N` | numbers | value must be `>= N` |
| `le = N` | numbers | value must be `<= N` |
| `gt = N` | numbers | value must be `> N` |
| `lt = N` | numbers | value must be `< N` |
| `custom = fn` | any | run your own validator (may be repeated) |
| `nested` | nested model | validate the inner model too |
| `title = "..."` | any | sets the field's OpenAPI schema title |
| `description = "..."` | any | sets the field's OpenAPI schema description |
| `default` | any | use the type's `Default` when the key is absent |

`min_length` and `max_length` map to a length check, `ge`/`le`/`gt`/`lt` to a numeric range check. `title`, `description`, and `default` add no validation: the first two are schema metadata and `default` is a serde default. A field with no `#[field(...)]` simply has no rules.

## Reading a validated body

Take the model as `Valid<T>`. The body is read, deserialized, and validated before your code runs, so the value you unwrap is already valid.

```rust
use tork::{post, Valid};

#[post("/orders")]
pub async fn create_order(body: Valid<CreateOrder>) -> tork::Result<OrderOut> {
    let order = body.into_inner();
    // order is validated, use it safely
    Ok(OrderOut { name: order.name, price: order.price })
}
```

`Valid<T>` works for any `T` that is an `#[api_model]` (it needs `serde::DeserializeOwned` and `garde::Validate`). Unwrap it with `into_inner()` or the tuple field `.0`.

## The error response

When validation fails, the status is `422` and the body lists every offending field:

```json
{
  "status": 422,
  "code": "VALIDATION_ERROR",
  "title": "Unprocessable Entity",
  "message": "The submitted data failed validation.",
  "details": [
    { "field": "name", "issue": "TOO_SHORT", "message": "length is lower than 1" },
    { "field": "price", "issue": "TOO_SMALL", "message": "must be greater than 0" }
  ],
  "traceId": "req-7f3c...",
  "timestamp": "2026-06-19T10:00:00Z"
}
```

`traceId` is a fresh id minted for the response (prefixed `req-`), and `timestamp` is the UTC time it was built. Both are always present. `details` is omitted when empty.

Each entry carries a machine-readable `issue` code derived from the failed rule:

| Issue | Comes from |
| --- | --- |
| `TOO_SHORT` | `min_length` |
| `TOO_LONG` | `max_length` |
| `TOO_SMALL` | `ge` or `gt` |
| `TOO_LARGE` | `le` or `lt` |
| `INVALID_FORMAT` | a custom validator whose message mentions an email |
| `INVALID` | any other failure, including most custom validators |

A body that is not valid JSON, or that is missing a required (non-`Option`, non-`default`) field, also returns `422`, but with `code` `UNPROCESSABLE_ENTITY` and the message `request body is not valid JSON`. It is a parse failure, not a field-rule failure, so it carries no `details`.
