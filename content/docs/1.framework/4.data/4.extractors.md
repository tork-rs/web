# Extractors

Every handler parameter that is not a path parameter is resolved through `FromRequest`. Tork ships extractors for the common request data, and you write your own for anything specific to your app.

## How extraction works

When a request matches a route, each handler parameter is resolved in order through `FromRequest`. If any resolution fails, the request is rejected with that error's status code, and the handler never runs.

```rust
pub trait FromRequest: Sized + Send {
    fn from_request(ctx: &RequestContext) -> impl Future<Output = Result<Self>> + Send;
}
```

Every extractor receives a `&RequestContext`, which exposes the method, URI, headers, peer address, path parameters, registered state, and the body. At most one extractor may read the body.

## Path parameters

Path parameters are matched by name. The handler parameter name matches the `{placeholder}` in the route path.

```rust
#[get("/users/{user_id}/orders/{order_id}")]
pub async fn get_order(user_id: i64, order_id: String) -> tork::Result<OrderOut> {
    // user_id and order_id come from the URL
}
```

Any type that implements `FromStr` works: `i64`, `u32`, `String`, a UUID, and so on. A value that fails to parse returns `422 Unprocessable Entity`.

## Body extractors

Four extractors read the request body, each for one content type. A handler may use only one of them.

| Extractor | Body | Covered in |
| --- | --- | --- |
| `Json<T>` | `application/json`, no validation | [Validation](/docs/framework/data/validation) |
| `Valid<T>` | `application/json`, with `#[field]` rules | [Validation](/docs/framework/data/validation) |
| `Form<T>` | `application/x-www-form-urlencoded` | [File Uploads](/docs/framework/data/file-uploads) |
| `Multipart<T>` | `multipart/form-data`, with files | [File Uploads](/docs/framework/data/file-uploads) |

```rust
use tork::{post, Valid};

#[post("/orders")]
pub async fn create_order(body: Valid<CreateOrder>) -> tork::Result<OrderOut> {
    let order = body.into_inner();
    // order is deserialized and validated
}
```

`Json<T>` and `Valid<T>` fail with `422` on a body that is not valid JSON, and `Valid<T>` also fails `422` with field details on a broken rule. `Multipart<T>` adds `413` for an oversized file. The body chapters cover each in full.

## Bearer token

`BearerToken` reads the `Authorization: Bearer <token>` header. Call `token()` for the part after `Bearer `.

```rust
use tork::{get, BearerToken};

#[get("/secure")]
pub async fn secure(token: BearerToken) -> tork::Result<&'static str> {
    let _value: &str = token.token();
    Ok("secure data")
}
```

It fails with `401 Unauthorized` when the header is missing, not valid UTF-8, or lacks the `Bearer ` prefix.

## Application state

Register a value on the app with `App::state(...)`, then read it in a handler. There are two extractors for this.

`State<T>` clones a registered value of type `T`. The value must be `Clone`, and it is cloned per request, so keep it cheap (hold pools and handles behind `Arc`). Access the inner value through `.0`.

```rust
use tork::{get, State};

#[get("/info")]
pub async fn info(state: State<AppState>) -> tork::Result<InfoOut> {
    let app_state = state.0;
    Ok(InfoOut { name: app_state.name })
}

// Registered when building the app:
// App::new().state(AppState { /* ... */ })
```

`Arc<T>` resolves a value registered as `Arc<T>`, sharing it by pointer instead of cloning the inner value. This is the usual way to inject a database pool or another shared resource.

```rust
use std::sync::Arc;
use tork::get;

#[get("/count")]
pub async fn count(db: Arc<Database>) -> tork::Result<i64> {
    // db is the Arc<Database> registered on the app
}

// Registered with App::new().state(Arc::new(database)) or a #[resource] field.
```

Both fail with `500` when no value of that type is registered.

## Injectable framework handles

Several framework features are injected the same way, by type. Each has its own guide:

| Extractor | Provides | Guide |
| --- | --- | --- |
| `Redis` | a shared Redis handle | [Redis](/docs/framework/techniques/redis) |
| `Cache` | the key/value cache | [Caching](/docs/framework/performance/caching) |

## SSE resume

Two extractors read the `Last-Event-ID` header for Server-Sent Events resumption. Neither fails: when the header is absent, both yield `None`.

```rust
use tork::{get, LastEventId, Sse, SseResume};

#[get("/events")]
pub async fn events(last: LastEventId) -> tork::Result<Sse<EventOut>> {
    let _id: Option<&str> = last.as_str();
    // ...
}

#[get("/events-typed")]
pub async fn events_typed(resume: SseResume<i64>) -> tork::Result<Sse<EventOut>> {
    let _id: Option<&i64> = resume.last_id(); // the id parsed into i64
    // ...
}
```

`SseResume<T>` parses the header into any `T: FromStr`. See [Server-Sent Events](/docs/framework/realtime/server-sent-events).

## The request context

`RequestContext` is the value every extractor receives. It is not a handler parameter itself, you reach it by writing an extractor. It exposes:

- `method()`, `uri()`, `headers()` for the request line and headers
- `peer_addr()` for the client's socket address, when known
- `path_param(name)` and `path_params()` for path parameters
- `state()` and `resource::<T>()` for the registered state and a single resource by type
- `take_body()` to consume the request body, once

## Writing a custom extractor

Implement `FromRequest` for your own type to turn request data into something a handler can ask for by type. This is how you build auth guards, session types, or a typed view of a header. Pull what you need out of the borrowed `&RequestContext` before the async block, since the context is only borrowed for the call.

::code-group

```rust [src/auth.rs]
use std::future::Future;

use tork::{Error, FromRequest, RequestContext};

// Reads and checks an API key from the `x-api-key` header.
pub struct ApiKey(pub String);

impl FromRequest for ApiKey {
    fn from_request(ctx: &RequestContext) -> impl Future<Output = tork::Result<Self>> + Send {
        let key = ctx
            .headers()
            .get("x-api-key")
            .and_then(|value| value.to_str().ok())
            .map(|value| value.to_owned());

        async move {
            match key {
                Some(key) if key == "secret" => Ok(ApiKey(key)),
                Some(_) => Err(Error::unauthorized("invalid api key")),
                None => Err(Error::unauthorized("missing api key")),
            }
        }
    }
}
```

```rust [src/routers.rs]
use tork::{api_router, get};

use crate::auth::ApiKey;

#[api_router(tags = ["secure"])]
pub mod secure_router {
    use super::*;

    #[get("/secure")]
    pub async fn secure(key: ApiKey) -> tork::Result<String> {
        Ok(format!("welcome, {}", key.0))
    }
}

pub use secure_router::router;
```

```rust [src/lib.rs]
pub mod auth;
pub mod routers;
```

```rust [tests/extractors.rs]
use tork::testing::TestClient;
use tork::App;

use myapp::routers;

async fn client() -> tork::Result<TestClient> {
    let app = App::new().include_router(routers::router());
    Ok(TestClient::new(app.build_test().await?).await?)
}

#[tokio::test]
async fn requires_a_valid_api_key() -> tork::Result<()> {
    let client = client().await?;

    // No header: rejected before the handler runs.
    let missing = client.get("/secure").send().await?;
    assert_eq!(missing.status(), 401);

    // Valid key: the handler runs.
    let ok = client.get("/secure").header("x-api-key", "secret").send().await?;
    assert_eq!(ok.status(), 200);

    Ok(())
}
```

::

Because `ApiKey` implements `FromRequest`, any handler can take it by type, and a failed extraction rejects the request with the status from the error you return.

## Dependencies

A dependency is a type that builds itself from the request. It is the same `FromRequest` contract with less boilerplate, generated by one of two macros:

- `#[derive(Inject)]` resolves each field of a struct by type (a resource, state, or another dependency).
- `#[tork::dependency]` runs a custom async `resolve(...)` whose parameters are themselves resolved first.

```rust
use std::sync::Arc;
use tork::{BearerToken, Inject};

// Resolve each field by type.
#[derive(Inject)]
pub struct UserRepository {
    db: Arc<Database>,
}

// Or run custom logic, taking other extractors and dependencies as parameters.
#[derive(Clone)]
pub struct CurrentUser {
    pub id: i64,
}

#[tork::dependency]
impl CurrentUser {
    pub async fn resolve(token: BearerToken, users: UserRepository) -> tork::Result<Self> {
        let id = users.authenticate(token.token())?;
        Ok(CurrentUser { id })
    }
}
```

A handler then takes `UserRepository` or `CurrentUser` by type. See [Dependencies](/docs/framework/getting-started/dependencies).

## Summary of built-in extractors

| Extractor | Reads from | Fails with |
| --- | --- | --- |
| Path params (`T: FromStr`) | URL path | `422` |
| `Json<T>` | body, JSON | `422` |
| `Valid<T>` | body, JSON with validation | `422` with details |
| `Form<T>` | body, urlencoded | `422` |
| `Multipart<T>` | body, multipart | `413` / `422` |
| `BearerToken` | `Authorization` header | `401` |
| `State<T>` | registered state (cloned) | `500` |
| `Arc<T>` | registered resources | `500` |
| `LastEventId` | `Last-Event-ID` header | never |
| `SseResume<T>` | `Last-Event-ID` header | never |

`Redis`, `Cache`, and `Logger` are also injectable by type when their feature is configured.
