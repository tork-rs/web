# Serialization

A route handler returns `tork::Result<T>`. On success, Tork serializes `T` to JSON and sends it with `Content-Type: application/json`. The value's type just needs to be serializable, which every `#[api_model]` is.

```rust
use tork::{get, api_model};

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}

#[get("/me")]
pub async fn me() -> tork::Result<UserOut> {
    Ok(UserOut { id: 1, name: "Alice".into() })
}
```

The response is `200` with body `{"id":1,"name":"Alice"}`.

Handlers declared with the route macros (`#[get]`, `#[post]`, and so on) always serialize their return value to JSON. To send something else, such as raw bytes or HTML, you build a response by hand in middleware or an exception handler, covered at the end of this page.

## Status code

The default success status is `200`. Set another with `status_code` on the route attribute. It takes an integer like `201`, or a `StatusCode` value.

```rust
use tork::post;

#[post("/users", status_code = 201)]
pub async fn create() -> tork::Result<UserOut> {
    Ok(UserOut { id: 1, name: "Alice".into() })
}
```

## Response model

When the value your handler returns differs from the shape you want on the wire, declare a `response_model`. Tork converts the return value into that model with `From`, then serializes the model. So the return type must implement `Into<ResponseModel>`.

```rust
use tork::{post, api_model, Valid};

#[api_model]
pub struct CreateUser {
    pub name: String,
}

#[api_model]
pub struct UserOut {
    pub id: i64,
    pub name: String,
}

impl From<CreateUser> for UserOut {
    fn from(input: CreateUser) -> Self {
        UserOut { id: 0, name: input.name }
    }
}

#[post("/users", response_model = UserOut, status_code = 201)]
pub async fn create(body: Valid<CreateUser>) -> tork::Result<CreateUser> {
    Ok(body.into_inner())
}
```

The `From` impl runs before serialization, and `UserOut` is the schema shown in OpenAPI. A `response_model` can be a generic type, such as `response_model = Vec<UserOut>` for a handler that returns `Vec<UserOut>`.

## Dynamic JSON

For a response with no fixed model, return a `serde_json::Value`. It serializes like any other value.

```rust
use serde_json::json;
use tork::get;

#[get("/health")]
pub async fn health() -> tork::Result<serde_json::Value> {
    Ok(json!({ "status": "ok" }))
}
```

## OpenAPI schema

The response model (or the return type, when no `response_model` is set) produces the JSON Schema in the OpenAPI document. Each model field becomes a schema property, and `title` and `description` from `#[field(...)]` carry into the schema.

```rust
#[api_model]
pub struct UserOut {
    pub id: i64,

    #[field(title = "Display name", description = "The name shown in the UI")]
    pub display_name: String,
}
```

Nested `#[api_model]` types are registered under `components.schemas` and referenced by `$ref`. See [OpenAPI](/docs/framework/fundamentals/openapi) for the full document.

## Building a response by hand

Route handlers always serialize to JSON. When you need full control over the body and its content type, you build a `Response` directly. That happens in two places: middleware, which returns `tork::Result<Response>`, and exception handlers, which return `Response`. Both rest on the `IntoResponse` trait and the response constructors.

### IntoResponse

`IntoResponse` turns a value into a `Response`. These types implement it:

| Type | Produces |
| --- | --- |
| `Response` | returned unchanged |
| `StatusCode` | empty body with that status (no content type) |
| `()` | empty body with `200` (no content type) |
| `Result<T, E>` (`T: IntoResponse`, `E: Into<Error>`) | forwards `Ok`, converts `Err` |
| `Json<T>` (`T: Serialize`) | `T` as JSON, `200`, `application/json` |
| `Error` | the standard error JSON body |
| `Sse<T>` | a `text/event-stream` streaming body |

### Constructors

```rust
use bytes::Bytes;
use tork::{bytes_response, json_response, Json, StatusCode};

// Serialize a value to JSON with a chosen status.
json_response(StatusCode::CREATED, &user);

// Wrap a serializable value as a 200 application/json response.
Json(user).into_response();

// Raw bytes with an explicit content type (a 'static string).
bytes_response(StatusCode::OK, "text/plain; charset=utf-8", Bytes::from("hello"));
```

Import these from the crate root (`tork::json_response`, `tork::bytes_response`), not from a `tork::response` path.

### A custom type

Implement `IntoResponse` to give your own type a response shape, then call `.into_response()` where a `Response` is expected.

```rust
use bytes::Bytes;
use tork::{bytes_response, IntoResponse, Response, StatusCode};

pub struct HtmlPage(pub String);

impl IntoResponse for HtmlPage {
    fn into_response(self) -> Response {
        bytes_response(StatusCode::OK, "text/html; charset=utf-8", Bytes::from(self.0))
    }
}
```

A middleware can return it instead of calling the next handler:

```rust
use tork::{middleware, Next, Request, Response};

#[middleware]
pub async fn maintenance(req: Request, next: Next) -> tork::Result<Response> {
    if req.uri().path() == "/down" {
        return Ok(HtmlPage("<h1>Down for maintenance</h1>".into()).into_response());
    }
    next.run(req).await
}
```

See [Middleware](/docs/framework/getting-started/middleware) for wiring middleware, and [Error Handling](/docs/framework/getting-started/error-handling-and-exceptions) for exception handlers.

## Response body types

A response body is one of two kinds. Buffered (`RespBody::new(bytes)`) holds the whole body in memory, which is what JSON responses use. Streaming (`RespBody::stream(body)`) sends the body frame by frame, which is what Server-Sent Events use. You only touch `RespBody` directly when writing middleware that replaces a body.
