# Migrations

Every schema change is a migration with an `up()` and a `down()`. State lives in
the `_tork_migrations` table, so `status` always reflects the true version and a
rollback is never a guess.

## The lifecycle

```bash
tork-orm migrate init
# ✓ initialized migrations/
# ✓ created _tork_migrations

tork-orm migrate create add_users
# ✓ wrote 0001_add_users.rs  (up + down stubs)

tork-orm migrate up
# → 0001_add_users  applying…
# ✓ 0001_add_users  migrated (12ms)

tork-orm migrate status
# 0001_add_users  ● applied

tork-orm migrate down
# ← 0001_add_users  reverting…
# ✓ rolled back to base
```

## Reversible by design

A generated migration carries both directions:

```rust
fn up(s: &mut Schema) {
    s.create_table::<User>();
}

fn down(s: &mut Schema) {
    s.drop_table::<User>();
}
```

Because the model is the source of truth, regenerating after a struct change
produces the exact diff — add a column, and the migration adds exactly that
column, with a `down()` that drops it.
