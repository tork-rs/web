# Migrations

Every schema change is a single `.sql` file that holds both the forward change (`-- migrate:up`) and the rollback (`-- migrate:down`). The `tork` CLI, or the standalone `tork-orm` binary, manages these files. Applied migrations are recorded in `_tork_migrations`, so `status` reflects the real database state and every rollback is deterministic.

## Installing the CLI

If you created your project with `tork new`, you already have the `tork` binary:

```bash
cargo install tork-cli
```

This gives you `tork migrate` alongside `tork new`, `tork dev`, `tork build`, and `tork check`. To install only the ORM's migration tool:

```bash
cargo install tork-orm-cli   # provides the `tork-orm` binary
```

Both expose the same `migrate` subcommands, so `tork migrate` and `tork-orm migrate` are interchangeable below.

## Project configuration

If you used `tork new`, the `Cargo.toml` already has the configuration:

```toml
[package.metadata.tork]
dialect = "sqlite"

[package.metadata.tork.migrations]
dir                  = "migrations"
file_template        = "{rev}_{slug}"
revision_style       = "sequence"
truncate_slug_length = 40
```

For a project you set up by hand, add these sections. Then point the CLI at your database with a `DATABASE_URL` variable, in `.env` or the environment:

```
DATABASE_URL=sqlite://app.db
```

or pass it inline:

```bash
tork migrate up --database-url sqlite://app.db
```

If `DATABASE_URL` is unset, the CLI also reads `DB_URL` as a fallback.

## Migration file format

Each migration is one `.sql` file holding both sections. When you run `tork migrate create add_users`, the CLI writes `migrations/0001_add_users.sql`:

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
-- Write the schema changes here, for example:
-- CREATE TABLE "example" ("id" INTEGER PRIMARY KEY AUTOINCREMENT);

-- migrate:down
-- Undo the changes, for example:
-- DROP TABLE "example";
```

Two headers define the chain:

| Header | Purpose |
| --- | --- |
| `-- revision:` | The unique id for this migration. Its format follows `revision_style`: a zero-padded number with `sequence`, or a 12-character hex string with `hash`. |
| `-- down_revision:` | The `revision` of the previous migration, empty on the first. This is how the CLI orders migrations and knows which one to revert next. |

`create` also writes a `-- name:` line for readability, but the name shown in `status` comes from the file name (the part after the revision). The section markers `-- migrate:up` and `-- migrate:down` split the forward SQL from the rollback SQL; everything under a marker runs for that direction.

## Workflow

### init

```bash
tork migrate init
```

Creates the migrations directory. It does not touch the database; the `_tork_migrations` table is created automatically the first time you run `up`, `down`, or `status`.

### create

```bash
tork migrate create add_users
```

Writes one file, `migrations/0001_add_users.sql`. With `revision_style = "sequence"` the number is zero-padded and increments automatically. Open the file and replace the placeholder comments with real SQL.

### Write the SQL

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
CREATE TABLE users (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    username  TEXT    NOT NULL UNIQUE,
    email     TEXT    NOT NULL UNIQUE,
    is_active INTEGER NOT NULL DEFAULT 1
);

-- migrate:down
DROP TABLE users;
```

Always fill in the down section. It is how `tork migrate down` and `tork migrate redo` undo the change.

### up

```bash
tork migrate up
```

Applies every pending migration in revision order. To stop at a specific revision, pass it as an argument:

```bash
tork migrate up 0003   # apply up to and including revision 0003
```

Bare `up` and `up head` both apply everything pending.

### status

```bash
tork migrate status
```

Lists each migration as applied or pending.

### down

```bash
tork migrate down        # revert the most recent migration
tork migrate down 2      # revert the last 2 migrations
tork migrate down base   # revert everything
```

The step count and `base` are positional arguments, not flags. You can also revert down to a specific revision by passing it, like `tork migrate down 0001`. When more than one migration would be reverted, the CLI asks for confirmation unless you pass `-y`.

### redo

```bash
tork migrate redo
```

Reverts the most recent migration and re-applies it. Useful while you are editing a migration and want to test the full round-trip.

## A real migration sequence

### First migration: users table

```bash
tork migrate create add_users
```

`migrations/0001_add_users.sql`:

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
CREATE TABLE users (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    username  TEXT    NOT NULL UNIQUE,
    email     TEXT    NOT NULL UNIQUE,
    is_active INTEGER NOT NULL DEFAULT 1
);

-- migrate:down
DROP TABLE users;
```

```bash
tork migrate up
```

### Second migration: posts table

```bash
tork migrate create add_posts
```

`migrations/0002_add_posts.sql`:

```sql
-- revision: 0002
-- down_revision: 0001
-- name: add_posts

-- migrate:up
CREATE TABLE posts (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title      TEXT    NOT NULL,
    body       TEXT,
    published  INTEGER NOT NULL DEFAULT 0,
    created_at TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX posts_user_id ON posts (user_id);

-- migrate:down
DROP INDEX IF EXISTS posts_user_id;
DROP TABLE posts;
```

`-- down_revision: 0001` was filled in automatically by `create`. It points at the current head, so the CLI knows `0002` follows `0001`. You never set it by hand.

### Third migration: add a column

```bash
tork migrate create add_users_bio
```

`migrations/0003_add_users_bio.sql`:

```sql
-- revision: 0003
-- down_revision: 0002
-- name: add_users_bio

-- migrate:up
ALTER TABLE users ADD COLUMN bio TEXT;

-- migrate:down
ALTER TABLE users DROP COLUMN bio;
```

SQLite supports `DROP COLUMN` from version 3.35 (2021). On older versions, the down migration has to recreate the table without the column.

## The revision chain

`-- down_revision` links each migration to its predecessor, forming a single linear chain:

```
0001_add_users.sql          down_revision: (empty)
0002_add_posts.sql          down_revision: 0001
0003_add_users_bio.sql      down_revision: 0002
```

`up` walks the chain forward from the lowest unapplied revision. `down` follows `down_revision` backwards from the head. The chain must stay linear: exactly one migration has an empty `down_revision` (the base), and no two migrations may share the same `down_revision`. If two ever do, the CLI reports the conflict, and you fix it by re-pointing one migration's `down_revision`.

**File names are cosmetic.** Identity and order come from the headers, not from sorting file names, so you can rename a file freely, for example to add a date prefix. This is the Alembic-style approach, where the headers, not the file system, define history.

## File name templates

The `file_template` key controls the file name:

```toml
[package.metadata.tork.migrations]
file_template = "{rev}_{slug}"                          # 0001_add_users.sql
file_template = "{year}{month}{day}_{rev}_{slug}"       # 20260614_0001_add_users.sql
file_template = "{year}{month}{day}_{hour}{minute}_{rev}_{slug}"
```

Available tokens: `{rev}`, `{slug}`, `{year}`, `{month}`, `{day}`, `{hour}`, `{minute}`, `{second}`.

`revision_style` controls the format of `{rev}`:

| Style | Example revision | Example file name |
| --- | --- | --- |
| `sequence` | `0001` | `0001_add_users.sql` |
| `hash` | `a1b2c3d4e5f6` | `a1b2c3d4e5f6_add_users.sql` |

Projects from `tork new` use `sequence`. If you omit `revision_style`, the built-in default is `hash`. Sequence is easy to read and sort; hash avoids collisions when two people add migrations on separate branches.

## The _tork_migrations table

Applied migrations are recorded with these columns:

| Column | Description |
| --- | --- |
| `id` | Auto-increment primary key |
| `revision` | Matches the `-- revision:` header (unique) |
| `down_revision` | The predecessor's revision |
| `name` | The migration name |
| `checksum` | A hash of the up section |
| `batch` | Which `up` run applied it |
| `applied_at` | RFC 3339 timestamp |
| `execution_time_ms` | How long it took to apply |

The checksum is an FNV-1a hash of the `-- migrate:up` section, verified on every `up`. If a migration file changed after it was applied, the CLI stops with a mismatch error so the database cannot silently drift from the files. Pass `--allow-checksum-mismatch` to proceed anyway.

## Per-migration transactions

Each migration runs in its own transaction on a single pinned connection. If the up SQL fails partway, the partial work is rolled back and the migration is not recorded, so the database stays consistent.

Some statements cannot run inside a transaction, such as `CREATE INDEX CONCURRENTLY` in PostgreSQL. Disable the transaction for one file by adding `transaction:false` to its `-- migrate:up` marker:

```sql
-- revision: 0005
-- down_revision: 0004
-- name: add_concurrent_index

-- migrate:up transaction:false
CREATE INDEX CONCURRENTLY users_email ON users (email);

-- migrate:down
DROP INDEX IF EXISTS users_email;
```

Use this only when the database requires it. The transactional default is safer.

## Generating migrations from models

`tork-orm` can also write a migration by diffing your `#[derive(Model)]` structs against the live database. Because the diff needs your Rust model types, this is a feature you call from your own application code (typically a small generator binary), not a `tork migrate` subcommand. It detects new tables, added and removed columns, and added and removed indexes, and writes a `.sql` file chained onto the current head. In-place column type or nullability changes are written as `-- NOTE:` comments for you to apply by hand, since not every database can alter a column in place. Review the generated file, then run `tork migrate up`.

## Running migrations programmatically

Apply migrations from application code, which is handy for tests and for running them at startup, with `FileMigrator`:

```rust
use std::sync::Arc;

use tork::{LifespanContext, Resources, Result};
use tork_orm::migration::FileMigrator;
use tork_orm::prelude::*;

#[derive(Clone, Resources)]
pub struct Db {
    #[resource]
    pub database: Arc<Database>,
}

#[tork::lifespan]
impl Db {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        let url = std::env::var("DATABASE_URL").unwrap_or_else(|_| "sqlite://app.db".into());
        let database = Database::connect(&url, 10).await?;

        // Apply all pending migrations before accepting traffic.
        FileMigrator::new(database.clone(), "migrations").up().await?;

        Ok(Db { database: Arc::new(database) })
    }

    async fn shutdown(self) -> Result<()> {
        self.database.close().await;
        Ok(())
    }
}
```

`FileMigrator::new(db, dir).up()` is the same operation as `tork migrate up`: it applies every pending migration and returns the list it ran. `FileMigrator` also offers `up_to`, `down`, `down_all`, `down_to`, `redo`, and `status`, plus `table`, `allow_destructive`, and `on_checksum_mismatch` builders.

## Destructive SQL

The CLI refuses to run a migration whose up section contains `DROP TABLE` or `DROP COLUMN` unless you pass `--allow-destructive`:

```bash
tork migrate up --allow-destructive
```

This guards against an up migration that drops schema running by accident. The check looks only at the up section, so the down section, which usually drops things, is never blocked. It matches `DROP TABLE` and `DROP COLUMN` only, so it is a safety net rather than a guarantee: a `TRUNCATE` or a hand-written `DELETE` is not caught.

## Flags and environment

| Flag or variable | Purpose |
| --- | --- |
| `DATABASE_URL` (env) | Database connection string (`DB_URL` is a fallback) |
| `--database-url <url>`, `-d` | The database URL |
| `--dir <path>` | Migrations directory (default `migrations`) |
| `--table <name>` | Tracking table name (default `_tork_migrations`) |
| `--allow-destructive` | Permit `DROP TABLE` or `DROP COLUMN` in an up section |
| `--allow-checksum-mismatch` | Proceed when a file's checksum no longer matches |
| `-y`, `--yes` | Skip confirmation prompts (for `down` and `redo`) |
| `--no-color` | Disable colored output |

Connection string formats:

```
sqlite://app.db
sqlite::memory:
postgres://user:pass@localhost:5432/mydb
mysql://user:pass@localhost:3306/mydb
```

## Quick reference

| Command | What it does |
| --- | --- |
| `tork migrate init` | Create the migrations directory |
| `tork migrate create <name>` | Scaffold a new migration file |
| `tork migrate up` | Apply all pending migrations |
| `tork migrate up <rev>` | Apply up to a revision |
| `tork migrate status` | Show applied and pending migrations |
| `tork migrate down` | Revert the most recent migration |
| `tork migrate down <n>` | Revert the last N migrations |
| `tork migrate down base` | Revert every migration |
| `tork migrate redo` | Revert and re-apply the most recent migration |
