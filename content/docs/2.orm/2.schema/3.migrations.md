# Migrations

Every schema change is a single `.sql` file that holds both the forward change (`-- migrate:up`) and the rollback (`-- migrate:down`). The `tork` CLI (or the standalone `tork-orm` binary) manages these files. Applied migrations are recorded in `_tork_migrations` so `status` always reflects the real database state and every rollback is deterministic.

## Installing the CLI

If you created your project with `tork new`, you already have the `tork` binary. Install it with:

```bash
cargo install tork-cli
```

This gives you `tork migrate` alongside `tork new`, `tork dev`, `tork build`, and `tork check`.

Alternatively, install only the ORM binary:

```bash
cargo install tork-orm-cli
```

Both binaries expose the same `migrate` subcommands. In the examples below, `tork migrate` and `tork-orm migrate` are interchangeable.

## Project configuration

If you used `tork new --db sqlite`, the `Cargo.toml` already contains:

```toml
[package.metadata.tork]
dialect = "sqlite"

[package.metadata.tork.migrations]
dir               = "migrations"
file_template     = "{rev}_{slug}"
revision_style    = "sequence"
truncate_slug_length = 40
```

This tells the CLI which dialect to use for SQL generation and where to find migration files. For a project you set up manually, add these sections and create the directory:

```bash
mkdir migrations
```

Then point the CLI at your database. The easiest approach is a `DATABASE_URL` variable in `.env`:

```
DATABASE_URL=sqlite://app.db
```

Or pass it inline:

```bash
tork migrate up --database-url sqlite://app.db
```

## Migration file format

Each migration is a **single `.sql` file** containing both the up and down SQL. When you run `tork migrate create add_users`, the CLI writes this to `migrations/0001_add_users.sql`:

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
-- Write the schema changes here, for example:
-- CREATE TABLE "example" ("id" INTEGER PRIMARY KEY AUTOINCREMENT);

-- migrate:down
-- Undo the changes, for example:
-- DROP TABLE "example";
```

### Header comments

The first three lines are the revision chain headers. The CLI reads them to determine the order migrations must run; they must not be removed or edited.

| Header | Purpose |
| --- | --- |
| `-- revision: 0001` | The unique identifier for this migration. The format depends on `revision_style` in `Cargo.toml`. With `sequence` (the default), it is a zero-padded number. With `hash`, it is a 12-character hex string. |
| `-- down_revision:` | The `revision` value of the previous migration. Empty on the first migration. The CLI uses this to build the chain and to know which migration to revert next. |
| `-- name: add_users` | The human-readable name you passed to `create`. Used in status output. |

### Section markers

| Marker | Purpose |
| --- | --- |
| `-- migrate:up` | Everything below this line until `-- migrate:down` is executed when the migration is applied. |
| `-- migrate:down` | Everything below this line to the end of the file is executed when the migration is reverted. |

The placeholder comments inside each section (`-- Write the schema changes here...`) are just hints. Replace them with real SQL.

## Workflow

### 1. init

```bash
tork migrate init
```

Creates the `_tork_migrations` table in the database. Run this once per database. It is safe to run again; it does nothing if the table already exists.

```
  Initialized: _tork_migrations
```

### 2. create

```bash
tork migrate create add_users
```

Writes one file into `migrations/`:

```
migrations/
  0001_add_users.sql
```

The sequence number is padded to four digits and increments automatically. Open the file and replace the placeholder comments with real SQL.

### 3. Write the SQL

Open `migrations/0001_add_users.sql` and fill in both sections:

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
CREATE TABLE users (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    username  TEXT    NOT NULL UNIQUE,
    email     TEXT    NOT NULL UNIQUE,
    is_active INTEGER NOT NULL DEFAULT 1
);

-- migrate:down
DROP TABLE users;
```

The `-- migrate:up` section runs when the migration is applied. The `-- migrate:down` section runs when it is reverted. Always fill in the down section. It is how `tork migrate down` and `tork migrate redo` know how to undo the change.

### 4. up

```bash
tork migrate up
```

Applies every pending migration in revision order:

```
  0001_add_users   applying...
  0001_add_users   done (6ms)

  Applied 1 migration
```

Apply only the next pending migration:

```bash
tork migrate up --steps 1
```

Apply up to a specific revision:

```bash
tork migrate up 0003
```

### 5. status

```bash
tork migrate status
```

```
  0001_add_users    applied    2026-06-14 10:00:00
  0002_add_posts    applied    2026-06-14 10:01:00
  0003_add_indexes  pending

  2 applied · 1 pending
```

### 6. down

Reverts the most recently applied migration:

```bash
tork migrate down
```

```
  0002_add_posts   reverting...
  0002_add_posts   done (3ms)

  Reverted 1 migration
```

Revert multiple at once:

```bash
tork migrate down --steps 2
```

Revert all the way to an empty schema:

```bash
tork migrate down base
```

### 7. redo

```bash
tork migrate redo
```

Runs `down` on the last applied migration, then `up` again. Useful during development when you are editing a migration and want to test the full round-trip.

## A real migration sequence

### First migration: users table

```bash
tork migrate create add_users
```

`migrations/0001_add_users.sql`:

```sql
-- revision: 0001
-- down_revision:
-- name: add_users

-- migrate:up
CREATE TABLE users (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    username  TEXT    NOT NULL UNIQUE,
    email     TEXT    NOT NULL UNIQUE,
    is_active INTEGER NOT NULL DEFAULT 1
);

-- migrate:down
DROP TABLE users;
```

```bash
tork migrate up
# Applied 1 migration
```

### Second migration: posts table

```bash
tork migrate create add_posts
```

`migrations/0002_add_posts.sql`:

```sql
-- revision: 0002
-- down_revision: 0001
-- name: add_posts

-- migrate:up
CREATE TABLE posts (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title      TEXT    NOT NULL,
    body       TEXT,
    published  INTEGER NOT NULL DEFAULT 0,
    created_at TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT    NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX posts_user_id ON posts (user_id);

-- migrate:down
DROP INDEX IF EXISTS posts_user_id;
DROP TABLE posts;
```

Notice that `-- down_revision: 0001` was filled in automatically by the CLI. It chains `0002` onto `0001`, so the CLI knows the order. You never set this by hand.

```bash
tork migrate up
# Applied 1 migration
```

### Third migration: add a column

```bash
tork migrate create add_users_bio
```

`migrations/0003_add_users_bio.sql`:

```sql
-- revision: 0003
-- down_revision: 0002
-- name: add_users_bio

-- migrate:up
ALTER TABLE users ADD COLUMN bio TEXT;

-- migrate:down
ALTER TABLE users DROP COLUMN bio;
```

For SQLite versions older than 3.35 (2021), `DROP COLUMN` is not supported. On those versions the down migration must recreate the table without the column. SQLite 3.35+ supports `DROP COLUMN` directly.

## The revision chain

The `-- down_revision` header is how the CLI knows the order. It forms a linked list:

```
0001_add_users.sql          down_revision: (empty)
0002_add_posts.sql          down_revision: 0001
0003_add_users_bio.sql      down_revision: 0002
```

`tork migrate up` walks the chain forward from the lowest unapplied revision. `tork migrate down` follows `down_revision` backwards from the current head.

**File names are cosmetic.** You can rename migration files freely (for example to add a date prefix). The chain is maintained by the headers inside each file, not by lexicographic sort of file names.

This design follows the same approach as Alembic (Python) and is sometimes called a "revision graph." The chain makes it possible to merge two separate migration sequences from different branches without renaming anything.

## Alternative file name templates

The `file_template` key controls the file name. Examples:

```toml
[package.metadata.tork.migrations]
# Default: 0001_add_users.sql
file_template = "{rev}_{slug}"

# Date-prefixed: 20260614_0001_add_users.sql
file_template = "{year}{month}{day}_{rev}_{slug}"

# Timestamp-prefixed: 20260614_1430_0001_add_users.sql
file_template = "{year}{month}{day}_{hour}{minute}_{rev}_{slug}"
```

Available tokens: `{rev}`, `{slug}`, `{year}`, `{month}`, `{day}`, `{hour}`, `{minute}`, `{second}`.

The `revision_style` key controls the format of `{rev}`:

| Style | Example revision | Example file name |
| --- | --- | --- |
| `sequence` (default) | `0001` | `0001_add_users.sql` |
| `hash` | `a1b2c3d4e5f6` | `a1b2c3d4e5f6_add_users.sql` |

Sequence is predictable and easy to sort visually. Hash avoids conflicts when two developers create migrations simultaneously on different branches.

## The _tork_migrations table

Applied migrations are recorded with:

| Column | Type | Description |
| --- | --- | --- |
| `revision` | TEXT | Matches the `-- revision:` header |
| `name` | TEXT | Matches the `-- name:` header |
| `applied_at` | TEXT | ISO-8601 timestamp |
| `checksum` | TEXT | SHA-256 of the up section |

The checksum is verified on `up`. If the file was modified after being applied, the CLI refuses to continue. This prevents silent schema drift.

## Per-migration transactions

Each migration runs inside its own transaction. If the `-- migrate:up` SQL fails partway through, the partial work is rolled back and the migration is not recorded. The database stays consistent.

Some DDL statements (notably `CREATE INDEX CONCURRENTLY` in PostgreSQL) cannot run inside a transaction. To disable the per-migration transaction for a specific file, add the `-- tork:no-transaction` directive below the revision headers:

```sql
-- revision: 0005
-- down_revision: 0004
-- name: add_concurrent_index
-- tork:no-transaction

-- migrate:up
CREATE INDEX CONCURRENTLY users_email ON users (email);

-- migrate:down
DROP INDEX IF EXISTS users_email;
```

Use this only when the database requires it. The default (transactional) behavior is safer.

## Generating migrations from model definitions

Run `migrate generate` to diff your model definitions against the live database and write SQL automatically:

```bash
tork migrate generate
```

This inspects the models registered at compile time (via the `inventory` crate) and compares their index and schema declarations against what is actually in the database. It writes new migration files for the diff. It does not run the migrations; review the generated files and then run `tork migrate up`.

`migrate generate` handles:
- New tables (from new `#[derive(Model)]` structs)
- New indexes from `#[field(index)]`, `#[field(unique)]`, and `#[table(indexes = [...])]`

Column-level diffs (adding or removing columns on an existing table) are written by hand.

## Running migrations programmatically

Call the migration runner from application code. This is useful for test setups and for running migrations automatically at startup:

```rust
use std::sync::Arc;
use tork::{lifespan, LifespanContext, Resources, Result};
use tork_orm::prelude::*;

#[derive(Clone, Resources)]
pub struct Db {
    #[resource]
    pub database: Arc<Database>,
}

#[lifespan]
impl Db {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        let db = Database::connect(
            &std::env::var("DATABASE_URL").unwrap_or("sqlite://app.db".into()),
            10,
        )
        .await?;

        // Apply all pending migrations before accepting traffic.
        db.run_migrations("migrations/").await?;

        Ok(Db { database: Arc::new(db) })
    }

    async fn shutdown(self) -> Result<()> {
        self.database.close().await;
        Ok(())
    }
}
```

`db.run_migrations("migrations/")` is the same operation as `tork migrate up`; it applies every pending migration in revision order. If all migrations are already applied, it is a no-op.

## Destructive SQL

The CLI refuses to run destructive SQL (`DROP TABLE`, `DROP COLUMN`, `TRUNCATE`) unless you pass the `--allow-destructive` flag:

```bash
tork migrate up --allow-destructive
```

This protects against accidentally running a down migration section against a production database. CI pipelines typically never set `--allow-destructive`, so a migration that drops a table in the up section will fail the pipeline until the flag is added intentionally.

## Environment variables and flags

| Variable / Flag | Purpose |
| --- | --- |
| `DATABASE_URL` | Database connection string |
| `--database-url <url>` | Override `DATABASE_URL` |
| `--dir <path>` | Override the migrations directory |
| `--table <name>` | Override the tracking table name (default `_tork_migrations`) |
| `--allow-destructive` | Permit `DROP`, `TRUNCATE`, and similar statements |
| `-y` | Skip confirmation prompts (for `down` and `redo`) |

Connection string formats:

```
sqlite://app.db
sqlite::memory:
postgres://user:pass@localhost:5432/mydb
mysql://user:pass@localhost:3306/mydb
```

## Quick reference

| Command | What it does |
| --- | --- |
| `tork migrate init` | Create `_tork_migrations` table |
| `tork migrate create <name>` | Scaffold a new migration file |
| `tork migrate up` | Apply all pending migrations |
| `tork migrate up --steps N` | Apply N migrations |
| `tork migrate up <rev>` | Apply up to a specific revision |
| `tork migrate down` | Revert the last applied migration |
| `tork migrate down --steps N` | Revert N migrations |
| `tork migrate down base` | Revert all migrations |
| `tork migrate redo` | down + up on the last migration |
| `tork migrate status` | Show applied and pending migrations |
| `tork migrate generate` | Diff models against DB, write SQL |
