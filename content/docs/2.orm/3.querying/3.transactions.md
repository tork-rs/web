# Transactions

A transaction groups multiple reads and writes so they either all succeed or all fail together. tork-orm gives you three ways to use transactions, from the safest and simplest to the most explicit.

## The closure API (recommended)

`db.transaction(|tx| Box::pin(async move { ... }))` opens a transaction, calls your closure with the handle, commits on `Ok`, and rolls back on `Err`. You never have to call `commit` or handle rollback manually.

```rust
use tork_orm::prelude::*;

let db = Database::connect("sqlite://app.db", 4).await?;

let (user, post) = db.transaction(|tx| Box::pin(async move {
    let user = User::create(tx, &User {
        id: 0,
        username: "alice".into(),
        email: "alice@example.com".into(),
        is_active: true,
    })
    .await?;

    let post = Post::create(tx, &Post {
        id: 0,
        user_id: user.id,
        title: "Hello world".into(),
        published: true,
    })
    .await?;

    Ok((user, post))
}))
.await?;

println!("user={} post={}", user.id, post.id);
```

The closure receives a `&Transaction`. Because `Transaction` implements `Executor`, every ORM method (`.all()`, `.create()`, `.update()`, `.delete()`, and so on) works against the transaction handle without any changes.

If `User::create` or `Post::create` returns an error, the whole transaction is rolled back and neither row is stored.

## The `transaction!` macro

The `Box::pin` boilerplate is required because the compiler cannot name the return type of an async closure. The `transaction!` macro adds it automatically:

```rust
use tork_orm::prelude::*;
use tork_orm::transaction;

let count = transaction!(db, |tx| async move {
    tx.execute("INSERT INTO t VALUES (1)".into(), vec![]).await?;
    tx.execute("INSERT INTO t VALUES (2)".into(), vec![]).await?;
    Ok(2_usize)
})
.await?;
```

This is exactly equivalent to `db.transaction(|tx| Box::pin(async move { ... }))`.

## The explicit API

Use `db.begin()` when you need to hold the transaction handle across several await points without a closure:

```rust
let mut tx = db.begin().await?;

tx.execute("INSERT INTO orders (total) VALUES (9900)".into(), vec![]).await?;
tx.execute("UPDATE inventory SET stock = stock - 1 WHERE id = 7".into(), vec![]).await?;

tx.commit().await?;
```

Call `tx.rollback()` to undo all work:

```rust
let mut tx = db.begin().await?;
tx.execute(/* ... */.into(), vec![]).await?;
tx.rollback().await?;
```

If you drop the handle without calling `commit` or `rollback`, the transaction is rolled back automatically. The rollback happens synchronously on the thread that drops the value:

```rust
{
    let tx = db.begin().await?;
    tx.execute("UPDATE counters SET n = 99 WHERE id = 1".into(), vec![]).await?;
    // tx drops here; rolled back automatically.
}
// The counter is still at its original value.
```

This auto-rollback is a safety net, not the primary tool. Prefer the closure API so rollback is guaranteed even on a panic.

## Passing the transaction to ORM methods

All ORM model methods accept `impl Executor`. Pass `tx` (a `&Transaction`) anywhere you would pass `&db`:

```rust
db.transaction(|tx| Box::pin(async move {
    // read
    let user = User::query()
        .filter(User::id.eq(1))
        .one(tx)
        .await?;

    // write
    User::query()
        .filter(User::id.eq(1))
        .update(tx, [User::is_active.set(false)])
        .await?;

    // create
    let log = AuditLog::create(tx, &AuditLog {
        id: 0,
        user_id: user.id,
        action: "deactivated".into(),
    })
    .await?;

    Ok(log)
}))
.await?;
```

## Savepoints: partial rollback within a transaction

A savepoint is a nested checkpoint inside an open transaction. Rolling back a savepoint undoes only the work done inside it; the outer transaction continues.

```rust
db.transaction(|tx| Box::pin(async move {
    // This write survives even if the savepoint below fails.
    Order::create(tx, &Order { id: 0, total: 1999 }).await?;

    // Try to attach an order item; if this step fails, only it is undone.
    let inner: tork_orm::Result<()> = tx.savepoint(|sp| Box::pin(async move {
        OrderItem::create(sp, &OrderItem { id: 0, order_id: 1, qty: 1 }).await?;
        // Simulate a failure (e.g. stock check).
        Err(OrmError::query("out of stock"))
    }))
    .await;

    // Swallow the inner error: the order was committed but has no items.
    if inner.is_err() {
        // handle the partial failure here
    }

    Ok(())
}))
.await?;
```

Savepoints can be nested. Each call to `tx.savepoint(...)` creates a uniquely named savepoint and releases it on `Ok` or rolls it back on `Err`.

## Isolation levels

By default, `db.begin()` and `db.transaction()` use deferred locking (SQLite's `BEGIN DEFERRED`). Use `db.transaction_with()` to change the isolation level:

### SQLite isolation levels

```rust
// DEFERRED (default): no locks until the first read or write.
let mut tx = db.transaction_with().deferred().begin().await?;
tx.commit().await?;

// IMMEDIATE: acquires a write lock up front. Other readers still proceed;
// no other writer can start. Use when you know the transaction will write.
let mut tx = db.transaction_with().immediate().begin().await?;
tx.commit().await?;

// EXCLUSIVE: blocks all other connections (read and write) until the transaction ends.
let mut tx = db.transaction_with().exclusive().begin().await?;
tx.commit().await?;
```

### Using the closure style with an isolation level

```rust
db.transaction_with()
    .immediate()
    .run(|tx| Box::pin(async move {
        tx.execute("UPDATE balances SET amount = amount - 100 WHERE id = 1".into(), vec![]).await?;
        tx.execute("UPDATE balances SET amount = amount + 100 WHERE id = 2".into(), vec![]).await?;
        Ok(())
    }))
    .await?;
```

`.run()` on a `TransactionBuilder` behaves exactly like `db.transaction()`: commits on `Ok`, rolls back on `Err`.

### Standard SQL levels (PostgreSQL, MySQL)

```rust
use tork_orm::IsolationLevel;

db.transaction_with()
    .level(IsolationLevel::Serializable)
    .run(|tx| Box::pin(async move {
        // ...
        Ok(())
    }))
    .await?;
```

Available levels: `Deferred`, `Immediate`, `Exclusive` (SQLite), `ReadUncommitted`, `ReadCommitted`, `RepeatableRead`, `Serializable` (standard SQL).

## Retrying on conflict

Under serializable isolation or heavy write contention, the database may abort a transaction and signal a retryable error. `db.transaction_retry(max, |tx| ...)` re-runs the closure up to `max` times, stopping as soon as it succeeds:

```rust
db.transaction_retry(5, |tx| Box::pin(async move {
    let balance = Account::find(tx, account_id).await?;
    if balance.amount < amount {
        return Err(OrmError::query("insufficient funds"));
    }
    Account::query()
        .filter(Account::id.eq(account_id))
        .update(tx, [Account::amount.set(Account::amount.sub(amount))])
        .await?;
    Ok(())
}))
.await?;
```

The closure must be safe to run multiple times. Do not generate UUIDs or call external services inside it; do those after the transaction commits.

## A real example: order placement

```rust
use tork_orm::prelude::*;
use tork_orm::transaction;

pub async fn place_order(
    db: &Database,
    user_id: i64,
    items: &[(i64, i32)], // (product_id, quantity)
) -> tork_orm::Result<Order> {
    let total: i64 = items.iter().map(|(_, qty)| *qty as i64 * 1000).sum();

    transaction!(db, |tx| async move {
        // 1. Create the order header.
        let order = Order::create(tx, &Order {
            id: 0,
            user_id,
            total_cents: total,
            status: OrderStatus::Pending,
        })
        .await?;

        // 2. Insert all line items; any failure rolls back the order too.
        for (product_id, qty) in items {
            OrderItem::create(tx, &OrderItem {
                id: 0,
                order_id: order.id,
                product_id: *product_id,
                quantity: *qty,
            })
            .await?;
        }

        // 3. Decrement stock inside a savepoint so a stock error only
        //    prevents that specific item from being reserved.
        for (product_id, qty) in items {
            let reserved: tork_orm::Result<()> = tx.savepoint(|sp| Box::pin(async move {
                let rows = Product::query()
                    .filter(Product::id.eq(*product_id))
                    .update(sp, [Product::stock.set(Product::stock.sub(*qty as i64))])
                    .await?;
                if rows == 0 {
                    return Err(OrmError::query("product not found"));
                }
                Ok(())
            }))
            .await;

            if let Err(e) = reserved {
                // Log or handle but continue; the order still commits
                // without stock reservation for this item.
                eprintln!("stock reservation failed for product {product_id}: {e}");
            }
        }

        Ok(order)
    })
    .await
}
```

## What to watch out for

**Connection pooling.** A `Transaction` pins one connection from the pool for its entire lifetime. With a pool size of 1 and a transaction that tries to open another transaction, you get a deadlock. Keep pool size at least as large as the maximum concurrent transactions you expect.

**Auto-rollback on drop.** If you ignore a `Result` from `commit()`, the transaction is still dropped, and the rollback runs. But the `commit` error is lost. Always propagate or handle the `commit()` result.

**Hooks.** Model lifecycle hooks (`after_create`, etc.) run inside the transaction. An `after_create` that opens its own database connection and queries data outside the transaction may see an inconsistent state. Keep hooks lightweight.

**Retryable errors.** Under serializable isolation, the database can abort a transaction with a serialization error. `transaction_retry` handles this automatically. In the plain `db.transaction()` API, check `OrmError::is_retryable()` if you need to implement your own retry loop.

## Quick reference

| API | Description |
| --- | --- |
| `db.transaction(\|tx\| Box::pin(...))` | Closure API: commit on Ok, rollback on Err |
| `transaction!(db, \|tx\| async move {...})` | Macro sugar (no Box::pin needed) |
| `db.begin()` | Open a transaction explicitly |
| `tx.commit()` | Commit the open transaction |
| `tx.rollback()` | Roll back the open transaction |
| Drop without commit | Automatic rollback |
| `tx.savepoint(\|sp\| Box::pin(...))` | Nested partial rollback |
| `db.transaction_with().immediate().begin()` | Explicit isolation level |
| `db.transaction_with().immediate().run(\|tx\| ...)` | Closure + isolation level |
| `db.transaction_retry(n, \|tx\| ...)` | Retry on serializable conflict |

Continue to [Repository pattern](/docs/orm/advanced/repository-pattern) for structuring database access in a Tork application.
