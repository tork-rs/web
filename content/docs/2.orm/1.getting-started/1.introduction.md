# tork-orm

`tork-orm` is the data layer of the Tork toolchain. You derive `Model` on a plain Rust struct, and the ORM derives the table definition, a typed query builder, and a reversible migration, all checked by the compiler before anything touches a database.

## Starting a new project

The fastest path is `tork new`. Install the CLI first:

```bash
cargo install tork-cli
```

Then scaffold a project with a database backend:

```bash
tork new myapp --db sqlite
cd myapp
```

`tork new` creates the directory structure, the `Cargo.toml` with the right dependencies and migration metadata, a `.env.example`, and the core data-layer modules (`src/core/db.rs`, `src/models/`, `src/repositories/`, `migrations/`).

For PostgreSQL:

```bash
tork new myapp --db postgres
```

For MySQL:

```bash
tork new myapp --db mysql
```

## What tork new generates

After `tork new myapp --db sqlite` you get:

```
myapp/
  Cargo.toml
  rust-toolchain.toml
  .gitignore
  .env.example
  migrations/
  src/
    main.rs
    core/
      mod.rs
      settings.rs
      db.rs          database connection and lifespan
    schemas/
      mod.rs
    routers/
      mod.rs
      health.rs
    services/
      mod.rs
    models/
      mod.rs
    repositories/
      mod.rs
```

The generated `Cargo.toml`:

```toml
[package]
name = "myapp"
version = "0.1.0"
edition = "2021"
publish = false

[dependencies]
tork     = { git = "https://github.com/tork-rs/tork-framework.git" }
tork-orm = { git = "https://github.com/tork-rs/tork-orm.git", default-features = false, features = ["sqlite", "tork", "migrations"] }
garde    = { version = "0.23", features = ["derive", "email"] }
tokio    = { version = "1", features = ["rt-multi-thread", "macros"] }

[package.metadata.tork]
dialect = "sqlite"

[package.metadata.tork.migrations]
dir               = "migrations"
file_template     = "{rev}_{slug}"
revision_style    = "sequence"
truncate_slug_length = 40
```

The `[package.metadata.tork]` section tells the `tork migrate` commands which dialect to use and where to find migration files. The `features` on `tork-orm` enable the SQLite backend (`sqlite`), the migration engine (`migrations`), and the Tork framework integration (`tork`).

## Adding tork-orm to an existing project

If you already have a Rust project, add the dependency manually:

```toml
# SQLite (default backend)
[dependencies]
tork-orm = { git = "https://github.com/tork-rs/tork-orm.git" }
# default features already include sqlite + migrations

# PostgreSQL
[dependencies]
tork-orm = { git = "https://github.com/tork-rs/tork-orm.git", default-features = false, features = ["postgres", "migrations"] }

# MySQL
[dependencies]
tork-orm = { git = "https://github.com/tork-rs/tork-orm.git", default-features = false, features = ["mysql", "migrations"] }
```

When using tork-orm inside a Tork web application, add the `tork` feature to get automatic DI and error bridging:

```toml
tork-orm = { git = "...", default-features = false, features = ["sqlite", "tork", "migrations"] }
```

Add the migration metadata to your `Cargo.toml` so `tork migrate` knows where to look:

```toml
[package.metadata.tork]
dialect = "sqlite"

[package.metadata.tork.migrations]
dir = "migrations"
```

Then create the migrations directory:

```bash
mkdir migrations
```

## Feature flags

| Feature | What it enables |
| --- | --- |
| `sqlite` | Bundled SQLite backend via rusqlite (no system library needed) |
| `postgres` | PostgreSQL backend |
| `mysql` | MySQL and MariaDB backend |
| `migrations` | Migration engine and `migrate generate` model registry |
| `tork` | Integration with the Tork web framework (DI, error bridging) |

`sqlite` and `migrations` are on by default.

## Migrations with the tork CLI

The `tork` CLI exposes migration commands identical to the standalone `tork-orm` binary. If you installed `tork-cli`, use `tork migrate`:

```bash
# Create the _tork_migrations table and verify the migrations directory
tork migrate init

# Scaffold a new up/down migration pair
tork migrate create add_users

# Apply all pending migrations
tork migrate up

# Check which migrations have been applied
tork migrate status

# Roll back the last migration
tork migrate down

# Roll back and re-apply the last migration
tork migrate redo
```

Point the commands at your database with the `DATABASE_URL` variable or the `--database-url` flag:

```bash
export DATABASE_URL=sqlite://app.db
tork migrate up
```

If you installed only the ORM binary, substitute `tork-orm migrate` for `tork migrate` (the commands are identical).

## First run

Starting from a freshly scaffolded project:

```bash
# Copy the example env and point it at a local database
cp .env.example .env
# (edit DB_URL in .env if needed; default is sqlite://app.db)

# Create the migration tracking table
tork migrate init

# Start developing
tork dev
```

`tork dev` compiles and runs the project, then watches for file changes and restarts automatically.

## A minimal example (standalone)

This works without the Tork framework:

```rust
use tork_orm::prelude::*;

#[derive(Debug, Clone, Model)]
#[table(name = "users")]
pub struct User {
    #[field(primary_key, auto)]
    pub id: i64,
    #[field(varchar(length = 50), unique)]
    pub username: String,
    pub email: String,
    pub is_active: bool,
}

#[tokio::main]
async fn main() -> tork_orm::Result<()> {
    let db = Database::connect("sqlite://app.db", 4).await?;

    db.execute(
        "CREATE TABLE IF NOT EXISTS users (
            id INTEGER PRIMARY KEY,
            username TEXT NOT NULL UNIQUE,
            email TEXT NOT NULL,
            is_active INTEGER NOT NULL
        )"
        .into(),
        vec![],
    )
    .await?;

    let alice = User::create(
        &db,
        &User { id: 0, username: "alice".into(), email: "alice@example.com".into(), is_active: true },
    )
    .await?;
    println!("created id={}", alice.id);

    let active = User::query()
        .filter(User::is_active.eq(true))
        .order_by(User::username.asc())
        .all(&db)
        .await?;
    println!("active users: {}", active.len());

    db.close().await;
    Ok(())
}
```

## Using the prelude

Every example in the ORM docs starts with:

```rust
use tork_orm::prelude::*;
```

This brings in `Model`, `Database`, `Executor`, `QuerySet`, `Expr`, `Column`, `Transaction`, `OrmError`, `ErrorKind`, `Value`, `Row`, `IsolationLevel`, and all aggregate and window functions. It also re-exports the derive macros (`Model`, `#[relations]`, `DbEnum`).

## What the ORM gives you

**One source of truth.** The struct is the schema. No separate SQL file or YAML to keep in sync.

**Typed queries.** Columns are checked at compile time. A wrong column name or a type mismatch is a build error, not a runtime crash.

**Reversible migrations.** Every schema change is a pair of SQL files (`up` / `down`), tracked in `_tork_migrations`.

**Connection pooling.** `Database::connect(url, max)` opens a pool. Every query borrows a connection and returns it automatically.

**Works inside transactions.** Every ORM method accepts `impl Executor`. Pass a `&Transaction` or `&Database`; the same code works either way.

**Async.** Built on Tokio. Every terminal method is `async`.
