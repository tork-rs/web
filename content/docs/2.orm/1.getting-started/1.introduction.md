# tork-orm

`tork-orm` is the data layer of the Tork toolchain. You derive `Model` on a plain Rust struct, and the ORM gives you the table definition, a typed query builder, and reversible migrations, all checked by the compiler before anything touches a database.

It supports SQLite, PostgreSQL, and MySQL (and MariaDB), with the same code across all three.

## Standalone or with the Tork framework

`tork-orm` is a normal Rust crate with its own CLI. It does not depend on the Tork web framework. With its default features it pulls in no framework at all, so you can use it in an Axum or Actix app, a background worker, or a plain binary, the same way you would any other crate.

If you build your backend with the Tork framework, enable the `tork` feature to turn on the adapter. The adapter bridges ORM errors to HTTP responses, so a query that finds no row answers `404`, a unique conflict answers `409`, and a connection failure answers `503`, with no extra code. The framework also injects the connection pool as `Arc<Database>`. See [Database](/docs/framework/techniques/database) for that integration.

The CLI follows the same split. The standalone `tork-orm` binary manages migrations in any project. The unified `tork` binary, which you use when building with the framework, includes the exact same migration commands natively, so you do not install a second tool.

## Installing the crate

Add `tork-orm` to your `Cargo.toml`. SQLite and the migration engine are on by default.

```toml
[dependencies]
tork-orm = { version = "0.1", default-features = false, features = ["sqlite", "migrations"] }
```

Choose your backend with a feature. Inside a Tork web app, add `tork` for the adapter.

```toml
# PostgreSQL
tork-orm = { version = "0.1", default-features = false, features = ["postgres", "migrations"] }

# MySQL or MariaDB
tork-orm = { version = "0.1", default-features = false, features = ["mysql", "migrations"] }

# SQLite, inside a Tork web app
tork-orm = { version = "0.1", default-features = false, features = ["sqlite", "tork", "migrations"] }
```

The features:

| Feature | What it enables | Default |
| --- | --- | --- |
| `sqlite` | Bundled SQLite backend via rusqlite, no system library needed | yes |
| `postgres` | PostgreSQL backend over tokio-postgres with a pool | no |
| `mysql` | MySQL and MariaDB backend over mysql_async | no |
| `migrations` | The migration engine | yes |
| `tork` | The Tork framework adapter (error bridging) | no |

All three backends are fully implemented. Connecting with a feature you did not enable returns a clear error that names the missing feature.

## The CLI

Migrations are driven by a command-line tool. There are two ways to get it, and they run the same migration engine.

### The standalone ORM CLI

For any project, including an Axum or Actix app, install the standalone CLI. The crate is `tork-orm-cli`, and it gives you a binary named `tork-orm`.

```bash
cargo install tork-orm-cli
```

All migration commands live under `migrate`:

```bash
tork-orm migrate init
tork-orm migrate create add_users
tork-orm migrate up
tork-orm migrate status
tork-orm migrate down
tork-orm migrate redo
```

### The Tork CLI

When you build with the Tork framework, install the unified CLI instead. The crate is `tork-cli`, and it gives you a binary named `tork`.

```bash
cargo install tork-cli
```

The `tork` binary includes the same migration commands natively, so you do not need the standalone ORM CLI:

```bash
tork migrate up
tork migrate status
```

`tork migrate` and `tork-orm migrate` run the same migration engine (see the one `generate` caveat below). The `tork` binary adds project commands too: `tork new` scaffolds a project, `tork dev` rebuilds and restarts on file changes, and `tork build`, `tork check`, and `tork format` wrap the matching cargo commands.

### Migration commands

| Command | What it does |
| --- | --- |
| `migrate init` | Creates the migrations directory |
| `migrate create <name>` | Scaffolds a new migration file (an `up` and a `down` section) |
| `migrate up [target]` | Applies all pending migrations, or up to `target` |
| `migrate down [target]` | Rolls back one migration, `<n>` of them, or `base` for all |
| `migrate status` | Lists each migration as applied or pending |
| `migrate redo` | Rolls back the most recent migration and re-applies it |
| `migrate generate <name>` | Diffs your models against the live database and writes a migration with the SQL to close the gap |

`tork migrate` and `tork-orm migrate` share the same engine, with one exception: `migrate generate` reads your compiled model schema, so it runs only through the `tork` CLI (which builds the project first). On the standalone `tork-orm` binary it returns an error pointing you to `tork migrate generate`.

Point the commands at your database with the `--database-url` flag (short `-d`), or the `DATABASE_URL` environment variable (with `DB_URL` as a fallback).

```bash
export DATABASE_URL=sqlite://app.db
tork migrate up
```

Applied migrations are tracked in a `_tork_migrations` table, which you can rename with `--table`. The migrations directory defaults to `migrations`, overridable with `--dir`.

## TOML configuration

The CLI reads its settings from a `[package.metadata.tork]` section in your project's `Cargo.toml`. Every key is optional and has a default, so you only set what you want to change.

```toml
[package.metadata.tork]
dialect = "sqlite"

[package.metadata.tork.migrations]
dir = "migrations"
file_template = "{rev}_{slug}"
revision_style = "sequence"
truncate_slug_length = 40
```

`[package.metadata.tork]`:

| Key | Default | Values |
| --- | --- | --- |
| `dialect` | `sqlite` | `sqlite`, `postgres`, `mysql` (the SQL dialect your migrations target) |

`[package.metadata.tork.migrations]`:

| Key | Default | What it controls |
| --- | --- | --- |
| `dir` | `migrations` | Where migration files live |
| `file_template` | `{rev}_{slug}` | The migration file name |
| `revision_style` | `hash` | How a new revision id is generated: `hash` or `sequence` |
| `truncate_slug_length` | `40` | Maximum length of the slug taken from the migration name |
| `timezone` | UTC | Timezone for the date tokens in `file_template` |

`file_template` understands the tokens `{rev}`, `{slug}`, `{year}`, `{month}`, `{day}`, `{hour}`, `{minute}`, and `{second}`. With `revision_style = "sequence"`, `{rev}` is a zero-padded counter like `0001`; with `hash`, it is a short random hex id.

Connection settings, such as the host, credentials, and pool size, are deliberately not part of this file. They stay in your environment and application code, which is why the CLI takes the database URL from `DATABASE_URL` or `--database-url`.

## Scaffolding a project with tork new

When you build with the framework, `tork new` sets up a working project, including the `Cargo.toml`, the migration config above, and the data-layer modules.

```bash
tork new myapp --db sqlite   # or postgres, mysql, or none
cd myapp
```

It generates this layout:

```
myapp/
  Cargo.toml          dependencies and the [package.metadata.tork] config
  .env.example
  migrations/
  src/
    main.rs
    core/
      db.rs           database connection and lifespan
      settings.rs
    models/
    repositories/
    routers/
    schemas/
    services/
```

The `--db` flag accepts `sqlite`, `postgres`, `mysql`, or `none`. With `none`, the ORM dependency and the migration config are left out. By default the generated `Cargo.toml` uses release versions of `tork` and `tork-orm`; pass `--orm-git` or `--framework-git` to depend on the Git repositories instead.

## A minimal standalone example

This program uses `tork-orm` on its own, with no web framework. The struct is the model, `Database::connect` opens a pool, and the typed query builder reads it back.

```rust
use tork_orm::prelude::*;

#[derive(Debug, Clone, Model)]
#[table(name = "users")]
pub struct User {
    #[field(primary_key, auto)]
    pub id: i64,
    #[field(varchar(length = 50), unique)]
    pub username: String,
    pub email: String,
    pub is_active: bool,
}

#[tokio::main]
async fn main() -> tork_orm::Result<()> {
    let db = Database::connect("sqlite://app.db", 4).await?;

    db.execute(
        "CREATE TABLE IF NOT EXISTS users (
            id INTEGER PRIMARY KEY,
            username TEXT NOT NULL UNIQUE,
            email TEXT NOT NULL,
            is_active INTEGER NOT NULL
        )"
        .into(),
        vec![],
    )
    .await?;

    let alice = User::create(
        &db,
        &User { id: 0, username: "alice".into(), email: "alice@example.com".into(), is_active: true },
    )
    .await?;
    println!("created id={}", alice.id);

    let active = User::query()
        .filter(User::is_active.eq(true))
        .order_by(User::username.asc())
        .all(&db)
        .await?;
    println!("active users: {}", active.len());

    db.close().await;
    Ok(())
}
```

A real project applies migrations with the CLI instead of a hand-written `CREATE TABLE`. The raw statement here just keeps the example self-contained.

## Using the prelude

Every example in the ORM docs starts with:

```rust
use tork_orm::prelude::*;
```

This brings in the core types you work with, including `Model`, `Database`, `Executor`, `QuerySet`, `Expr`, `Column`, `Transaction`, `OrmError`, `ErrorKind`, `Value`, `Row`, and `IsolationLevel`, along with the aggregate and window functions and the derive macros `Model`, `#[relations]`, and `DbEnum`.

## What the ORM gives you

- **One source of truth.** The struct is the schema. There is no separate SQL file or YAML to keep in sync.
- **Typed queries.** Columns are checked at compile time, so a wrong name or a type mismatch is a build error, not a runtime crash.
- **Reversible migrations.** Each schema change is an `up` and a `down`, tracked in `_tork_migrations`.
- **Connection pooling.** `Database::connect(url, max)` opens a pool, and every query borrows and returns a connection automatically.
- **Works in transactions.** Every ORM method takes `impl Executor`, so the same code runs against a `&Database` or a `&Transaction`.
- **Async.** Built on Tokio. Every terminal method is `async`.
