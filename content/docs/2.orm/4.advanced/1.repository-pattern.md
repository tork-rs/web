# Repository pattern

A repository collects every database operation for one model into a single struct. Handlers ask for the repository rather than touching `Arc<Database>` directly. This keeps handlers thin, makes the database code easy to find, and makes testing possible by swapping the repository for a test double.

## The basic shape

```rust
use std::sync::Arc;
use tork::Inject;
use tork_orm::prelude::*;

#[derive(Debug, Clone, Model)]
#[table(name = "users")]
pub struct User {
    #[field(primary_key, auto)]
    pub id: i64,
    #[field(varchar(length = 50), unique)]
    pub username: String,
    pub email: String,
    pub is_active: bool,
}

#[derive(Inject)]
pub struct UserRepository {
    db: Arc<Database>,
}

impl UserRepository {
    pub async fn find_all(&self) -> tork_orm::Result<Vec<User>> {
        User::query()
            .order_by(User::id.asc())
            .all(&*self.db)
            .await
    }

    pub async fn find_by_id(&self, id: i64) -> tork_orm::Result<User> {
        User::find(&*self.db, id).await
    }

    pub async fn find_active(&self) -> tork_orm::Result<Vec<User>> {
        User::query()
            .filter(User::is_active.eq(true))
            .order_by(User::username.asc())
            .all(&*self.db)
            .await
    }

    pub async fn create(&self, username: String, email: String) -> tork_orm::Result<User> {
        User::create(
            &*self.db,
            &User { id: 0, username, email, is_active: true },
        )
        .await
    }

    pub async fn deactivate(&self, id: i64) -> tork_orm::Result<u64> {
        User::query()
            .filter(User::id.eq(id))
            .update(&*self.db, [User::is_active.set(false)])
            .await
    }

    pub async fn delete(&self, id: i64) -> tork_orm::Result<u64> {
        let user = User::find(&*self.db, id).await?;
        user.delete(&*self.db).await
    }
}
```

`#[derive(Inject)]` makes `UserRepository` injectable into handlers. Tork instantiates it automatically when a handler declares it as a parameter, resolving `Arc<Database>` from the app's shared resources.

## Injecting into handlers

```rust
use tork::{api_router, delete, get, post, put, Valid};
use tork::Error;
use serde::{Deserialize, Serialize};

#[derive(Serialize)]
pub struct UserOut {
    pub id: i64,
    pub username: String,
    pub email: String,
    pub is_active: bool,
}

impl From<User> for UserOut {
    fn from(u: User) -> Self {
        Self { id: u.id, username: u.username, email: u.email, is_active: u.is_active }
    }
}

#[derive(Deserialize, garde::Validate)]
pub struct CreateUserBody {
    #[garde(length(min = 2, max = 50))]
    pub username: String,
    #[garde(email)]
    pub email: String,
}

#[api_router(prefix = "/users", tags = ["users"])]
pub mod users_router {
    use super::*;

    #[get("", response_model = Vec<UserOut>)]
    pub async fn list_users(repo: UserRepository) -> tork::Result<Vec<UserOut>> {
        let users = repo.find_all().await?;
        Ok(users.into_iter().map(UserOut::from).collect())
    }

    #[get("/{id}", response_model = UserOut)]
    pub async fn get_user(id: i64, repo: UserRepository) -> tork::Result<UserOut> {
        let user = repo.find_by_id(id).await
            .map_err(|e| if e.kind() == tork_orm::ErrorKind::NotFound {
                Error::not_found("user not found")
            } else {
                Error::internal(e.to_string())
            })?;
        Ok(UserOut::from(user))
    }

    #[post("", response_model = UserOut, status_code = 201)]
    pub async fn create_user(
        repo: UserRepository,
        body: Valid<CreateUserBody>,
    ) -> tork::Result<UserOut> {
        let input = body.into_inner();
        let user = repo.create(input.username, input.email).await?;
        Ok(UserOut::from(user))
    }

    #[delete("/{id}")]
    pub async fn delete_user(id: i64, repo: UserRepository) -> tork::Result<()> {
        repo.delete(id).await?;
        Ok(())
    }
}
```

The handler only knows about the repository. It does not know which database backend is in use or how the query is built.

## Multiple repositories in one handler

A handler can ask for more than one repository:

```rust
#[post("/orders", response_model = OrderOut, status_code = 201)]
pub async fn create_order(
    order_repo: OrderRepository,
    product_repo: ProductRepository,
    body: Valid<CreateOrderBody>,
) -> tork::Result<OrderOut> {
    let input = body.into_inner();

    // Validate stock before creating the order.
    for item in &input.items {
        let product = product_repo.find_by_id(item.product_id).await?;
        if product.stock < item.quantity as i64 {
            return Err(Error::unprocessable_entity(
                format!("product {} is out of stock", product.name)
            ));
        }
    }

    let order = order_repo.create(&input).await?;
    Ok(OrderOut::from(order))
}
```

## Transactions inside a repository

Pass `Arc<Database>` to the transaction API. Methods that need a transaction accept `impl Executor` or take an explicit `&Transaction`:

```rust
use tork_orm::transaction;

#[derive(Inject)]
pub struct OrderRepository {
    db: Arc<Database>,
}

impl OrderRepository {
    pub async fn place(
        &self,
        user_id: i64,
        items: Vec<(i64, i32)>,
    ) -> tork_orm::Result<Order> {
        let total: i64 = items.iter().map(|(_, qty)| *qty as i64 * 1000).sum();

        transaction!(self.db, |tx| async move {
            let order = Order::create(tx, &Order {
                id: 0,
                user_id,
                total_cents: total,
            })
            .await?;

            for (product_id, qty) in items {
                OrderItem::create(tx, &OrderItem {
                    id: 0,
                    order_id: order.id,
                    product_id,
                    quantity: qty,
                })
                .await?;
            }

            Ok(order)
        })
        .await
    }
}
```

Because the transaction handle implements `Executor`, every ORM call inside the closure (`Order::create`, `OrderItem::create`) participates in the same transaction without any special wiring.

## A repository with pagination and filtering

```rust
pub struct UserFilters {
    pub is_active: Option<bool>,
    pub username_contains: Option<String>,
}

pub struct Page {
    pub offset: u64,
    pub limit: u64,
}

#[derive(Inject)]
pub struct UserRepository {
    db: Arc<Database>,
}

impl UserRepository {
    pub async fn list(
        &self,
        filters: UserFilters,
        page: Page,
    ) -> tork_orm::Result<(Vec<User>, i64)> {
        let mut q = User::query();

        if let Some(active) = filters.is_active {
            q = q.filter(User::is_active.eq(active));
        }

        if let Some(ref substr) = filters.username_contains {
            q = q.filter(User::username.like(format!("%{substr}%")));
        }

        let total = q.count(&*self.db).await?;

        let users = q
            .order_by(User::id.asc())
            .limit(page.limit)
            .offset(page.offset)
            .all(&*self.db)
            .await?;

        Ok((users, total))
    }
}
```

## Connecting the database at startup

Register `Arc<Database>` as a resource in the Tork lifespan so repositories receive it automatically:

```rust
use std::sync::Arc;
use tork::{lifespan, settings, LifespanContext, Resources, Result};
use tork_orm::prelude::*;

#[settings(prefix = "DB")]
pub struct DatabaseConfig {
    #[setting(default = "sqlite://app.db")]
    pub url: String,
    #[setting(default = 10, ge = 1, le = 64)]
    pub max_connections: u32,
}

#[derive(Clone, Resources)]
pub struct Db {
    #[resource]
    pub database: Arc<Database>,
}

#[lifespan]
impl Db {
    async fn startup(_ctx: LifespanContext) -> Result<Self> {
        let config = DatabaseConfig::load()?;
        let database = Database::connect(&config.url, config.max_connections).await?;
        database.run_migrations("migrations/").await?;
        Ok(Db { database: Arc::new(database) })
    }

    async fn shutdown(self) -> Result<()> {
        self.database.close().await;
        Ok(())
    }
}
```

```rust
// main.rs
use tork::App;

#[tork::main]
async fn main() -> tork::Result<()> {
    App::new()
        .lifespan::<core::db::Db>()
        .include_router(routers::router())
        .serve("0.0.0.0:8000")
        .await
}
```

With this in place, any repository that declares `db: Arc<Database>` receives the pool connection automatically when the handler is invoked.

## Suggested project layout

```
src/
  core/
    db.rs          Db lifespan, DatabaseConfig settings
  models/
    mod.rs
    user.rs        User struct + ModelHooks
    post.rs        Post struct + relations
  repositories/
    mod.rs
    user_repo.rs   UserRepository
    post_repo.rs   PostRepository
  schemas/
    mod.rs
    user.rs        UserOut, CreateUserBody
  routers/
    mod.rs
    users.rs       handlers that inject UserRepository
```

`tork new myapp --db sqlite` scaffolds this layout for you. Add a repository by creating the file in `src/repositories/` and declaring it in `mod.rs`.

## Summary

| Concept | How it fits |
| --- | --- |
| `#[derive(Inject)]` | Makes the repository injectable into handlers |
| `Arc<Database>` field | Resolved from the app's shared resources at startup |
| ORM methods on `&*self.db` | Same as calling on a plain `&Database` |
| `transaction!(self.db, ...)` | Runs a block of writes atomically |
| Multiple repositories per handler | Tork resolves each independently |
